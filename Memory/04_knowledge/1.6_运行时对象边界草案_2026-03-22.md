# 1.6 运行时对象边界草案

> 版本：v0.1  
> 日期：2026-03-22  
> 范围：为 1.6 第一阶段固定 `memory_unit / source_assets / archive_anchor / retrieval_tools` 的运行时边界，作为后续提取层、DataLayerAPI、AI Chat 的共同起点。

---

## 1. 这一步为什么现在做

`1.5` 和 `1.7` 已经把 session 级图检索与三路混合检索跑通了。  
接下来如果直接进入提取实现，而不先固定运行时对象边界，后面很容易出现这几类反复返工：

- 提取层产物到底写成 `session`、`unit` 还是临时摘要对象，口径不一致。
- vector / graph 索引到底挂在 transcript/session 还是挂在 unit 上，主写入对象不稳定。
- AI Chat、DataLayerAPI、压缩归档各自引入一套“自己的对象模型”，后续很难收敛。

所以 1.6 的第一段不应该先写提取代码，而应该先把边界钉住。

---

## 2. 四个核心对象的定位

### 2.1 `memory_unit`

这是运行时主记忆对象。  
它不等于整段 session，也不等于一次 commit，而是“可以被独立检索、独立加权、独立关联、独立修订”的最小知识单元。

当前建议：

- `memory_unit` 是后续 vector / graph 的主写入对象。
- `memory_unit` 是 AI Chat 精细召回时的主对象。
- `memory_unit` 需要可回源到 `source_assets`，也允许关联 `commit_ref`，但不被 `commit` 取代。

### 2.2 `source_assets`

这是证据层 / 来源层对象。  
它保存 transcript、document、web、video 等原始或近原始内容，以及这些内容的导入来源。

当前建议：

- `source_assets` 不承担精细知识检索主对象职责。
- `source_assets` 主要负责回源、引用、审计、定位、原文展开。
- `memory_unit` 应指向 `source_assets`，而不是反过来依赖 unit 承载原始全文。

### 2.3 `archive_anchor`

这是压缩 / 归档后的轻量入口对象。  
它不是原文，也不是完整 unit 集合，而是一个“重新进入历史内容”的运行时锚点。

当前建议：

- `archive_anchor` 应引用相关 `memory_unit` 集合和 `source_assets`。
- `archive_anchor` 可以附带 `commit_ref`，但本体不是 commit。
- 后续做压缩、长对话归档、跨 session 回溯时，优先用它做入口。

### 2.4 `retrieval_tools`

这不是持久化数据表对象，而是运行时能力边界。  
它回答的是“系统允许上层怎么取回这些对象”。

当前建议：

- `retrieval_tools` 至少要覆盖 `searchSessions / searchMemoryUnits / getSourceAsset / getArchiveAnchor`。
- `getCommitContext` 保留为可选能力，定位在事件层 / 溯源层，而不是通用主召回入口。
- 它是 1.9 `DataLayerAPI` 的前置轮廓，不应等到 1.9 才第一次讨论。

---

## 3. 明确不做的混淆

### 3.1 `session` 不是 `memory_unit`

- `session` 仍然是来源容器。
- `memory_unit` 才是后续细粒度知识对象。
- `1.7` 的 session 级 mixed retrieval 是 Phase 0 闭环，不是最终对象模型。

### 3.2 `Git commit` 不是通用运行时记忆单元

- `commit` 继续承担事件层 / 溯源层 / 粗筛入口 / 归档锚点辅助角色。
- `commit` 可以服务检索，但不取代 `memory_unit`。
- 这条边界在 1.6 期间不再反复摇摆。

### 3.3 `retrieval_tools` 不是“让模型自由发挥”

- 它是显式能力集合，不是模糊提示词。
- 后续模型只能通过这些能力边界访问底层对象，而不是直接把所有对象层混在一起。

---

## 4. 这次落到代码里的第一版草案

本次先在共享类型层落地第一版草案，再把它提升成 `core` 内部的独立 memory domain：

- `packages/types/src/memory.ts`
- `packages/types/src/index.ts`
- `packages/core/src/index.ts`
- `packages/core/src/memory/domain.ts`
- `packages/core/src/memory/data-layer-api.ts`
- `packages/core/src/memory/index.ts`

当前落地内容：

- `MemoryUnit`
- `SourceAsset`
- `ArchiveAnchor`
- `RetrievalTools`
- `DataLayerAPI`
- runtime constants / guards / query normalizers
- 配套的 `RetrievalQuery / RetrievalReference / unit_type / status / source_kind`

这一步的目标不是把 1.6 全做完，而是先把“后面到底围绕什么对象继续做”固定下来。

---

## 5. 对后续执行顺序的直接影响

现在更稳的主线顺序是：

1. 先固定运行时对象边界  
2. 再写提取层，让 heartbeat / extractor 产出 `memory_unit + entity/relation + archive_anchor`  
3. 再把 vector / graph 主索引从 session 级迁到 unit 级  
4. 再把 AI Chat / DataLayerAPI 正式接到这些对象上

---

## 6. 当前判断

这一步是最佳时机做，而且应该在 1.6 前半段尽快做完。  
原因不是“它最炫”，而是“它最能减少后续返工”。

一句话总结：

> 从现在开始，OpenMnemo 的主运行时对象默认视角应从 `session` 过渡到 `memory_unit`；`source_assets` 负责回源，`archive_anchor` 负责压缩入口，`retrieval_tools` 负责调用边界，而 `commit` 继续留在事件与溯源层。
