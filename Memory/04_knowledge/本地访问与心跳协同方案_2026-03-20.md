# 本地访问与心跳协同方案

> 版本：v0.2  
> 日期：2026-03-20  
> 范围：方案讨论稿，不包含本次代码改动

---

## 1. 问题定义

当前有三个容易混淆的概念，需要先拆开：

- `heartbeat`：负责定时发现新会话、导入数据、生成报告、按规则提交和推送。
- `report serve`：一个本地临时 HTTP 服务，适合开发和调试。
- `Caddy`：可以作为本地常驻 Web 服务，把静态报告目录长期挂出来。

核心结论：

- `heartbeat` 不应该承担长期网页服务。
- 本地长期访问应该由常驻 Web 服务承担，例如 Caddy。
- 地址控制不应只靠一个全局固定端口，应该保留项目级控制能力。

---

## 2. 目标

本方案希望同时满足以下目标：

- 本地访问地址稳定，浏览器可以长期打开。
- 一个机器上可以承载多个项目和多个本地 Web 服务。
- 支持本机访问、局域网访问和未来公网访问的分层治理。
- `heartbeat` 继续保持“定时执行一次就退出”的模型，不变成常驻 Web 进程。
- 自动写仓库的逻辑不污染正常开发分支。
- 后续可以平滑扩展到公网发布。

---

## 3. 方案结论

推荐采用“三层分工”：

1. `heartbeat` 负责更新内容  
   发现 transcript、导入 Memory、重建 `Memory/07_reports`。

2. `Caddy` 负责本地常驻访问  
   长期将 `Memory/07_reports` 作为静态站点发布。

3. 公网发布层负责外部访问  
   后续再将同一份静态报告发布到 GitHub Pages 或其他静态托管。

可视化理解：

```text
本地 transcript / 对话记录
            |
            v
       heartbeat
  discover -> import -> build report
            |
            v
  Memory/07_reports 静态文件
            |
            v
         Caddy
  常驻提供本地 HTTP 访问
            |
            v
 浏览器 / 本地入口 / 后续公网入口
```

---

## 4. 组件职责

### 4.1 heartbeat

职责：

- 定时扫描新 transcript
- 导入当前项目的 Memory 数据
- 按需生成 `Memory/07_reports`
- 在满足安全规则时提交和推送 Memory 相关变更

边界：

- 不负责长期监听端口
- 不负责充当稳定网页服务器
- 不负责替代 Caddy 或 Nginx

### 4.2 report serve

职责：

- 本地开发和调试时，临时查看报告页面

边界：

- 不建议作为正式长期运行的本地访问层
- 不适合作为多项目本地服务编排中心

### 4.3 Caddy

职责：

- 常驻监听本地地址
- 按项目把静态目录稳定暴露出来
- 为多个本地项目提供统一入口

边界：

- 不负责生成报告内容
- 不负责导入 transcript

### 4.4 当前实现风险提醒

当前 `openmnemo` / `memorytree-workflow` 的报告构建逻辑，核心是“先清空输出目录，再重写文件”。

这意味着：

- 如果 Caddy 直接长期指向 `Memory/07_reports`
- 同时 heartbeat 又在重建该目录

那么在重建窗口内，可能出现短时间：

- 404
- 页面资源缺失
- 页面内容部分更新、部分未更新

因此，文档中的“Caddy 直接指向 `Memory/07_reports`”只能作为阶段性方案，不应视为最终稳定方案。

更稳妥的方向应是以下二选一：

1. 临时目录构建，构建完成后再做原子切换
2. heartbeat 写入独立 publish 目录，由 Caddy 只指向稳定目录

一句话说，这里的问题不是 Caddy，而是“报告产物更新方式目前不是原子发布”。

## 5. 地址控制原则

本地访问地址应采用“项目级控制”，而不是只有一个全局端口。

原因：

- 一台机器上可能同时存在多个项目。
- Caddy 可能同时承载其他本地 Web 服务。
- 单一全局 `report_port` 对多项目场景不够灵活。

推荐优先级：

1. 项目级域名优先  
   例如：`openmnemo.localhost`

2. 项目级端口作为兜底  
   例如：`http://localhost:10086`

3. 全局端口只作为默认值  
   不应成为唯一控制方式

### 5.1 `localhost`、`127.0.0.1`、局域网 IP 的区别

三者不是互斥关系，可以共存。

- `localhost:10086`  
  面向本机访问，书写友好。
- `127.0.0.1:10086`  
  也是本机访问，但更明确，适合作为默认安全入口。
- `192.168.x.x:10086`  
  面向局域网其他设备访问，例如手机、平板、另一台电脑。

如果同一个服务监听的是所有网卡地址，并且系统防火墙放行，那么同一份报告理论上可以同时通过下面三种入口访问：

```text
http://localhost:10086
http://127.0.0.1:10086
http://192.168.1.99:10086
```

### 5.2 推荐的本地访问优先级

推荐默认策略：

1. 默认入口使用 `127.0.0.1 + 项目端口`  
   最明确，默认只给本机使用。

2. 保留 `localhost + 项目端口` 作为友好别名  
   方便用户记忆和输入。

3. 在需要跨设备访问时，按项目决定是否开放 `局域网 IP + 项目端口`  
   不建议默认无条件开放。

### 5.3 与 Caddy 的关系

Caddy 可以同时支持这三种入口，但具体暴露范围应由项目配置和监听策略决定。

推荐理解：

- 是否监听回环地址，只决定“本机能否访问”
- 是否监听局域网地址，只决定“其他设备能否访问”
- 这不改变 `heartbeat` 的职责

也就是说：

- `heartbeat` 负责更新内容
- Caddy 负责决定“哪些地址可以访问这些内容”

也就是说，未来更合理的模型不是：

```text
全局 report_port = 10086
```

而是：

```text
项目 A -> openmnemo.localhost:10086
项目 B -> another-project.localhost:10087
项目 C -> docs.localhost:10100
```

或者直接基于 host 做区分：

```text
openmnemo.localhost
project-x.localhost
memory-demo.localhost
```

## 6. 对 memorytree-workflow 的复用判断

### 6.1 可以直接借鉴的精华

#### A. heartbeat 不做常驻网页服务

这点是对的，应该保留。

含义：

- `heartbeat` 继续做后台自动化
- 本地访问由独立服务承担

#### B. 自动提交的分支隔离

`memorytree-workflow` 当前实现的核心价值之一，是自动 repo-local commit/push 只在 `memorytree/*` 分支上发生。

这意味着：

- 正常产品开发分支不会被自动 Memory 提交污染
- heartbeat 在非 `memorytree/*` 分支上仍可导入全局 archive，但保持仓库工作分支干净

这套逻辑值得优先复用到 `openmnemo`。

#### C. Windows 下 daemon 静默运行

这对 Windows 用户体验很重要。

收益：

- 计划任务运行时不弹黑窗
- 本地常驻感更强
- 更接近正式产品体验

#### D. GitHub Pages 发布链路

`memorytree-workflow` 最新版已经具备“生成报告后发布到 GitHub Pages”的完整链路，这一点值得直接参考。

应复用的不是“把公网发布单独做一套”，而是：

- 继续沿用同一份 `Memory/07_reports` 输出
- 在 heartbeat 或 report build 阶段，根据项目配置触发 GitHub Pages 发布
- 让本地访问和公网访问共用同一份静态报告产物

补充说明：

- 这里指的是 GitHub 仓库上游 `main` 的最新实现
- 不等于当前机器已安装的 skill 副本一定已经同步到同一版本

因此在参考时，需要区分两层：

1. 上游 `main` 已实现什么
2. 当前本机安装版本实际具备什么

后续若要工程落地，应以上游源码和目标版本为准，而不是默认相信本机已安装副本完全同步。

### 6.2 可以参考，但不应直接照搬的部分

#### A. `report_port`

`memorytree-workflow` 最新版已经把 `report_port` 做进项目级配置模型，这个方向是正确的。

因此建议：

- 直接参考它“项目级 `report_port`”的思路
- 在 `openmnemo` 中进一步扩展为“项目级地址配置”
- 地址配置不只包括端口，还应包括 host、是否允许局域网访问、未来公网基址等

#### B. 项目级发布配置

`memorytree-workflow` 最新版已经把以下能力纳入项目级配置思路：

- `memory_branch`
- `gh_pages_branch`
- `report_base_url`
- `report_port`

这意味着：

- 一个项目可以决定自己的本地访问端口
- 一个项目可以决定自己的公网发布分支
- 一个项目可以决定自己的公网基址

这部分应作为 `openmnemo` 后续演进的重要参考。

#### C. 整套 skill / MemoryTree 规则文本

不建议直接搬入 `openmnemo` 产品本体。

原因：

- `memorytree-workflow` 是工作流工具
- `openmnemo` 是正式产品
- 两者目标不同，边界不同

应复用的是“后台执行逻辑”和“安全约束”，不是整套规则文本。

## 7. 推荐的最终架构

### 7.1 本地形态

推荐本地形态如下：

```text
heartbeat
  -> 周期性更新 Memory 和 Memory/07_reports

Caddy
  -> 常驻提供本地访问入口

Browser
  -> 始终访问稳定地址
```

推荐行为：

- `heartbeat` 开启 `generate_report`
- Caddy 始终指向项目的 `Memory/07_reports`
- 用户固定通过项目级地址访问

### 7.2 公网形态

在本地方案稳定后，再增加公网发布层：

```text
heartbeat -> build report -> 本地静态目录
                           -> GitHub Pages / 静态托管
```

这样本地和公网共用同一份报告产物，避免两套渲染链路。

## 8. 建议的配置方向

下面是未来更合理的配置方向示意，不代表本次已实现：

```toml
heartbeat_interval = "5m"
auto_push = true
generate_report = true

[[projects]]
# 以下字段属于上游 memorytree-workflow 已验证方向
development_path = "D:/demo1/openmnemo"
memory_path = "C:/Users/ai/.memorytree/worktrees/openmnemo"
name = "openmnemo"
memory_branch = "memorytree/openmnemo"
report_port = 10086
gh_pages_branch = "gh-pages"
report_base_url = "https://you.github.io/openmnemo"

# 以下字段属于 openmnemo 可考虑扩展的项目级访问配置
local_host = "openmnemo.localhost"
local_serve_root = "Memory/07_reports"
allow_lan_access = false
```

说明：

- `development_path`：正常产品开发目录
- `memory_path`：Memory 专用工作目录
- `memory_branch`：自动 Memory 提交的专用分支
- `report_port`：项目级访问端口
- `gh_pages_branch`：项目级 GitHub Pages 发布分支
- `report_base_url`：项目级公网访问基址
- `local_host`：项目级访问主机名
- `local_serve_root`：本地静态目录
- `allow_lan_access`：是否允许局域网访问

其中：

- `development_path`、`memory_path`、`memory_branch`、`report_port`、`gh_pages_branch`、`report_base_url` 属于上游已验证方向
- `local_host`、`local_serve_root`、`allow_lan_access` 属于本文为 `openmnemo` 提出的扩展字段，不代表上游现成已有

如果后续采用 Caddy 统一管理，本项目就可以明确告诉 Caddy：

- 这个项目的站点根目录在哪里
- 这个项目希望使用什么 host/port
- 这个项目是否允许被局域网设备访问
- 这个项目是否还要同步到公网地址

## 9. 推荐的落地顺序

### 第一阶段：零代码或极少代码方案

目标：

- 先把访问层跑通
- 不让 heartbeat 承担错误职责

做法：

- Caddy 阶段性常驻服务 `Memory/07_reports`
- heartbeat 只负责生成报告
- 本地先固定使用一个项目地址

限制：

- 该阶段存在“重建目录时瞬时不一致”的已知风险
- 更适合先验证访问模式，不适合直接当作最终稳定发布方案

### 第二阶段：增强 heartbeat 安全性

目标：

- 避免自动 Memory 提交影响正常产品分支

做法：

- 借鉴 `memorytree-workflow` 的 `memorytree/*` 分支隔离逻辑
- 只在专用分支允许 repo-local commit/push

### 第三阶段：把地址配置项目化

目标：

- 支持一台机器多个项目并存

做法：

- 把访问参数升级为项目级配置
- 不再依赖单一全局端口
- 区分“上游已验证字段”和“openmnemo 自定义扩展字段”

### 第三阶段补充：把发布方式稳定化

目标：

- 避免 heartbeat 重建目录时影响 Caddy 正在对外服务

做法：

- 引入临时构建目录
- 或引入稳定 publish 目录
- 构建完成后再切换对外暴露目录

### 第四阶段：增加公网发布

目标：

- 本地和公网共用同一套报告输出

做法：

- 继续复用 `generate_report`
- 接入 GitHub Pages 或其他静态托管

---

## 10. 设计原则总结

这套方案的核心不是“让 heartbeat 开网页”，而是：

- 让 heartbeat 负责内容更新
- 让 Caddy 负责长期访问
- 让地址控制回到项目级
- 让自动提交受分支隔离保护
- 让本地与公网共用同一份静态报告产物

一句话总结：

> `heartbeat` 是后台更新器，`Caddy` 是常驻前台，项目级地址配置是多项目扩展的关键。
