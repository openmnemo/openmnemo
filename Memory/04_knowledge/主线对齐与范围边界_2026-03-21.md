# 主线对齐与范围边界

> 版本：v0.1  
> 日期：2026-03-21  
> 范围：对齐 2026-03-21 豆包新想法与当前 OpenMnemo 主线程，不包含本次产品代码改动

---

## 1. 背景

用户提供了一份新的豆包思路稿，核心包括：

- 将 OpenMnemo 从“记忆归集工具”继续推进为“个人 AI 长期记忆中枢”。
- 把原始对话拆成原子化记忆单元，而不是只按整段 session 存储和检索。
- 将知识图谱从展示层升级为检索层，用于跨项目、跨平台、跨主题联想。
- 增加统一对话入口，让 AI 在回答时主动复用用户历史记忆。
- 延伸出分区隔离、记忆包分享、付费授权、防盗版、链上存证等长期商业化想法。

同时，Claude 最近主线程收尾强调：当前重点不是继续扩功能，而是先完成 SQLite 迁移收口，修复 review 已确认的问题，再决定是否继续 1.4。

---

## 2. 对齐结论

核心判断：

- 技术与产品大方向相符。
- 当前执行顺序不相符。

一句话说：

新想法可以作为现有 Phase 0 / Phase 1 的“上层愿景和设计牵引”，但不能抢在 SQLite 迁移收口之前成为主线执行焦点。

---

## 3. 与当前主线相符的部分

以下内容与 active goal 已有路线高度一致，可以继续保留：

- `better-sqlite3` 作为 Phase 0 单一引擎。
- `sqlite-vec` + SQLite 图关系表 + FTS 组成三路混合检索。
- 报告页内嵌 AI Chat，作为后续独立 Web 客户端的 0 号原型。
- 通过统一入口查询“自己的全部记忆”，而不是只做 transcript 归档。
- 通过结构化抽取、关系建模、主动召回，提升跨 session 记忆复用能力。

这些点与当前 goal 中的以下方向一致：

- Phase 0 基于 `better-sqlite3` 的三路检索。
- `SearchAdapter` / `VectorAdapter` / `GraphAdapter` + `factory.ts`。
- report serve 的 `/api/chat` 与嵌入式 chat widget。
- Phase 1 再把 chat 从纯 FTS RAG 升级为三路混合检索驱动。

---

## 4. 当前真正的执行焦点

虽然上层想法是相符的，但当前仓库状态说明主线仍应继续收口：

- `repo_mirror_enabled` 目前仍以 `TEXT` 列存储，而不是布尔语义对应的整数列。
- `vector_backend` / `graph_backend` 配置已经进入 config，但默认后端并未真正落到 factory。
- 相关测试虽然已有一定覆盖，但对 boolean/fallback 这类边界仍需要更明确的回归保护。

因此，当前最稳妥的执行顺序应是：

1. 先完成 1.0 SQLite 迁移的复盘修补。
2. 再收口 1.3 factory 与默认后端接线。
3. 再继续 1.4 / 1.5 / 1.7 等向量、图、混合检索能力。
4. 最后再把统一入口和主动记忆唤醒做成更完整的产品体验。

---

## 5. 可以现在吸收进主线的想法

以下内容可以视为“当前路线的设计牵引”，但不必单独起大范围新目标：

- 原子化记忆单元  
  从整段 transcript 继续下钻到 fact / entity / relation / conclusion / preference 等更小粒度。

- 知识图谱从展示层走向索引层  
  图不只是报告页可视化，而要成为检索与关联召回的一部分。

- 统一对话入口  
  AI Chat 不只是问答 UI，而是用户调用全量记忆的统一入口。

- 主动关联而非被动搜索  
  后续回答应能基于历史偏好、历史决策、历史踩坑记录做主动提示。

这些内容更适合作为：

- 1.6 抽取层的设计原则
- 1.7 混合检索的排序目标
- 1.10 / 1.11 AI Chat 的产品体验目标
- Phase 1 Reasoning Engine 的问题定义

---

## 6. 应后置保留的想法

以下内容有价值，但不适合现在进入 active implementation：

- 四大分区 / 传承分区 / 更细粒度的人格或身份隔离
- 记忆包分享
- 付费授权运行
- 防盗版与设备绑定
- 文本水印、审计、授权封禁
- 链上存证、智能合约、版税分账

原因不是方向错，而是它们已经超出当前“Phase 0 数据层夯实”的边界，进入协作层、商业化层，甚至新的产品线。

---

## 7. 当前不建议做的事

在 1.0 收口之前，不建议：

- 因为上层愿景成立，就新增一批“记忆包 / 授权 / 防盗版”实现任务到 active todo。
- 让 AI Chat 的产品外延反过来驱动底层 schema 持续变更。
- 在 factory、默认适配器、数据库类型语义未收稳之前，继续往前堆后端种类。

否则很容易出现：

- 目标越来越大，但阶段产物不可用
- 配置已暴露、默认路径未打通
- chat 入口存在，但底层召回质量与类型安全不足

---

## 8. 建议的阶段表达

为了保持路线稳定，可以把这份新想法改写成下面这种阶段表达：

- Phase 0：把“记忆不丢”升级为“记忆可检索、可关联、可通过统一入口访问”。
- Phase 1：把“统一入口”升级为“能主动复用用户历史决策与偏好的记忆代理”。
- Phase 2 以后：再考虑多用户隔离、记忆分享、授权、商业化与防盗版体系。

这样既不否定新想法，也不打断当前主线程的工程收口节奏。

---

## 9. 本次结论

本次豆包新想法应被视为：

- 对当前主线“相符”的上层产品愿景
- 对当前执行“后置吸收”的设计输入
- 对 active todo “暂不直接纳入”的商业化分支

当前最重要的仍然是：

- 先把 SQLite 迁移和复盘里已确认的真实问题修平
- 再让三路检索与 AI Chat 按既定 Phase 0 路线继续推进
