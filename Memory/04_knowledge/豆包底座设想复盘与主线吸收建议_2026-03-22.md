# 豆包底座设想复盘与主线吸收建议

> 版本：v0.1
> 日期：2026-03-22
> 范围：基于 2026-03-21 豆包长对话，对齐当前 OpenMnemo 主线，并补充 `Git commit` 边界与 `memory_unit` 设计建议

---

## 1. 结论先行

这份豆包对话不是“要不要改主线”的问题，而是“如何把更远的产品愿景安全吸收到现有主线”。

本次核心结论：

- 豆包方案的大方向与当前主线一致。
- 豆包方案的抽象层级明显高于当前阶段。
- 其中最值得立即吸收的，是：
  - 原子化记忆单元
  - 图谱从展示层走向检索层
  - 统一对话入口
  - 外部资料进入统一底座
  - 压缩归档锚点
- 其中最需要明确边界的，是：
  - `Git commit` 的职责
  - `branch` 与权限隔离的关系
  - “由大模型自由决定检索逻辑”的边界

一句话总结：

`Git commit` 很重要，但它更适合作为 **事件层 / 溯源层 / 第一层粗筛入口 / 归档锚点层**，不应直接取代运行时的通用记忆单元。

---

## 2. 和当前主线真正对齐的部分

以下方向与当前 goal / todo 高度一致，可以继续作为主线牵引：

- `better-sqlite3` 作为 Phase 0 单一引擎
- `FTS + vector + graph` 的三路混合检索
- AI Chat 作为统一记忆入口
- heartbeat 作为后台同步与提取调度器
- 通过结构化提取，把 transcript 从“归档文件”推进到“可调用记忆”

也就是说，豆包方案最有价值的部分，并不是推翻现有路线，而是把现有路线继续向上抽象了一层。

---

## 3. 为什么要单独讨论 `Git commit` 的边界

豆包方案里最容易让人兴奋、也最容易走偏的点，是把 `Git commit` 同时抬成：

- 第一层检索入口
- 版本历史
- 知识图谱骨架
- 隐私隔离边界
- 记忆分享封装层
- 压缩归档载体

这些职责里，有些是合理的，有些是危险的。

问题不在于 `commit` 能不能参与记忆系统，而在于：

> 它到底应该是“主记忆对象”，还是“围绕主记忆对象的事件与索引结构”？

本次判断倾向后者。

---

## 4. 为什么 `Git commit` 不适合直接成为通用主运行时记忆单元

### 4.1 `commit` 表达的是“变化事件”，不是“知识对象”

`commit` 天然描述的是：

- 这次改了什么
- 为什么改
- 和上一次相比发生了什么变化

但运行时记忆更关心的是：

- 这条知识本身是什么
- 它适用于什么场景
- 它和哪些实体 / 项目 / 约束 / 偏好有关
- 它能否被单独召回、单独加权、单独修正

也就是说：

- `commit` 是事件视角
- `memory_unit` 是知识视角

两者可以强关联，但不应该混为同一对象。

### 4.2 粒度不匹配

一个 `commit` 往往会同时包含：

- 一个决策
- 一个待办
- 一个约束
- 一条事实
- 一段上下文说明

如果把整个 `commit` 当成主记忆单元，就会出现：

- 预算、时间、偏好、风险被打包在同一个对象里
- 检索某一个细节时，必须拉出整包上下文
- 同一知识点会在多个 commit 中重复出现

这会让系统得到的是“变化记录包”，而不是“可复用知识对象”。

### 4.3 稳定标识不匹配

运行时记忆对象最好有可稳定 `upsert` 的 ID。

例如：

- 同一条偏好可以补充标签
- 同一条约束可以调整权重
- 同一条决策可以被后续结论覆盖

但 `commit hash` 是不可变、一次性的。

只要正文或元数据发生变化，hash 就变了。

这意味着：

- 同一条知识的修订会产生多个事件副本
- 向量与图的挂载对象会不断分裂
- “纠错 / 覆盖 / 合并”会比“新增”难很多

所以 `commit` 更适合 append-only 的事件历史，不适合作为通用运行时对象 ID。

### 4.4 权限粒度不匹配

`branch` 适合做大区分，例如：

- work
- family
- public

但运行时权限真正需要的粒度通常更细：

- 某条记忆只能脱敏后跨区调用
- 某个任务只允许访问某类事实，不允许访问整个分区
- 同一文档可公开其中几段，不可公开其他几段

因此：

- `branch` 可以参与隔离
- 但不能等价于运行时权限模型

### 4.5 图谱与向量更适合挂在“知识单元”上

图谱最自然描述的是：

- 项目 uses 技术
- 决策 depends_on 约束
- 方案 solves 需求
- 偏好 applies_to 场景

这些关系的主节点通常不是 `commit`，而是：

- `Fact`
- `Decision`
- `Constraint`
- `Preference`
- `Task`
- `Concept`
- `DocumentChunk`
- `Session`

同理，向量层通常更适合挂在：

- 语义完整的知识块
- 经过标准化提炼的摘要
- 可单独复用的结构化片段

而不是直接挂在“事件记录包”上。

### 4.6 外部资料一进来，`commit` 的边界会立刻暴露

对于 PDF / 网页 / 视频 / 音频：

- `commit` 很适合记录“这次上传 / 这次解析 / 这次修订”
- 但运行时真正需要检索的，通常是：
  - 文档版本
  - 章节块
  - 时间戳片段
  - 提炼出的观点
  - 与现有记忆的关联

如果强行让每个运行时对象都以 `commit` 为中心，会让 Git 历史被机器生成日志淹没，也会让运行时检索粒度过粗。

---

## 5. 现有代码已经隐含的正确边界

当前仓库实现其实已经站在更健康的位置上：

- `commit_layer` 被设计为 Layer 1 粗筛入口
- `content` 被设计为完整内容层
- `VectorAdapter` 要求稳定 `id + embedding + metadata`
- `GraphAdapter` 要求节点 / 边，而不是提交历史本身

换句话说，当前代码已经在表达一件事：

> `commit` 应当参与检索，但不应独占运行时主对象地位。

这和本次分析结论一致。

---

## 6. `Git commit` 的正确职责边界

本项目里，建议把 `commit` 固定在以下 4 类职责中：

### 6.1 第一层粗筛入口

适合回答：

- 最近这个项目改过什么
- 某个想法大概在哪个阶段出现
- 某个问题和哪些文件 / 模块相关

### 6.2 版本溯源与审计层

适合回答：

- 这条记忆 / 这份资料是什么时候进入系统的
- 后来是谁修订了它
- 哪些知识是后来覆盖旧结论的

### 6.3 压缩归档锚点

适合在上下文压缩后留下：

- 一条摘要
- 一个归档锚点 ID
- 若干关联对象

压缩后的锚点可以引用 `commit`，但压缩内容本体仍应回到 `memory_unit` / `archive_anchor`。

### 6.4 分享 / 导出 / 传承的封装边界

适合把一组知识对象封装为：

- 一个可追溯版本包
- 一个可增量同步的导出单元

也就是说，`commit` 很适合做“包”和“版本”，不适合直接做所有运行时知识的最小单位。

---

## 7. 推荐的底座分层

为了同时保留 Git 的优势、避免 Git 过载，建议把底座固定为四层：

### 7.1 源事实层

保存未经破坏的原始来源：

- raw transcript
- raw docs
- raw assets
- clean transcript / extracted text

这是最终证据层，不应被运行时摘要替代。

### 7.2 运行时记忆层

保存真正供召回与推理使用的对象：

- `session`
- `memory_unit`
- `document_chunk`
- `entity`
- `relation`
- `archive_anchor`

这是后续 `DataLayerAPI` 真正应该围绕的层。

### 7.3 索引层

为运行时层提供不同视角的检索：

- FTS
- vector
- graph
- commit_index

### 7.4 版本 / 审计层

保存变化事件与历史轨迹：

- git commit
- git tag
- branch

---

## 8. 推荐的 `memory_unit` 设计原则

`memory_unit` 不应等于“随便切一小段文本”，而应满足以下原则：

### 8.1 语义单一

一个 unit 尽量只表达一类可复用信息，例如：

- 一个事实
- 一个决策
- 一个约束
- 一个偏好
- 一个待办
- 一个灵感

### 8.2 可单独检索

它应能单独：

- 做 FTS
- 做 embedding
- 做 graph 关联
- 做权重调整

### 8.3 可回溯

每个 unit 都必须能追溯到：

- 来源 session / 文档 / 视频
- 来源位置
- 提取时间
- 关联 commit / import / heartbeat 事件

### 8.4 可修订

unit 应允许：

- 被新结论覆盖
- 被后续提取补充字段
- 被标记为过期 / 无效 / 已完成

### 8.5 可权限化

unit 必须带有足够的范围信息，支持未来做：

- project scope
- partition scope
- visibility
- redact policy

---

## 9. 推荐的 `memory_unit` 对象结构

下面是一个偏 Phase 0 / 1 的实用结构，不追求一步到位，但要给后续扩展留口：

### 9.1 核心字段

- `id`
  - 稳定主键，不等于 commit hash
- `unit_type`
  - 建议枚举：
  - `fact`
  - `decision`
  - `constraint`
  - `preference`
  - `task`
  - `insight`
  - `summary`
  - `document_chunk`
  - `archive_anchor`
- `title`
  - 简短标题，供粗筛与展示
- `body`
  - 语义完整正文，供向量化与精细召回
- `summary`
  - 可选短摘要，便于结果拼装
- `project`
  - 项目归属
- `partition`
  - 所属分区，未来用于隔离
- `source_kind`
  - `transcript | document | video | audio | web | manual | heartbeat`
- `source_id`
  - 对应源对象 ID
- `source_ref`
  - 文件路径 / 页码 / 时间戳 / message span
- `commit_ref`
  - 可选，记录相关 commit 事件
- `confidence`
  - 提取可信度
- `weight`
  - 召回权重
- `status`
  - `active | superseded | archived | done | hidden`
- `created_at`
- `updated_at`

### 9.2 关联字段

- `entity_refs`
  - 指向提取出的实体
- `related_unit_ids`
  - 与其他 unit 的显式关联
- `supersedes`
  - 覆盖哪些旧 unit
- `superseded_by`
  - 被哪些新 unit 覆盖
- `tags`
  - 辅助过滤

### 9.3 不建议一开始就塞满的字段

以下可以后置：

- 商业价值评分
- 自动变现建议
- 长链路自治执行状态
- 分身人格快照

这些更适合 Phase 1 以后。

## 10. `memory_unit` 与其他对象的关系

### 10.1 与 `session`

- `session` 是来源容器
- `memory_unit` 是从 `session` 中提炼出的运行时对象

一个 session 可产出多个 unit。

### 10.2 与 `commit`

- `commit` 是某次导入 / 归档 / 修订的事件记录
- `memory_unit` 是事件影响到的知识对象

一个 commit 可关联多个 unit，一个 unit 也可出现在多个 commit 演化链中。

### 10.3 与 `document_chunk`

- `document_chunk` 可以视为一种 `memory_unit`
- 也可以单独保留为更原始的内容层对象

Phase 0 更建议先把它视为 `memory_unit` 的一个 `unit_type`，保持实现简洁。

### 10.4 与 `archive_anchor`

- `archive_anchor` 是压缩 / 归档后留下的轻量运行时入口
- 它不是原文，也不是单纯 commit
- 它应同时指向：
  - 归档摘要
  - 相关 unit 集合
  - 原始内容来源
  - 对应 commit / tag

## 11. 对 1.6 / 1.7 / AI Chat 的直接影响

这份分析最应该反哺的，不是远期商业化，而是当前主线后续设计：

### 11.1 对 1.6 提取层

提取层不应只产出“embedding + entity + relation”。

更合理的是统一产出：

- `memory_unit`
- `entity`
- `relation`
- `archive_anchor`

### 11.2 对 1.7 三路混合检索

未来更稳的混合检索对象应逐步从“session 行”过渡到：

- unit 级召回
- unit 聚合回 session / document / project

### 11.3 对 AI Chat

统一入口的上下文不应长期只由：

- title
- commit_layer
- transcript content

组成。

后续应逐步升级为：

- 灵根 / 全局原则
- 高权重 `memory_unit`
- 必要的 source evidence
- 关联 entity / relation
- 可回源的 archive anchors

---

## 12. 当前最值得落地的设计前置

在不打断当前主线节奏的前提下，最值得提前固定的不是“记忆包”“传承市场”，而是下面 4 个基础对象：

- `memory_unit`
- `source_assets`
- `archive_anchor`
- `retrieval_tools`

这 4 个对象一旦边界明确，后续的：

- 1.6 提取层
- 1.7 混合检索
- 1.10 / 1.11 AI Chat
- 外部资料接入
- 压缩归档

都会明显更稳。

---

## 13. 本次建议

本次不建议把豆包方案直接改写成 active implementation。

更稳妥的吸收方式是：

1. 保持当前 Phase 0 主线不变。
2. 把“commit 的正确职责边界”固定为设计原则。
3. 把 `memory_unit / source_assets / archive_anchor / retrieval_tools` 固定为 1.6 前置设计对象。
4. 后续在 1.6 中正式定义提取规范，在 1.7 中正式接入运行时召回。

一句话落地表达：

> 保留 Git 的强项，但不要让 Git 承担所有运行时职责；让 `memory_unit` 成为记忆底座的主对象，让 `commit` 成为围绕它工作的事件与溯源层。
