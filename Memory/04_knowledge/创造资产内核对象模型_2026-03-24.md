# 创造资产内核对象模型
> 日期：2026-03-24
> 目标绑定：`goal_v003_20260324.md`

---

## 1. 核心判断

OpenMnemo 未来的开源核心，不应只被定义为“记忆底座”，而应被定义为：

**创造资产内核**

它要统一沉淀三类资产：

- `创作结果` -> `ProductAsset`
- `创作过程` -> `MemoryAsset`
- `创作能力` -> `SkillAsset`

这三类资产不是三套孤立系统，而是同一套 Git-native 资产体系中的三种对象类型。

---

## 2. 共享基类：BaseAsset

三类资产都应共享一套最小公共字段。

### 2.1 建议字段

- `id`
- `asset_type`
- `project_id`
- `scope_id`
- `title`
- `summary`
- `status`
- `tags`
- `source_refs`
- `session_refs`
- `commit_refs`
- `created_at`
- `updated_at`
- `privacy_level`
- `version`
- `lineage`

### 2.2 设计含义

- `source_refs`：回指原始对话、原始文件、原始文档、原始工作流结果
- `session_refs`：回指形成该资产的对话会话
- `commit_refs`：回指代码或文档变更事实
- `lineage`：记录该资产由什么资产演化而来

---

## 3. ProductAsset

`ProductAsset` 对应创作结果。

它不只代表“代码文件”，而是泛指自然语言创作最终落地的产品性产物。

### 3.1 可包含的对象

- 文件
- 模块
- API
- 文档
- 配置
- 数据集
- 发布物
- workflow 运行产物

### 3.2 建议字段

- `product_kind`
- `repo_uri`
- `repo_path`
- `branch`
- `commit_range`
- `artifact_format`
- `owner_module`
- `build_state`
- `related_memory_ids`
- `related_skill_ids`

### 3.3 它解决的问题

- 一个项目最终产出了什么
- 这些产物位于仓库的什么位置
- 这些产物与哪些对话、决策、技能相关

---

## 4. MemoryAsset

`MemoryAsset` 对应创作过程。

它是这次规划的核心，因为“记忆树 + 探针检索”主要服务于它。

### 4.1 可包含的对象

- 问题
- 决策
- 约束
- 偏好
- 尝试
- 失败
- 洞见
- 里程碑
- 原始问题演化链

### 4.2 建议字段

- `memory_kind`
- `tree_level`
- `anchor_text`
- `raw_refs`
- `parent_id`
- `child_ids`
- `branch_key`
- `evidence_refs`
- `related_product_ids`
- `related_skill_ids`
- `probe_policy`

### 4.3 核心原则

- 不是删除原文，而是压缩进入上下文的路径
- 原始内容必须保留
- 树上节点用于导航、压缩、定位、回填
- 召回采用“命中即止”的渐进展开

---

## 5. SkillAsset

`SkillAsset` 对应创作能力。

它不是“对话里某句建议”，而是经过复盘后，能重复调用的稳定能力对象。

### 5.1 可包含的对象

- 方法论
- checklist
- prompt 模板
- workflow 模板
- 解决模式
- 反模式
- 调试套路
- 评审套路

### 5.2 建议字段

- `skill_kind`
- `trigger_conditions`
- `prerequisites`
- `steps`
- `inputs`
- `outputs`
- `success_signals`
- `anti_patterns`
- `evidence_refs`
- `derived_from_memory_ids`
- `applies_to_product_ids`
- `confidence`

### 5.3 它解决的问题

- 这次创作最后沉淀出了什么能力
- 这些能力是从哪些对话和结果里提炼出来的
- 下一次遇到类似问题，如何直接复用

---

## 6. 三类资产之间的关系

建议至少支持以下关系：

- `MemoryAsset -> ProductAsset`
  - `LEADS_TO`
  - 表示某段过程最终导向了某个产物

- `SkillAsset -> ProductAsset`
  - `APPLIES_TO`
  - 表示某个技能可用于某类产物

- `SkillAsset -> MemoryAsset`
  - `EXTRACTED_FROM`
  - 表示技能是从某段过程提炼出来的

- `MemoryAsset -> MemoryAsset`
  - `EVOLVES_TO`
  - 表示问题、决策、思路的演化

- `ProductAsset -> ProductAsset`
  - `DEPENDS_ON`
  - 表示模块、文件、发布物之间的依赖关系

---

## 7. 记忆树最小结构

建议将 `MemoryAsset` 的树形结构明确为四层：

- `ConversationLeaf`
  - 最小对话叶子
  - 对应 1-3 轮或一个最小主题片段

- `ArchiveBatch`
  - 一组叶子节点的阶段性归档

- `MilestoneNode`
  - 一个阶段的核心里程碑

- `StageNode`
  - 更高层的阶段主线节点

### 7.1 作用分工

- `Leaf`：保留局部具体语境
- `Batch`：做小范围收敛
- `Milestone`：形成阶段主线
- `Stage`：形成长期导航主干

---

## 8. 探针式检索最小接口

建议最小接口为：

- `probe_main(query, root_ref)`
  - 先拿主干节点，`root_ref` 可为 `session_id / repo_id / scope_id`

- `probe_branch(node_id, query?)`
  - 展开某个里程碑、stage、workstream 下的分支

- `probe_leaf(node_id)`
  - 查看具体叶子、小归档或 artifact leaf

- `hydrate_raw(ref)`
  - 必要时回填原始对话或原始文档

- `probe_repo_main / probe_workstream`
  - 可作为内部 wrapper，但不建议成为公共最小接口

### 8.1 检索原则

- 先主线，后分支，再叶子，最后原文
- 命中即停
- 不为“可能有用”而全量灌入
- 让上下文负载稳定，不随历史轮数线性膨胀

---

## 9. 与当前代码现实的映射

当前已有对象可以看作未来三类资产模型的前身，而不是错误方向。

### 9.1 现状映射

- `memory_unit`
  - 更接近 `MemoryAsset` 的叶子层或结构化记忆单元

- `source_asset`
  - 更接近来源/证据层的原始载体对象
  - 当前主要是 transcript source，不等同于未来 `ProductAsset`

- `archive_anchor`
  - 更接近 `MemoryAsset` 中的锚点/里程碑节点

### 9.2 这意味着

当前代码基础可以继续沿用，但后续命名、关系、接口应逐步从“记忆对象”升级为“创造资产对象”。

### 9.3 ProductAsset 的当前缺口

- 当前代码中还没有与 `ProductAsset` 一一对应的一等运行时对象
- 未来 `ProductAsset` 更适合来自 repo/file/doc/workflow artifact 的独立导入与聚合
- 不应直接把当前 transcript `source_asset` 误当成 `ProductAsset`

### 9.4 与当前 DataLayerAPI 的兼容桥

三类资产模型在短期内不应直接替换当前 `RetrievalReference kind`。

- `MemoryAsset`
  - 短期先投影到 `memory_unit + archive_anchor + session context`

- `source_asset`
  - 继续作为来源/证据对象存在

- `ProductAsset`
  - 短期先通过 repo path / commit context / 文档导入等聚合视图表达

- `SkillAsset`
  - 短期先作为从 `MemoryAsset` 派生出来的稳定对象

- 当前 `DataLayerAPI`
  - 继续暴露 `session / memory_unit / source_asset / archive_anchor / mixed`

- 后续资产级接口
  - 应优先作为 `DataLayerAPI` 之上的投影层或单独的 `AssetLayerAPI`
  - 不应在没有映射关系的前提下平行创造第二套运行时 kind

---

## 10. 对开源核心边界的启发

从该对象模型出发，开源核心应该至少包含：

- 三类资产对象模型
- 统一导入与提炼
- Git/Gitea 回写
- 记忆树生成与探针检索
- FTS / vector / graph / probe 检索层
- 面向上层产品的最小运行时接口

而不应急于直接吸收：

- 完整 deer-flow 式 super agent harness
- 重型工作台 UI
- 商业化协作系统
- 市场、计费、SaaS 托管层

---

## 11. 一句话总结

OpenMnemo 的下一阶段，不是单纯把“记忆做得更强”，而是要把：

**创作结果、创作过程、创作能力**

统一变成可拥有、可检索、可复用、可演化的 Git-native 长期资产。
