# MemoryTree 统一记忆仓库方案说明

## 背景

当前讨论的目标不是单纯给某一个项目加一个 `Memory/` 目录，而是要逐步形成一套长期可用的个人记忆资产系统。这个系统需要同时满足几件事：

1. 所有聊天记录和记忆内容都能长期保存。
2. 这些数据可以通过 GitHub 做同步和备份。
3. 后续可以做统一检索和统一向量化。
4. 每个项目又能保留只属于自己的记忆视图。

用户已明确修正一个点：

- 不按模型分层。
- 只按工具分层，例如 `codex`、`claude`、`gemini`。


## 核心判断

### 我认同的部分

以下方向是合理的：

1. 个人记忆资产需要统一保管。
2. 所有聊天内容都应该有完整总备份。
3. 后续统一索引和统一向量化是重要方向。
4. 目录结构需要按项目、按工具、按 `raw/clean/manifests` 分层。

### 我不认同的部分

我不建议把“全量所有项目记忆”复制到“每一个项目仓库的 `memorytree/*` 分支”里。

原因：

1. 同一份内容会在多个项目仓库里重复出现。
2. 重复语料会让向量索引质量下降，召回结果容易被重复内容淹没。
3. 某一个项目仓库权限变化时，其他项目记忆也会被连带暴露。
4. 后续如果要删除、修正、脱敏，需要同时改很多份副本，维护成本很高。

所以，“统一索引”不等于“复制到每个项目仓库”。


## 推荐方案

我建议使用两层存储结构，而不是每个项目都保存全量副本。

### 第一层：总记忆仓库

建立一个专门的私有仓库，例如：

```text
memorytree-vault
```

这个仓库是个人记忆资产的权威总库，保存所有项目的完整内容。

### 第二层：项目镜像分支

每个业务仓库内部仍然可以有自己的 `memorytree/*` 分支，例如：

```text
openmnemo
  branch: memorytree/openmnemo
```

但这个项目分支只保存：

- 属于 `openmnemo` 这个项目的记忆内容
- 不保存其他项目的全量记忆


## 通俗类比

可以把它理解成三样东西：

1. 业务仓库
   - 比如 `openmnemo`
   - 这是项目施工现场

2. 项目镜像分支
   - 比如 `memorytree/openmnemo`
   - 这是这个项目自己的档案柜

3. 总记忆仓库
   - 比如 `memorytree-vault`
   - 这是你的总档案馆

项目档案柜可以有，但总档案馆不应该复制到每一个项目档案柜里。


## 为什么这套方案更适合统一索引

统一索引真正需要的是：

1. 一个唯一的权威数据源。
2. 一套统一元数据。
3. 一套统一的索引过程。

这三个条件都更适合放在总记忆仓库里完成。

如果把同一份记忆复制到多个项目仓库里，再去做统一索引，会遇到：

1. 大量重复数据。
2. 去重逻辑变复杂。
3. 项目边界不清楚。
4. 索引和权限管理混在一起。

因此：

- 统一向量索引应该基于总记忆仓库。
- 项目仓库的 `memorytree/*` 分支应该是项目镜像，而不是全量总库。


## 目录结构建议

### 总记忆仓库 `memorytree-vault`

```text
memorytree-vault/
  projects/
    openmnemo/
      transcripts/
        raw/
          codex/2026/03/*.jsonl
          claude/2026/03/*.jsonl
          gemini/2026/03/*.jsonl
        clean/
          codex/2026/03/*.md
          claude/2026/03/*.md
          gemini/2026/03/*.md
        manifests/
          codex/2026/03/*.json
          claude/2026/03/*.json
          gemini/2026/03/*.json
      memory/
        goals/
        todos/
        chat-logs/
    another-project/
      ...
  catalog/
    sessions.jsonl
```

这里的层级是：

1. 按项目分
2. 项目内按内容类型分
3. 内容类型下按工具分
4. 再按年月分

### 项目仓库中的镜像分支

例如 `openmnemo` 仓库里的 `memorytree/openmnemo`：

```text
openmnemo
  branch: memorytree/openmnemo
    Memory/
      01_goals/
      02_todos/
      03_chat_logs/
      06_transcripts/
        raw/
          codex/
          claude/
          gemini/
        clean/
          codex/
          claude/
          gemini/
        manifests/
          codex/
          claude/
          gemini/
```

这个分支里只保留 `openmnemo` 本项目的内容。


## 数据流建议

推荐的数据流是：

1. 用户在任意项目、任意代码分支中与 AI 对话。
2. 新 transcript 先进入总记忆仓库对应的项目目录。
3. 如果该 transcript 属于当前项目，再镜像到当前项目仓库的 `memorytree/*` 分支。
4. 统一搜索和统一向量索引只从总记忆仓库构建。
5. 项目内 PR、审查、同步则走本项目的 `memorytree/*` 分支。

这样分工后：

- 总记忆仓库负责“完整总账”
- 项目镜像分支负责“项目视图”


## 关于“一个项目是否要有两个记忆仓库”

不是。

更准确地说：

1. 你个人只需要一个总记忆仓库 `memorytree-vault`
2. 每个业务项目本来就有自己的业务仓库
3. 业务仓库里可选有一个 `memorytree/*` 分支

所以不是“每个项目建两个记忆仓库”，而是：

- 一个全局共用的总记忆仓库
- 多个业务项目各自保留自己的项目镜像分支


## 最终结论

公允判断如下：

1. “个人记忆资产统一保管，并可同步到 GitHub”这个目标是合理的。
2. “未来做统一向量索引”这个目标也是合理的。
3. 但不应该把所有项目的全量记忆复制到每一个项目仓库的 `memorytree/*` 分支。
4. 更合理的做法是：
   - 用一个专门的总记忆仓库保存全量内容
   - 用各项目自己的 `memorytree/*` 分支保存各自镜像
   - 统一索引从总记忆仓库构建

一句话概括：

```text
总记忆仓库负责全量数字资产，项目 memorytree 分支负责项目镜像，不负责承载所有项目的全量总账。
```
