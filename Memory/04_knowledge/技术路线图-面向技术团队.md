> 面向技术团队 · 版本：v1.1 · 日期：2026-03-17

## 一、产品定位（技术视角）

OpenMnemo 是一个**个人 AI 记忆操作系统**，底层是 Gitea 仓库平台，上层是跨平台记忆捕获、结构化索引、可视化看板、以及未来的智能检索与续写能力。

核心设计原则：
- **Gitea 为底座**：仓库存储层基于 Gitea（开源用户用 GitHub，企业用户自部署 Gitea，两者 API 兼容）
- **本地优先**：所有数据默认存在用户本地，云端是可选项
- **白盒透明**：所有存储格式是 Markdown + JSONL，人可读，工具可解析
- **零锁定**：用户不依赖 OpenMnemo 也能访问自己的数据
- **渐进增强**：从 CLI → 插件 → App，每一层都是完整可用的产品

## 三、Gitea 底座架构

### Gitea 是什么

Gitea 是开源、轻量、可自部署的完整 Git 托管平台，提供仓库、Issues、Wiki、Actions、Webhooks、API 等全套能力。它是 GitHub 的开源替代品，两者 API 高度兼容。

```
用户选择方式（两者功能完全一致）：
  ├── GitHub          开源用户首选，零配置，直接用
  └── 自部署 Gitea    企业/隐私用户，数据不离开自己服务器
                      未来还支持：GitLab、Forgejo 等兼容平台
```

### Gitea 账号体系 = 记忆的语义命名空间

Gitea 有两种主体：**个人账号** 和 **组织（Organization）**。这套体系天然就是记忆的分类系统，不需要额外设计。

```
个人账号 @alice
  │
  ├── 个人仓库（私有，只有自己可见）
  │     ├── learning/        ← 学习成长类对话
  │     ├── health/          ← 健康医疗类对话
  │     ├── personal/        ← 个人生活类对话
  │     └── finance/         ← 财务规划类对话
  │
  ├── 加入/创建 家庭组织 @alice-family
  │     ├── Team: 家庭成员（读写）
  │     ├── household/       ← 家庭事务对话（internal 可见性）
  │     ├── children/        ← 孩子教育相关（internal 可见性）
  │     └── health-shared/   ← 家庭健康记录（internal 可见性）
  │
  └── 加入/创建 公司组织 @my-company
        ├── Team: 工程师（读写）
        ├── Team: 只读（仅查看）
        ├── project-alpha/   ← 工作项目对话（internal 可见性）
        └── tech-research/   ← 技术调研对话（internal 可见性）
```

**关键设计决策：Gitea 的命名空间就是语义分类系统。**
- 个人账号下的仓库 = 个人生活域
- 组织下的仓库 = 工作/家庭/社区上下文
- 团队（Team）= 协作者权限管理

### Gitea 可见性模型

| 可见性级别 | 含义 | OpenMnemo 场景 |
|-----------|------|---------------|
| `private` | 仅仓库成员可见 | 默认，个人隐私记忆 |
| `internal` | 同一 Gitea 实例所有登录用户可见 | 家庭组织内共享、公司内部共享 |
| `public` | 全网可见，无需登录 | 知识输出、公开分享 |

**注意**：`internal` 在 GitHub 上称为 `private`（GitHub 没有 internal 概念，仅在 Organization 中有类似逻辑）。自部署 Gitea 的 `internal` 等于"仅本实例用户"，是企业部署的核心优势。

### Gitea 功能 → OpenMnemo 用途映射

| Gitea 功能 | OpenMnemo 用途 |
|-----------|---------------|
| 仓库 + 版本控制 | 所有对话的版本化存储，可回溯任意 commit |
| Issues | 从对话中提取任务，自动创建 Issue 追踪 |
| Wiki | 知识沉淀区（`04_knowledge/` 对应 Wiki） |
| Actions（CI/CD） | 心跳触发自动构建报告、推送 Pages |
| Webhooks | 实时推送通知到飞书/Telegram/Discord/Slack |
| REST API（兼容 GitHub API） | CLI / 浏览器插件 / App 的统一数据接入层 |
| Gitea Pages | 记忆看板静态部署（与 GitHub Pages 等价） |
| 组织 + 团队 | 家庭/企业的多人协作与精细权限管理 |
| `internal` 可见性 | 家庭/团队内部共享，不对外暴露 |
| Packages（未来） | 导出记忆快照、模型 fine-tune 数据集 |

## 四、记忆存储结构

### 仓库目录约定

```
Memory/                         ← 记忆仓库根目录（git 管理）
  01_goals/                     ← 项目目标文件（Markdown）
  02_todos/                     ← 待办清单（Markdown）
  03_chat_logs/                 ← 实时对话日志（Markdown append-only）
  04_knowledge/                 ← 知识沉淀（Markdown）
  05_archive/                   ← 归档内容（Markdown）
  06_transcripts/
    raw/{client}/{date}/        ← 原始 transcript（JSONL，可选上传）
    clean/{client}/{date}/      ← 清洗后 Markdown（无工具调用事件）
    manifests/{client}/{date}/  ← 结构化元数据 JSON（session 索引）
  07_reports/                   ← 生成的静态网站（gitignore）
  .report-cache/                ← AI 摘要缓存（gitignore）
```

### ManifestEntry 核心字段

```typescript
interface ManifestEntry {
  session_id: string      // 全局唯一 ID
  client: string          // codex | claude | gemini
  started_at: string      // ISO 8601
  title: string           // 会话标题（从首条消息提取）
  message_count: number
  tool_event_count: number
  raw_sha256: string      // 原始文件哈希，用于 AI 摘要缓存 key
  cwd: string             // 工作目录（用于项目分类）
  branch: string          // git branch
  repo_clean_path: string // 相对于仓库根的清洗文件路径
  tags?: string[]         // AI 提取的关键词标签（可选）
}
```

## 五、用户上手流程（Onboarding）

首次运行 `openmnemo init` 时，引导用户完成以下步骤：

```
1. 连接 Gitea/GitHub 账号
   └── 输入 Personal Access Token（PAT）
   └── OpenMnemo 获得仓库创建、读写权限

2. 选择预设记忆仓库分类（可多选，也可自定义）
   └── 可选：工作、学习、家庭、情感、子女、健康、财务
   └── OpenMnemo 在用户账号下自动创建对应的私有仓库

3. 可选：创建/加入组织
   └── 家庭组织：邀请家庭成员，仓库设为 internal
   └── 工作组织：由企业管理员创建，员工加入

4. 设置默认隐私级别
   └── 推荐：private（仅自己可见）
   └── 单仓库可随时覆盖

5. OpenMnemo 开始后台心跳，自动捕获 AI 对话
```

## 六、AI 分类系统设计

### 双阶段分类时机

**阶段一：对话进行中（软提示）**

```
┌─────────────────────────────────────────────┐
│  AI 正在分析对话类型...                        │
│  建议归档到：📁 work / project-alpha           │
│  [继续] [稍后决定]                             │
└─────────────────────────────────────────────┘
```
- 不打断用户，只是非侵入式提示
- 用户可以忽略，系统在对话结束后再处理

**阶段二：对话结束后（推送通知）**

```
📩 新记忆已捕获
   "讨论了 React 状态管理方案"
   建议：📁 work / project-alpha

   [查看] [移动到其他仓库] [撤销（24h 内有效）]
```
- 通过 IM 机器人（飞书/Telegram）或系统通知推送
- 24 小时内可撤销归档决定

**待分类收件箱（Inbox）**

```
Memory/00_inbox/
  ← 用户撤销归档的对话
  ← 无法自动分类的对话
  ← 用户选择"稍后决定"的对话
```
- 定期提醒用户处理收件箱
- 可手动拖拽（CLI 命令 `openmnemo move <session-id> <repo>`）

### AI 分类逻辑

```typescript
// 分类流程
1. 读取 ManifestEntry.cwd（工作目录）→ 匹配已知项目仓库
2. 读取对话首条消息 + 最后一条消息 → 送入 AI 提取主题
3. 与用户已有仓库名做语义匹配（embedding 相似度）
4. 置信度 > 0.8 → 自动归档 + 推送通知
5. 置信度 0.5-0.8 → 推送通知请用户确认
6. 置信度 < 0.5 → 归入 00_inbox/，提醒用户处理
```

## 七、跨仓库搜索架构

### 问题背景

Gitea/GitHub 的原生搜索是**仓库级别**的——用户必须进入某个仓库才能搜索其中内容，无法跨仓库全文检索。

### OpenMnemo 的解法：统一本地 SQLite 索引

```
心跳触发（每 5 分钟）
        ↓
扫描所有仓库的 manifests/ 目录
        ↓
新 session → 解析 clean Markdown → 写入本地 SQLite FTS5 索引
        ↓
openmnemo search "认证方案"
        ↓
SQLite FTS5 全文检索 → 跨仓库、跨客户端统一结果
        ↓（未来）
向量嵌入索引 → 语义搜索（"找我讨论过的状态管理方案"）
```

**SQLite 索引 schema（核心字段）：**

```sql
CREATE VIRTUAL TABLE sessions_fts USING fts5(
  session_id,
  repo,          -- 所属仓库（@alice/work、@alice-family/household）
  client,        -- codex | claude | gemini
  title,
  content,       -- clean markdown 全文
  started_at,
  tags
);
```

**关键优势：**
- 完全本地，零网络延迟
- 跨仓库（个人仓库 + 组织仓库全部索引）
- 跨客户端（Claude / Codex / Gemini 统一检索）
- 未来可加 sqlite-vec 实现向量语义搜索

## 八、外部对话导入架构

### 背景

OpenMnemo 当前只能捕获 CLI 工具（Claude Code / Codex / Gemini CLI）的本地 JSONL 文件。用户在豆包、ChatGPT、Kimi 等平台积累的**历史对话存量**，以及没有 CLI 工具的网页端对话，需要一套独立的导入通道。

### 三层导入策略

```
Layer 1：结构化导出（现在就能做）
  ├── 豆包 TXT       → parseDoubao()
  ├── ChatGPT JSON   → parseChatGPT()
  ├── Kimi / 文心等  → parseKimi() / parseErnie()
  └── openmnemo import --client doubao ./dir/

Layer 2：AI 辅助解析（通用兜底）
  ├── 任意复制粘贴的对话文本
  ├── 无结构的 HTML 导出
  └── openmnemo import --client generic ./file.txt
      → 送入 AI，自动识别 User/AI 角色边界

Layer 3：浏览器插件实时捕获（Phase 2）
  ├── 无需导出，实时拦截 DOM
  └── 覆盖 Claude.ai / ChatGPT / Gemini Web 等无导出功能的平台
```

### 豆包 TXT 格式（已确认可解析）

豆包导出的 TXT 格式已经高度结构化，约 80-100 行 TypeScript 即可完整解析：

```
Title: 对话标题
URL: https://www.doubao.com/chat/xxxxx
Platform: 豆包
Created: 2026-03-16 10:22:17
Messages: 28

User: [2026-03-16 10:22:17]
用户消息内容...

AI: [2026-03-16 15:43:17]
AI 回复内容...
```

### `openmnemo import` 命令设计

```bash
# 单文件导入
openmnemo import --client doubao ./conversation.txt

# 批量导入整个目录
openmnemo import --client doubao ./chat-memo_61_20260316154801/

# 通用格式（AI 辅助识别结构）
openmnemo import --client generic ./any-conversation.txt

# 指定目标仓库和隐私级别
openmnemo import --client chatgpt ./conversations.json \
  --repo work \
  --visibility private
```

执行后自动走完整归档流程：
```
解析原文 → clean Markdown + manifest JSON
       ↓
SHA256 去重（已导入的跳过）
       ↓
AI 提取标签 + 摘要（可选，--no-ai 跳过）
       ↓
git commit + push
       ↓
SQLite FTS5 索引更新
```

### 各平台导出方式

| 平台 | 导出入口 | 格式 | Parser 难度 |
|------|---------|------|------------|
| **豆包** | 对话页 → 导出 | TXT（结构化） | ⭐ 低 |
| **ChatGPT** | 设置 → 数据控制 → 导出 | ZIP 内含 JSON | ⭐ 低 |
| **Kimi** | 对话页 → 导出 Markdown | MD | ⭐ 低 |
| **Claude.ai** | 无原生导出 | 复制文本 | ⭐⭐ 中（AI 辅助） |
| **Gemini Web** | 无原生导出 | 复制文本 | ⭐⭐ 中（AI 辅助） |
| **文心一言** | 无原生导出 | 复制文本 | ⭐⭐ 中（AI 辅助） |
| **语音录音** | 音频文件 | MP3/M4A | ⭐⭐⭐ 高（需 Whisper 转录） |

### 语音录音导入流程（长期）

```
音频文件（豆包语音备忘录 / 手机录音）
        ↓ Whisper API（本地或云端）
原始转录文本（含口语停顿）
        ↓ AI 清洗（去除"嗯、那个、就是说"等口语词）
结构化对话（识别 User/AI 角色边界）
        ↓ 走标准导入流程
clean Markdown + manifest → 归档
```

### 新增 ManifestEntry 字段

```typescript
interface ManifestEntry {
  // ... 现有字段 ...
  import_source?: 'cli' | 'browser' | 'manual' | 'voice'  // 导入来源
  original_url?:  string   // 原始平台链接（如豆包对话 URL）
  original_platform?: string  // 'doubao' | 'chatgpt' | 'kimi' | 'claude-web'
}
```

### 阶段 2：浏览器插件（下一个重点）

**目标：** 捕获网页版 AI 对话，写入记忆仓库

**技术选型：** Chrome Extension Manifest V3 + TypeScript

**架构：**
```
浏览器插件（content script）
    ↓ 拦截 / 读取 DOM
Claude.ai / ChatGPT / Gemini Web 的对话内容
    ↓ 序列化为标准 JSONL（与 CLI 工具格式相同）
    ↓
后台 Service Worker
    ↓ 通过 Native Messaging 或 HTTP
本地 openmnemo daemon（已在运行）
    ↓
写入 Memory/06_transcripts/raw/browser/{date}/
    ↓ 走现有心跳流程
清洗 → manifests → report → push
```

**支持的平台（按优先级）：**
1. Claude.ai
2. ChatGPT（chat.openai.com）
3. Gemini Web（gemini.google.com）
4. 豆包、Kimi、文心一言（中文市场）

**关键挑战：**
- 各平台 DOM 结构不同，需要单独适配 selector
- 流式输出的捕获时机（需要等 AI 回复完成）
- 用户授权与隐私提示（必须明确告知捕获范围）

### 阶段 4：Web 端

**定位：** 比 GitHub Pages 更完整的交互界面，支持编辑、检索、续写

**技术选型讨论：**

| 方案 | 优势 | 劣势 |
|------|------|------|
| Next.js（推荐） | 生态成熟，SSG + SSR 灵活 | 需要服务器（或 Vercel）|
| SvelteKit | 轻量，性能好 | 社区相对小 |

## 十三、开发规范

- **不可变数据**：所有数组/对象更新返回新副本，不用 `.push()` 原地修改
- **错误处理**：每层显式处理，不吞掉错误，UI 层给友好提示，服务层记详细日志
- **输入校验**：在系统边界（用户输入、外部 API 返回、文件内容）做验证，内部不过度校验
- **小文件原则**：单文件 200-400 行为宜，最大不超过 800 行
- **测试覆盖率**：≥ 80%，TDD 优先，先写测试再写实现
- **安全**：所有 HTML 输出必须 `escHtml()`，JSON 嵌入 `<script>` 需转义 `</`，不硬编码密钥
