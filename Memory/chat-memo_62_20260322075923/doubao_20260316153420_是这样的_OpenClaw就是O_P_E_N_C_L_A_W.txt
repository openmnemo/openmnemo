核心设计原则
分层优先级：记忆层严格按照「相关性概率从高到低、token 成本从低到高」排序，浅层优先，深层后置；
渐进式检索：从最浅层开始逐层向下检索，每一层完成后做命中判断，达标立即终止，绝不进入下一层；
命中即终止：设置多维度达标阈值，只要能完整回答用户问题，立刻停止检索，仅返回最小必要信息；
按需加深：仅当浅层结果不满足需求时，才进入下一层，直到信息足够或遍历全部分层；
自动优化：高频使用的深层信息自动上浮到浅层，越用越高效，token 成本越来越低。

第二部分：核心灵魂 —— 渐进式检索流水线（命中即止执行逻辑）
这是和 OpenClaw 原生架构最核心的区别，OpenClaw 是全量索引一次性检索，而这套系统是逐层检索、达标即停，完整执行流程如下：
第一步：靶点解析（定井位 / 取穴）先通过 LLM 解析用户当前问题，提取核心关键词、问题类型、时间范围、所需信息的最小粒度，生成「检索靶点清单」和「命中达标阈值」。例：用户问 “我昨天说的 XX 项目截止日期是多少？”，靶点为「XX 项目 + 截止日期 + 时间范围：昨天」，阈值为「找到明确的日期信息，100% 匹配项目名称」。这一步从根源上避免了盲目检索，是 token 节省的核心前提。
第二步：L0 层瞬时检索（井口探测 / 点刺靶点）仅检索当前会话上下文、上一轮关键信息，token 成本极低。检索后立即做命中判断：如果结果满足「靶点匹配度≥95%，信息完整度 100%」，直接终止全流程，返回结果，绝不进入下一层。例：用户上一轮刚说过项目截止日期，这一轮询问，直接在 L0 命中，零额外 token 消耗。
第三步：L1 层核心检索（浅油层钻探 / 常用穴扎刺）仅当 L0 未命中时，才进入 L1 层，仅检索 MEMORY.md 的核心策展记忆。命中判断：匹配度≥90%，信息完整度≥90%，达标即终止流程。例：用户问 “我的代码规范是什么？”，这类固定偏好信息直接在 L1 命中，无需检索任何历史日志。
第四步：L2 层短期检索（中油层钻探 / 经络循行）仅当 L1 未命中时，才进入 L2 层，检索近 7 天日志与近期会话摘要。命中判断：匹配度≥85%，信息完整度≥85%，达标即终止流程。
第五步：L3 层长期检索（深油层钻探 / 全经络排查）仅当 L2 未命中时，才进入 L3 层，检索历史归档记忆。命中判断：匹配度≥80%，信息完整度≥80%，达标即终止流程。
第六步：L4 层兜底检索（超深地层钻探 / 全身穴位排查）仅当前面所有层都未命中时，才触发全量知识库兜底检索，最大程度避免无效 token 消耗。
第七步：结果输出与记忆自动优化（采油输出 / 行针得气）无论在哪一层终止，都仅返回命中的最小必要信息，而非全量记忆块；同时自动判断是否需要将本次检索到的高频核心信息，提炼上浮到 L1 层核心记忆，下次用户再询问时，直接在浅层命中，无需再次深层检索，实现 “越用越省、越用越快”。
第三部分：关键的命中终止机制（确保不浪费 token 的核心参数）
加权匹配阈值：采用「向量相似度 + 关键词匹配度 + 语义匹配度」的加权分数，越浅层达标线越高，确保只有精准匹配才会停止；
信息完整度判断：通过 LLM 快速判断检索结果是否能完整回答用户问题，只要能完整解决需求，无论分数多少，都可立即终止；
自定义可控配置：支持用户自定义每一层的阈值、最大检索深度、单轮 token 上限，完全可控；
紧急终止机制：累计 token 消耗达到用户设置的上限时，立即终止检索，返回已找到的最优结果，彻底避免 token 爆炸。

② 创作闲聊：L0≥85%、L1≥80%、L2≥75%（中低阈值，宽松匹配）
③ 全量检索：全层级≥60%（低阈值，尽可能召回相关内容）
4. 检索范围锁定	明确「只在什么范围里找」，包括时间范围、文件范围、项目范围、内容类型	没有这个，就会全库扫描，哪怕用户问的是昨天的内容，也会翻 3 年前的日志，token 直接爆炸	① 精准问答：仅检索 2026-03-02 的日志文件
② 代码开发：仅检索「XX 项目」目录下的记忆文件
③ 创作场景：仅检索「短剧世界观」相关的归档记忆
④ 日常闲聊：仅检索近 1 个月的亲子相关对话
5. 硬熔断红线	明确「绝对不能突破的底线」，包括最大检索深度、单轮最大 token 消耗、最大重试次数	没有这个，就会出现无限循环检索、token 无上限消耗、系统失控的问题，是安全底座	① 日常闲聊：最大深度 L2、单轮 token 上限 500、重试 1 次（极致省 token）
② 精准问答：最大深度 L3、单轮 token 上限 1000、重试 2 次（平衡精准与成本）
③ 代码 / 商业分析：最大深度 L4、单轮 token 上限 3000、重试 3 次（优先保证信息完整）
④ 极端场景：可关闭深度限制，全库检索
6. 事实来源要求	明确「内容必须有可追溯的记忆来源」，无来源的内容不计入命中结果，不允许输出	没有这个，就会彻底失去对幻觉的控制，AI 会编造不存在的记忆，整个系统的可信度直接归零	① 精准问答 / 代码开发：必须标注文件名 + 行号，无来源直接作废（最高要求）
② 商业分析：必须有对应的调研记录、数据来源，无来源不计入
③ 创作场景：必须符合用户之前定的设定，允许适度创作，但核心设定必须有来源
④ 日常闲聊：必须来自用户之前的对话记录，禁止编造用户的偏好
关键补充：没有任何场景能跳出这个框架
哪怕是最极端的场景，也只是调整参数，不用改框架：
场景 1：用户说「我要把所有相关的记忆都找出来，不要停」—— 只需要把匹配度阈值调到最低、最大深度拉满、熔断红线关闭，还是在框架内；
场景 2：用户说「随便聊聊，不用太精准」—— 只需要把匹配度阈值调低、最大深度限制在 L2、token 上限压低，还是在框架内；
场景 3：用户的需求非常模糊，只说「帮我看看之前的那个项目」—— 前置定标环节会先触发极简的需求澄清，用 1-2 个封闭式问题锁定核心关键词和范围，再套入框架，不会盲目检索。
三、要不要针对每个项目单独做判断？答案是：绝对不用，用模板化管理一键适配
你完全不需要为每个项目重新设计一套判断机制，只需要做「预设模板 + 项目级参数绑定」，开发量极低，使用起来零门槛：
先预设 5 个通用基础模板，覆盖 99% 的场景你可以提前把最常用的场景参数做成固定模板，用户（包括你自己）不用懂背后的逻辑，直接选就行，开箱即用：
模板 1：默认精准问答模板（高阈值、浅深度、极致省 token，适配 80% 的日常场景）
模板 2：代码开发模板（高完整度、高来源要求、深深度，适配技术项目）
模板 3：创作辅助模板（中阈值、中深度、宽松匹配，适配写作 / 剧本 / 方案）
模板 4：日常闲聊模板（低阈值、浅深度、低 token 上限，适配陪伴场景）
模板 5：全量检索模板（低阈值、最大深度、无熔断，适配归档 / 复盘场景）
项目级定制：只需要 5 分钟微调，不用重写逻辑针对特定的项目，你只需要给这个项目绑定一个基础模板，再微调几个参数即可，比如：
你的「黄金投资分析项目」：绑定「商业分析模板」，把检索范围锁定在「黄金投资」相关的记忆文件，时间范围设为近 6 个月，最大深度到 L4，token 上限 3000；
你的「短剧创作项目」：绑定「创作辅助模板」，把核心关键词锁定为「马年、重生、围棋」，信息完整度要求设为「符合男主重生人设、爽点逻辑」，来源要求设为「核心设定必须有记忆来源」；
你的「日常带娃助理项目」：绑定「日常闲聊模板」，把检索范围锁定在近 3 个月的亲子对话，最大深度 L2，token 上限 500。
模板的自动适配能力：不用手动切换，系统自动匹配你可以再加一层极简的自动分类逻辑：前置定标环节，系统先判断用户的需求类型（精准问答 / 代码 / 创作 / 闲聊），自动加载对应的模板，用户完全感知不到背后的判断机制，零学习成本。
四、是不是做项目之前必须先对齐这 6 个标准？
完全不用，默认模板已经帮你对齐了 90% 的场景，只有特殊定制需求才需要微调。
对于普通用户 / 日常场景：直接用默认的「精准问答模板」，不用做任何提前对齐，开箱即用，系统会自动完成前置定标、逐层校验、闭环兜底，全程透明；
对于特定项目 / 专业场景：只需要在项目启动前，花 5 分钟调整模板的参数，把这个项目的固定规则（比如检索范围、核心关键词、信息完整度要求）提前设好，之后整个项目周期里，所有任务都会自动套用这个配置，不用每次都重新对齐；
对于极端个性化需求：你甚至可以给单轮对话设置临时参数，比如用户说「这次帮我把所有相关的内容都找出来」，系统临时把当前轮次的熔断红线关闭，不用修改项目的默认配置。
五、和 OpenClaw 原生架构的核心差距：为什么我们的方案更高级、更通用？
OpenClaw 原生架构完全没有这套全局通用的判断机制，这是它最核心的短板：
OpenClaw 原生没有「前置定标」环节，每次检索都是全库扫描，不管用户问的是简单问题还是复杂问题，都是一次性加载所有记忆文件，完全没有范围锁定、分层优先级，token 浪费是天生的；
OpenClaw 原生没有「标准化的停止判断逻辑」，它的停止完全靠「Top K 召回数量」，用户要自己手动设置每次召回多少条内容，要么召回不够导致信息不全，要么召回太多浪费 token，而且完全没有「信息完整度校验」，召回的内容能不能解决问题，它根本不管；
OpenClaw 原生没有「项目级的模板化配置」，如果你有多个项目，只能给每个项目单独写 prompt 约束，每次切换项目都要重新写一套规则，非常麻烦，而且 prompt 约束是软的，很容易被模型突破，而我们的方案是把规则做成了底层硬机制，绝对不会突破；
OpenClaw 原生没有「闭环校验机制」，召回内容之后直接塞给模型生成回答，不管内容对不对、全不全，很容易出现幻觉，而我们的方案是先校验内容能不能解决问题，能解决才停止，不能解决就继续挖，从根源上避免了这个问题。
六、最简落地建议
先把底层三级判断机制做成固定内核，不用改，所有项目、所有场景都跑在这个内核上；
把 6 条阈值框架做成可配置的参数模块，提前预设 5 个基础模板，覆盖绝大多数场景；
做一个极简的配置面板，支持给不同项目绑定模板、微调参数，支持单轮对话临时修改配置；
和你之前设计的分层记忆架构、断点续传机制完全打通，形成完整的闭环：模板定规则→分层检索→逐层校验→达标即停→不达标断点续传。
