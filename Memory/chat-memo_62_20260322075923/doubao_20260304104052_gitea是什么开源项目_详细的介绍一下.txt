一、Gitea 是什么
Gitea 是一款轻量级、自托管、开源的 Git 代码托管与 DevOps 平台，对标 GitHub/GitLab，主打极简部署、低资源占用、全功能覆盖
。
开发语言：Go + TypeScript，跨平台、单二进制运行
开源协议：MIT，完全免费、可商用、可二次开发
起源：2016 年从 Gogs 分叉，转为社区驱动开发
定位：个人/小团队/企业自建“私有 GitHub”，也可在树莓派等低功耗设备运行
二、核心定位与优势
极致轻量：内存通常 < 100MB，单文件运行，树莓派/老旧服务器都能跑
部署极简：二进制、Docker、K8s、源码编译等多种方式，新手友好
全功能：Git 托管 + 协作 + CI/CD + 包管理 + 项目管理，麻雀虽小五脏俱全
完全可控：数据私有、权限自定义、可深度定制
社区活跃：全球贡献，更新快，插件/集成丰富
三、核心功能一览
1. Git 仓库管理
公有/私有仓库、分支/标签/合并、Git LFS、Web 钩子
代码浏览、高亮、 blame、历史对比、文件编辑
2. 代码协作与审查
Pull Request（PR）：完整评审流程、行级评论、冲突解决
代码审查、审批流、合并策略
3. 项目管理
Issue 跟踪：标签、里程碑、指派、截止日期、时间跟踪、依赖
看板（Kanban）：可视化进度管理
Wiki、项目 Pages（静态站点）
4. CI/CD（Gitea Actions）
兼容 GitHub Actions，YAML 编写工作流
内置 runner，支持 20000+ 现有插件，无需额外 CI 服务
5. 软件包仓库（Package Registry）
支持 20+ 包类型：npm、PyPI、Maven、Docker、Helm、Composer、Cargo 等
私有/公有制品托管，替代 Nexus/JFrog
6. 安全与权限
SSH/HTTPS 认证、双因素（2FA）、组织/团队/仓库级权限
LDAP/OAuth2 集成、审计日志、IP 白名单
7. 多平台与生态
系统：Linux/macOS/Windows/FreeBSD
架构：x86_64/arm64/armv7（树莓派）
集成：Jenkins、Drone、SonarQube、钉钉/企业微信、OpenID

个人/小团队：自建私有代码库，替代 GitHub 私有仓库
企业内网：安全可控的代码托管与协作，数据不出内网
边缘/嵌入式：在低功耗设备做版本控制与 CI
教育/开源社区：低成本搭建协作平台
五、与同类产品对比（极简版）
Gitea：轻量、易部署、低资源、全功能、MIT、适合中小团队/个人

Gogs：更轻但功能少、社区活跃度低、更新慢
GitHub/GitLab 云：无需运维、但数据托管在第三方、付费

三、核心重点：Gitea 的 PR + 分支管理，到底怎么复用？怎么包装成非技术用户能用的功能？
这是 Gitea 最强大、最能拉开和飞书 / Notion 差距的核心能力，之前我们只聊了 Issue，而PR + 分支，才是解决「多人协作内容冲突、审批流程、成果验收、全链路追溯、版本迭代」的终极方案。
首先，我先把两个核心技术概念，完全翻译成非技术用户能懂的大白话，你直接就能拿去做前端包装：
表格
Gitea 技术术语	前端非技术化包装	大白话解释
分支（Branch）	「草稿箱 / 个人修改空间 / 项目迭代版本」	在完全不影响项目正式内容的前提下，给你开辟的独立修改空间。你在这里写方案、改内容、做任务，随便改都不会影响正式版本，也不会和其他人的修改冲突，改完了再申请合并到正式版本里。
PR（Pull Request）	「审批申请 / 成果验收申请 / 内容合并申请」	你把自己在草稿箱里改好的内容、做完的任务成果，提交给老板 / 项目负责人，申请审核、提修改意见、最终合并到项目正式版本里的完整流程。自带「内容高亮对比、行级批注、多人评审、审批流、全链路记录」的全能力。
先讲清楚：PR + 分支，对你的产品来说，到底解决了什么飞书 / Notion 解决不了的痛点？
飞书文档、Notion 这类工具，只能实现「简单的多人实时编辑、历史版本查看」，但解决不了 4 个核心的企业协作痛点：
多人同时改同一份内容，容易互相覆盖，版本混乱，找不到谁改了什么；
方案修改、任务验收没有规范的审批流，修改意见散落在微信群里，无法和内容绑定，全链路追溯不了；
正式内容和草稿内容混在一起，容易把没定稿的内容误发给客户 / 老板；
项目出了问题，找不到是谁改的、什么时候改的、为什么改、谁审批通过的，责任划分不清。
而 Gitea 的 PR + 分支，天生就是为了解决这些问题设计的，而且是工业级成熟的方案，你只需要换个前端包装，就能直接用在非技术的日常协作场景里。
分场景拆解：PR + 分支怎么落地到你的产品里？怎么和 OpenClaw 联动？
我给你设计的所有场景，全程对用户隐藏分支、PR 的技术概念，用户只需要点几个按钮，就能完成所有操作，底层完全由 Gitea 和 OpenClaw 自动处理。
场景 1：方案 / 文档的修改与审批（最常用的非技术场景）
这是所有企业都高频使用的场景，完美替代「微信群里改方案、Word 来回发、版本混乱、找不到修改记录」的传统模式。
前端用户的操作流程（完全无技术术语）
项目里的「正式方案」，存在项目仓库的默认「正式版本」里，只有项目负责人有权限直接修改，普通员工只能看，不能直接改。
员工张三想修改方案，只需要在前端点一下 **「修改方案」**，底层自动完成：给张三创建一个专属的「个人草稿空间」（分支），把正式方案复制进去，张三可以随便改，完全不影响正式版本，也不会干扰其他人的修改。
张三改完方案，点一下 **「提交审批」**，底层自动完成：创建一个 PR（审批申请），自动生成标题「张三 - 618 活动方案修改申请」，绑定对应的项目，通知项目负责人审批。
老板 / 项目负责人收到审批通知，点进去就能看到：
高亮修改对比：张三改了哪几行、改了什么内容、删掉了什么，用不同颜色高亮显示，不用自己对着两个版本找不同，一目了然；
行级批注提意见：可以在修改的任意一行，直接加评论 / 修改意见，比如在预算那一行直接写「预算太高，改成 8 万」，张三能实时看到，修改后自动同步到审批申请里；
多人审批：可以设置多个审批人，比如需要市场部经理、财务、老板都通过，才能最终定稿。
所有审批通过后，老板点一下 **「确认定稿」**，底层自动完成：把张三修改的内容合并到项目的「正式版本」里，生成永久的版本记录，谁改的、什么时候改的、审批人是谁、改了什么内容，全链路永久留存，随时可追溯。
OpenClaw 的核心联动（全程自动化）
张三改完方案，OpenClaw 自动帮他检查错别字、逻辑合理性，生成修改说明，自动提交审批申请；
审批过程中，OpenClaw 自动汇总所有修改意见，实时提醒张三修改，同步给所有审批人；
审批通过、内容合并后，OpenClaw 自动把最终定稿的方案，归档到项目知识库，同步给所有项目成员，更新项目进度。
场景 2：任务成果的验收与全流程闭环
完美解决「任务指派→执行→提交→验收→归档」的全流程管理，替代「微信群里发文件、验收意见散落在聊天记录里、找不到最终成果」的痛点。
前端用户的操作流程
项目负责人给张三指派了一个任务，前端对应一个「任务单」（底层是 Issue），明确了验收标准、截止日期。
张三在自己的「个人草稿空间」（分支）里完成任务，提交成果（方案、PPT、设计稿、甚至是代码）。
张三做完任务，点一下 **「提交验收」**，底层自动完成：创建 PR（验收申请），和对应的任务单绑定，自动通知负责人验收。
负责人收到验收通知，点进去就能看到：任务的完成情况、提交的成果、和验收标准的对比、修改的内容明细，直接在里面提验收意见，让张三修改。
验收通过后，负责人点一下 **「确认完成」**，底层自动完成：把任务成果合并到项目正式版本里，把对应的任务单状态改成「已完成」，自动归档所有过程记录。
核心优势
任务的「指派 - 执行 - 修改 - 验收 - 归档」全流程，所有数据都在同一个项目仓库里，不用跨多个工具；
所有的验收意见、修改记录、成果版本，全链路永久留存，随时可追溯，不会出现「扯皮、找不到记录」的情况；
完美和 Issue 联动，实现任务管理和成果管理的完全闭环，这是飞书、Notion 都做不到的。
场景 3：个人日常记忆与 OpenClaw 对话的归档
完美匹配你说的「OpenClaw 把每天的记忆，以 Markdown 格式保存，归档到个人仓库里」的需求，分支 + PR 能让个人记忆的归档更规范、更有条理，不会混乱。
落地方式
每个用户的个人仓库，默认有两个分支：
「正式归档分支」：前端显示为「我的记忆库」，用来存最终确认的、有价值的对话记忆、笔记、日常总结，永久留存，版本可追溯；
「临时草稿分支」：前端完全隐藏，用来临时存用户当天和 OpenClaw 的对话内容，不会直接写入正式归档，避免无效内容污染归档库。
OpenClaw 自动处理流程：
用户当天和 OpenClaw 在飞书里的所有对话，自动整理成 Markdown 格式，实时写入临时草稿分支；
每天晚上，OpenClaw 自动对当天的对话内容做总结，筛选出有价值的内容，生成归档申请（PR），推给用户；
用户确认没问题，点一下「确认归档」，底层自动把内容合并到正式归档分支，生成永久的版本记录；如果用户觉得没用，直接忽略，自动清理临时内容，不会占用空间。
核心优势：不会把所有闲聊内容都堆到归档库里，通过 PR 做了一层筛选，保证个人记忆库的内容都是有价值的、可复用的，同时所有的修改、归档都有完整的版本记录，随时可以回溯之前的对话记忆。
场景 4：多人协作的内容冲突解决与项目迭代
完美解决「多个人同时改同一份内容，互相覆盖、版本混乱」的痛点，同时实现项目的版本化迭代，随时可以回溯到之前的任意版本。
落地方式
每个项目的正式版本，只有负责人有权限修改，所有普通成员的修改，都必须在自己的专属草稿空间（分支）里完成，提交审批后才能合并，完全不会出现「误改正式内容、覆盖别人修改」的情况。
多个人同时修改同一份内容，提交审批的时候，Gitea 会自动检测内容冲突，前端给用户友好提示「你修改的第 3 段内容，和李四的修改有冲突，请先调整」，完全避免了内容覆盖的问题。
项目迭代的时候，前端可以创建「项目 V1.0、V2.0」的里程碑，底层对应独立的迭代分支，每个版本的内容、修改、成果，都完整留存，随时可以对比两个版本的差异，一键回滚到之前的任意版本，完美实现项目的全生命周期管理。
PR 的附加强大能力，怎么复用？
Gitea 的 PR 还有很多成熟的附加功能，你直接包装一下就能用，完全不用自己开发：
审批规则自定义：前端包装成「审批流程设置」，比如可以设置「必须 2 个以上审批人通过才能合并」「必须项目负责人最终审批」，完全适配企业的审批制度；
免审批规则：对于个人仓库、非核心内容，可以设置「提交后自动合并」，不用走审批流程，灵活适配不同场景；
内容自动检查：和 OpenClaw 联动，用户提交审批申请的时候，OpenClaw 自动检查内容的错别字、合规性、逻辑合理性，生成检查报告，不符合要求的可以自动拦截，提前规避问题；
合并策略自定义：前端包装成「定稿方式」，比如「保留修改记录合并」「直接覆盖合并」，用户可以根据需求选择，底层对应 Gitea 的不同合并策略；
关闭与回滚：如果审批不通过，可以直接关闭申请；如果合并后发现有问题，可以一键回滚到之前的版本，完全不会出现内容丢失的情况。

二、影响 Gitea 资源占用的核心因素（避坑重点）
Gitea 的基础服务本身极度省资源，真正会拉高服务器负载的，是你开启的附加功能和使用场景，结合你的产品规划，重点关注这几点：
Gitea Actions（CI/CD 流水线）这是资源占用的第一大头。如果你的平台要给用户提供自动化任务、AI 生成代码、自动归档等能力，大量跑工作流会显著消耗 CPU 和内存。
优化方案：把 Actions Runner 单独部署，不要和 Gitea 主服务共用一台服务器，主服务只做调度，不执行任务，彻底避免 CI 任务抢占主服务资源。
并发用户数与高频操作日常浏览、发帖、看任务、查看文档，对资源消耗极低；但大量用户同时提交内容、上传大文件、对比版本差异，会临时拉高 CPU 和 IO 负载。
优化方案：用 SSD 硬盘提升随机读写性能，开启 Gzip 压缩，配置 Redis 缓存热点数据，大幅降低数据库和磁盘 IO 压力。
Git LFS 大文件存储纯文本（Markdown 文档、任务内容、对话记录）对存储的消耗极小，Git 本身有高效的压缩机制，哪怕几万条文档也占不了多少空间；但如果用户大量上传视频、PPT、设计稿、安装包等大文件，会快速占用存储和带宽。
优化方案：开启 Git LFS，把大文件存储到独立的对象存储（如 MinIO、阿里云 OSS），不占用主服务器的存储和 IO 资源。
仓库数量与小文件占用Git 天生会生成大量小文件，哪怕单个文件只有几 KB，也会占用一个磁盘 inode，一旦 inode 耗尽，哪怕磁盘还有空间，服务也会崩溃。
优化方案：
数据目录单独挂载一块磁盘，不要放在系统根分区；
定期执行 Git 仓库垃圾回收（git gc），清理无用的松散对象；
限制单用户 / 单项目的仓库创建数量，避免指数级增长。

10 人以内小团队：用内置的 SQLite 即可，无需额外部署数据库，最省资源；
