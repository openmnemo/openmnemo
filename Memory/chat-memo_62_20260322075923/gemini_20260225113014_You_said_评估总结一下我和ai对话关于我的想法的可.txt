AI: [2026-02-25 19:10:02]
是的，OpenClaw支持并深度使用了模型上下文协议（Model Context Protocol, MCP）。

在OpenClaw的架构中，MCP扮演着连接AI大脑与外部世界的关键底层通道角色，主要体现在以下几个方面：

作为MCP客户端连接外部服务： OpenClaw本身是一个兼容MCP的客户端，它利用MCP标准与外部的数百种服务、数据库和API进行安全通信。这使得LLM可以跨越系统边界执行操作。

独有的 MCP 调度工具 ( mcporter)： OpenClaw 生态内有一个叫做mcporter的核心扩展技能（Skill）。这个工具专门用于让 OpenClaw 管理、配置和调用各种 MCP 服务器。只要接入目标工具的 MCP 服务器 URL，OpenClaw 就能瞬间“学会”该工具的用法。

与内置“技能”的互补的双重设计：在OpenClaw架构中，突破的“技能”（通常是Markdown文件）主要负责提供“怎么做”的流程指导和业务上下文，而“MCP”则负责提供实际执行动作的“连接和读取文件的能力”（例如真正的网络请求或读取文件的能力）。二者结合，使得系统既懂逻辑，又具备操实业务能力。

结合我们之前评估的爬虫架构，底层的调度中心（如Crawlab）是通过暴露MCP Server接口，让OpenClaw的总控逻辑能够以标准化的协议下达create_spider或run_task等指令，从而实现完美的决策与执行解耦。

AI: [2026-02-25 19:11:35]
是的，OpenClaw确实深度依赖技能，但技能和MCP并不是这即彼的替代关系，而是上下层紧密配合的搭档。

简单来说，它们的区别和协作方式如下：

Skill是“操作手册”或“工作流程”：它们位于知识与流程层。Skill通常由Markdown文件组成，负责教导AI “如何”做事。它包含了业务逻辑、团队规范、操作步骤和决策条件。

MCP 是“万能插头”和“底层执行者”： MCP 位于能力层。它是让 AI 能够真正连接外部工具、读取数据库、运行代码或调用 API 的标准通道。

两者的完美结合： Skill 负责告诉 AI 应该发生什么，而 MCP 使这一切成为可能。可以把 Skill 想象成 GitHub Actions 的 YAML 配置文件（定义了流程），而 MCP 底层实际运行命令的 Bash 环境。

如果只有技能没有MCP，AI就只拥有编写非常好的纯文本指令，但却无法对真实世界采取任何行动；如果只有MCP没有技能，AI虽然拥有了控制外部工具的强大能力，但缺乏业务规范和行为指导。

在我们的爬虫架构中，你可以编写一个“亚马逊爬取策略”的技能教给 OpenClaw 遇到反爬时怎么做，而当它决定更换 IP 或下发爬取任务时，则可以通过 MCP 协议去实际并连接驱动 Crawlab 等基础工具。
