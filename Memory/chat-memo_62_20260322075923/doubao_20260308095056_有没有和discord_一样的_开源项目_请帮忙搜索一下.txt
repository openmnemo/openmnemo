# 安装依赖（需Node.js 14+、yarn）
yarn install
# 启动开发服务器（自动热更新）
yarn dev

核心目录说明（重点关注）：
packages/rocketchat-api：API 相关代码，扩展接口需改这里；
packages/rocketchat-ui：前端 UI 组件，定制界面需改这里；
packages/rocketchat-lib：核心业务逻辑（消息、用户、频道）。
2. 避坑提醒（新手常踩）
不要直接改核心代码：优先用插件 / Hook，否则官方升级后，你的定制代码会难以合并；
控制定制范围：只改你需要的功能，保留核心模块（如实时通信、音视频），避免重复造轮子；
关注版本兼容性：选择 LTS 版本（如 6.x、7.x）进行二开，避免用最新开发版，减少 bug；
数据迁移预留：若后续需要升级官方版本，提前做好自定义表 / 字段的迁移方案。
3. 常见二开场景参考（可直接落地）
定制界面：修改前端 React 组件，替换 Logo、配色、布局，适配自有品牌；
权限扩展：新增自定义角色（如「只读管理员」「临时访客」），细化频道权限；
集成业务系统：对接 CRM/ERP，消息触发工单创建，或从业务系统同步用户数据；
功能裁剪：移除「客服」「市场」等非核心模块，打造轻量化版本；
音视频定制：对接自有音视频服务器（如阿里云 RTS），替换默认音视频引擎。
三、替代方案对比（为什么不选 Element 二开？）
如果你纠结过 Element，这里给出清晰对比，帮你确认选择：
表格
维度	Rocket.Chat	Element（Matrix）
二开难度	低（产品化架构，文档完善）	中（去中心化架构，需理解 Matrix 协议）
技术栈熟悉度	主流（Node.js/React）	偏小众（Matrix SDK 学习成本高）
功能复用	核心功能完整，可直接用	需先搭建 Synapse 服务端，再定制客户端
商用友好度	MIT 协议，无限制	Apache 2.0，同样友好，但部署更复杂
总结
可行性：Rocket.Chat 完全适合二开，MIT 协议无商业限制，技术栈主流、文档完善；
最佳实践：优先用「插件 / Hook」扩展功能（降低维护成本），仅在必要时修改核心代码；
核心建议：基于 LTS 版本开发，聚焦自身业务需求，保留官方核心模块（如实时通信），避免重复开发。
