OpenMnemo 项目核心总结
OpenMnemo 是一款跨平台 AI 对话内存管理器，名称源自希腊记忆女神 Mnemosyne，核心用于 AI 编码会话的统一管理、版本控制与可视化，基于 MIT 协议开源。
一、核心能力
多端 AI 会话统一管理：自动发现、导入、去重、索引主流 AI 编码工具的对话记录，基于 Git 做版本化存储。
可视化报告生成：可生成带会话列表、统计图表、知识图谱的静态 HTML 仪表盘，支持本地预览与在线部署。
AI 增强能力：支持接入 Claude 模型，为每个会话生成 AI 摘要，提升检索与回顾效率。
自动化同步：内置守护进程，可定时自动同步新会话，支持配置自动推送到 Git 远程仓库、自动重建报告。
高效检索：支持会话全文检索，以及快速获取最近会话摘要，可对接 AI Agent 钩子。
二、快速上手核心流程
安装 CLI：通过 npm 全局安装 npm install -g @openmnemo/cli
初始化项目：在 Git 仓库内执行 openmnemo init，创建标准化的 Memory 目录结构
导入会话：通过 openmnemo discover 自动扫描导入本地会话，或用 openmnemo import 手动导入指定文件
生成并查看报告：执行 openmnemo report build 生成 HTML 报告，可通过 openmnemo report serve 本地启动预览服务
三、支持的 AI 客户端
表格
客户端	支持方式	会话格式
Claude Code	自动发现导入	JSONL
Codex	自动发现导入	JSONL
Gemini CLI	自动发现导入	JSON/JSONL
豆包 (Doubao)	手动导入	TXT
四、配置与部署
全局配置文件位于 ~/.memorytree/config.toml，可设置同步间隔、日志级别、语言（支持中文 zh-CN）、自动推送、自动生成报告等参数。
支持一键将生成的报告部署到 GitHub Pages，可配置自定义域名、飞书 / Slack 等平台的消息通知 webhook。
五、技术与开发相关
环境要求：Node.js >= 20、Git
技术栈：100% TypeScript 开发，采用 pnpm monorepo 架构，turbo 构建，数据库使用 better-sqlite3
包结构：分为 types（共享类型）、core（核心解析索引）、report（报告生成）、sync（同步守护进程）、cli（命令行工具）5 个核心包
发布流程：通过 GitHub Actions 自动发布 npm 包，tag 推送触发自动发布

它的核心作用与能力
多平台AI对话统一归集支持自动发现并导入 Claude Code、Codex、Gemini CLI 等主流AI编码工具的本地对话记录，也支持手动导入豆包的对话导出文件，把你散在各处的AI开发对话全部集中管理。

生成可视化的记忆看板可以把所有导入的对话，生成一个带统计图表、知识图谱的静态HTML仪表盘。你可以在浏览器里直观查看所有项目的AI会话、按关键词全文搜索、快速回溯开发对话，不用再挨个打开不同的AI客户端翻历史。
自动化同步与AI增强能力内置后台守护进程，可设置定时自动同步新的AI对话，还能配置自动推送到Git远程仓库、自动更新看板。同时支持接入Claude大模型，给每个会话生成AI摘要，帮你快速提炼长对话的核心内容，不用手动翻阅冗长的聊天记录。
灵活的检索与集成能力支持对所有对话历史做全文检索，快速定位你需要的历史对话；还提供了快速调取最近会话摘要的能力，可方便地集成到AI Agent工作流中，为Agent提供项目的历史上下文。
基础信息补充
开源协议：MIT协议，完全免费，支持商用与二次开发
运行要求：仅需 Node.js 20+ 与 Git 环境，跨Windows、Mac、Linux全平台可用
技术栈：全程使用TypeScript开发，采用monorepo架构拆分核心模块，维护性强
额外支持：内置中文语言选项，支持将看板一键部署到GitHub Pages，可配置飞书、Slack等平台的消息通知

三、第二步：做「统一对话入口」，核心是「以个人记忆为核心的对话代理」，而不是再做一个聊天机器人
你想要的对话入口，绝对不是再做一个和 ChatGPT、豆包一样的聊天框，它的核心定位应该是：用户所有 AI 对话的「统一中控入口」，它的核心能力不是生成内容，而是「基于用户的全量个人记忆，为每一次对话提供完整的上下文支持」。
这个入口，完全可以基于你现有的能力快速落地，不用从零开发：
最快的 MVP：基于你现有的 CLI，加一个openmnemo chat命令，先实现命令行里的对话能力；
正式版：直接在你现有的 HTML 报告看板里，加一个对话界面，用户打开生成的 HTML，既能看记忆看板、管理记忆，也能直接聊天，一套前端搞定，不用额外开发。
这个对话入口的核心逻辑，是「个人记忆专属的 RAG 流程」，和普通的知识库 RAG 完全不同，它要做的是「先懂用户，再回答问题」，完整流程是：
用户输入问题：比如「我之前做的表格网站的 SEO 方案，能不能用到我现在的新项目里？」
意图解析与记忆召回：
先解析用户的核心意图、关键词、关联的项目 / 技术 / 需求；
用「全文检索 + 知识图谱关联」的双路径召回：既找到和关键词直接匹配的记忆单元，也顺着知识图谱，找到所有相关的关联记忆（比如当前新项目的技术栈、之前表格网站的 SEO 方案、踩过的坑、用户定过的 SEO 相关决策规则）。
记忆筛选与上下文整合：
按权重、相关性筛选出最核心的记忆，不会把所有内容都塞给大模型；
把这些记忆按逻辑整合成通顺的上下文，比如：「你之前的表格网站 SEO 方案是 XXX，踩过的坑是 XXX，你当前的新项目技术栈是 XXX，需求是 XXX，你之前定过的 SEO 规则是 XXX」。
生成回答，主动关联记忆：
大模型基于整合后的个人记忆，结合自身能力生成回答；
关键是要让 AI 主动关联相关记忆，比如主动提「你之前在 XX 项目里也用过类似的方案，当时的效果是 XXX，要不要参考？」，而不是只被动回答用户的问题。
新记忆自动沉淀，形成闭环：
这次对话的内容，会自动被拆分成新的原子化记忆单元，加入到知识图谱里，更新权重；
自动同步到你的 Git 仓库，完成版本化存储，真正实现「对话一次，沉淀一次，记忆永远增长，永不丢失」。
这个入口的额外优势：你不用绑定任何一个大模型
用户可以自己配置想用的大模型（Claude、GPT、豆包、Deepseek 等等），你的入口只负责「提供用户的个人记忆上下文」，把上下文和用户的问题一起传给用户指定的大模型，再把回答返回给用户。
这样用户就不用再在不同的 AI 平台之间切换，不管想用哪个模型，都能在这个入口里，用上自己完整的个人记忆，彻底打破平台壁垒。
四、第三步：让记忆「主动活起来」，而不是被动被检索
你想要的「AI 能全局考虑用户的所有记忆，主动复用过往的内容」，本质是让 AI 从「用户问什么，就找什么」的被动工具，变成「懂用户的需求，主动给用户匹配相关记忆」的智能助手。
