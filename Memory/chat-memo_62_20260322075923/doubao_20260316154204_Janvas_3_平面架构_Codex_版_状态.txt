## 1. 一句话（新定位）

Janvas 是一个“三平面（Authoring/ Sandbox/ Workflow）+ 双画布（X6/ Excalidraw）”的 AI 工作台：把用户的书写、语音与对话，变成可验证、可回放、可发布、可长期运行的工作流与产物。

---

## 2. 为什么要做（三平面带来的商业闭环）

### 2.1 你提出的新需求本质

把产品从“内容编辑 + 工作流编排”升级为：
- **沙盒验证**：先在受控环境里把不确定性跑出来（工具调用、副作用、生成质量）。
- **可观测可回放**：像 Manus 一样把“执行过程”变成时间线，用户能理解、能复盘、能复用。
- **资产化与复用**：一次对话/一次整理/一次运行的结果，能沉淀为长期可用的工作流（而不是一次性回答）。

### 2.2 三平面能形成的商业杠杆

- **从“工具”到“平台”**：Authoring 输入 → Sandbox 证据链 → Workflow 长期复用。
- **从“生成”到“交付”**：输出不仅是文本，还包括 HTML deck / PPTX / 结构化 WorkflowDraft。
- **从“可用”到“可信”**：关键过程可回放，可定位失败步骤，可审计（企业付费核心点）。

---

## 3. 命名口径：为什么不再用“三层 Layer”

当前仓库中，“Layer”已经被用于描述 UI 中的两张画布（Workflow/Content）以及 Glue Core 的语义（`janvas/SPECS/glue-core.md`）。如果再引入 “Sandbox Layer”，会出现三类常见问题：
- 文档与代码里 “layer” 的含义分裂：到底是产品分层、画布引擎层、还是渲染叠层？
- `InteractionTarget`/输入路由的语义被污染：本来 target 只在两张画布间切换。
- 后续扩展（例如浏览器自动化沙盒、容器/VM 沙盒）更难表达。

### 3.1 推荐的规范命名（两条轴）

**A) Plane（产品级平面/域）**：描述能力边界与业务闭环  
- `Authoring Plane`：创作输入（书写/标注/语音）
- `Sandbox Plane`：执行/验证/可观测/回放（一次 run 的证据链）
- `Workflow Plane`：可版本化、可长期复用的工作流资产

**B) Canvas（工作台画布）**：描述 UI 引擎与交互目标  
- `Workflow Canvas = X6`
- `Content Canvas = Excalidraw`

## 4. 三平面职责与边界（必须写死，避免跑偏）

### 4.1 Authoring Plane（创作平面）

**职责**
- 承接用户输入：文本、结构化笔记、画布标注、音频录制。
- 形成可被沙盒消费的“输入流”（Input Stream），支持增量更新。

**非职责**
- 不负责执行/验证（避免“创作页里塞满执行逻辑”导致耦合）。

### 4.2 Sandbox Plane（沙盒平面）

**职责**
- 作为“受控执行环境”：运行对话任务、内容整理任务、工作流草稿验证任务、（后续）浏览器/桌面自动化任务。
- 输出“逐步展示”的证据链：统一产出 `RunEvent` 序列（可回放）。
- 产出 `Artifact`：HTML deck（优先）、PPTX（增强）、文件/链接/截图等。
- 作为发布门槛：`WorkflowDraft` 只有通过沙盒验证才能进入 Workflow Plane。

**非职责**
- 不承担长期调度与资产管理（那是 Workflow Plane 的职责）。

### 4.3 Workflow Plane（工作流平面）

**职责**
- 存储与管理长期资产：`Workflow`（版本化）、运行配置、复用入口。
- 提供稳定的运行接口：运行时仍以 `RunEvent` 投影到沙盒时间线（可观测/可回放）。

**非职责**
- 不直接承担“在用户面前解释执行过程”的 UI 表达（统一交给沙盒时间线 UI）。

## 5. 三大核心能力（对应你提出的 3 个作用）

### 5.1 内容整理器（Authoring → Sandbox → Artifact / WorkflowDraft）

**输入**
- 用户在创作平面书写的内容（文本/结构化段落/画布标注）
- 麦克风录音（音频流）

**处理（沙盒内）**
- 语音转写：分段（两种策略二选一或并行支持）
  - 固定窗口：每 5 秒切片
  - 动态分段：VAD（语音活动检测）按停顿切片
- 增量整理：把“原始输入流”变成规范的结构（大纲/要点/行动项/待确认问题）
- 生成输出：
  - `HTML deck`（Phase 1 默认产物）
  - `PPTX`（增强项）
  - `WorkflowDraft`（草稿工作流）

**验证与同步**
- 若输出为 `WorkflowDraft`：必须在沙盒内完成 schema 校验 + dry-run（最小副作用）+ 最小回归后，才能同步到 Workflow Plane 成为 `Workflow`。

### 5.2 工作流执行逐步展示（Workflow → Sandbox Timeline）

**目标体验**
- 用户在工作流平面运行一个工作流时，沙盒平面展示每一步：
  - 节点状态变化（pending/running/succeeded/failed）
  - 工具调用（start/input/output/error）
  - 关键输入输出（可展开、可复制、可定位）
  - 产物（下载/预览/链接/截图）

**实现策略（Phase 1）**
- 不做“完整远程桌面回放”作为硬前置；先做“工具级可观测 + 产物可见 + 可回放 RunEvent”，必要时补“截图事件”。

### 5.3 对话任务沙盒化运行（Chat → SandboxRun）

**目标体验**
- 用户的对话需求不再是一次性回答，而是一次 `SandboxRun`：
  - 有明确步骤（哪怕只有 1 个节点）
  - 有工具调用（检索/解析/生成/导出）
  - 有产物与回放
  - 可一键固化为 `WorkflowDraft`

## 6. 与仓库现状对齐：哪些可直接复用，哪些需要新增

### 6.1 已具备的“沙盒雏形”（可直接复用）

- `RunEvent` 协议 + reducer + 回放能力（`janvas/SPECS/run-event.md`）
- SSE 推送通道（`janvas/services/ai-gateway`，`Content-Type: text/event-stream`）
- 已有的事件类型覆盖“逐步展示”的骨架：
  - `node.status`（步骤状态）
  - `node.tool`（工具调用阶段）
  - `node.asset`（产物/资源）
  - `node.text.delta`（流式文本）

### 6.2 需要新增/补齐的能力（后续通过 SPECS 驱动）

- **SandboxRun 的统一模型**：把“对话/整理/验证/自动化”抽象为同一种 run（而不是各自一套）。
- **WorkflowDraft schema**：可被验证、可被发布、可被投影为 X6 工作流。
- **产物规范**：
  - HTML deck 的结构/主题/可嵌入标准
  - PPTX（增强）何时引入、怎么回归、怎么避免依赖膨胀
- **音频链路**：
  - 浏览器录音、上传、分段策略、增量转写
  - 断点续传/丢包容错/隐私策略

## 7. 技术架构（提案）：服务、协议、数据流

### 7.1 推荐的最小服务划分（不等于马上要实现）

```
apps/workbench-web
  - Authoring UI（文本/画布/录音入口）
  - Timeline UI（逐步展示/回放）
  - Workflow UI（编排/发布/运行入口）

services/ai-gateway（已存在）
  - Provider 白名单 + CORS + 限流/超时
  - 将上游流式输出/工具调用翻译为 RunEvent（SSE）

services/sandbox（规划新增）
  - SandboxRun 执行与状态机（对话/整理/验证/自动化）
  - 工具路由与权限（白名单/配额/网络出站策略）
  - 产物生成（HTML deck 优先）
  - WorkflowDraft 校验与发布前验证
  - 统一输出 RunEvent（SSE 或通过 gateway 转发）
```

> 说明：Phase 1 可以先不拆新服务，先把“沙盒运行模型”落在现有 `ai-gateway` 的 mock/工具事件上验证 UI 与协议；但必须把“服务边界”先写清，避免未来拆分成本爆炸。

### 7.2 统一协议：RunEvent 作为“可观测底座”

原则：任何对用户可见的关键过程，必须可投影为 RunEvent。

**现有事件类型（仓库已定义）**
- `run.start` / `run.finish` / `run.error`
- `node.status`
- `node.text.reset` / `node.text.delta`
- `node.tool`
- `node.asset`

**Phase 1 推荐补充的事件（提案，需后续写入 SPECS）**
- `node.log`：纯日志（debug/trace），可在 UI 折叠
- `node.metric`：耗时、token、成本估算（为商业化埋点）
- `node.snapshot`：截图/HTML 片段预览（增强“所见即所得”）
- `run.link`：把一次 run 与 `WorkflowDraft`/`WorkflowVersion`/`Artifact` 建立强关联（便于回放与追溯）

### 7.3 数据模型（最小集合）

- `SandboxRun`
  - `runId`（全局唯一）
  - `events[]`（RunEvent 序列，可回放）
  - `artifacts[]`（HTML deck 等）
  - `source`（来自 Authoring/Chat/Workflow 的哪个入口）
- `WorkflowDraft`
  - `draftId`
  - `definition`（JSON/DSL，必须可校验）
  - `validation`（沙盒验证结果 + 证据链 runId）
- `Workflow`
  - `workflowId`
  - `versions[]`
  - `latestVersionId`

## 8. 安全与合规基线（Phase 1 必须具备）

### 8.1 密钥策略（BYOK）

- 用户 Key 只用于当次请求（例如 `X-Upstream-Authorization`），默认不落库、不进日志。
- 浏览器侧不得持有可滥用的长时效密钥；需要登录会话时再引入独立的 auth 方案与审计。

### 8.2 工具白名单与副作用控制

- 所有工具必须显式注册（白名单），并在事件流中可见（`node.tool`）。
- 对“有外部副作用”的工具（发邮件、写库、发起支付等）必须具备：
  - 二次确认策略（产品规则）
  - dry-run 模式（验证不产生副作用）
  - 运行配额与超时

### 8.3 网络出站策略（防 open-proxy）

- 沙盒默认禁止任意出站；只允许白名单域名/协议，按工具能力最小授权。
- 必须具备并发限制、超时与速率限制（现有 `ai-gateway` 已有雏形：限流、并发 guard、超时 abort）。

## 9. 分期路线图（建议）

### Phase 0（现状巩固）

- 保持双画布 + Glue Core + RunEvent 链路稳定（为三平面打地基）。

### Phase 1（Sandbox Plane MVP：先把“证据链”做出来）

- Authoring：录音入口 + 文本输入的增量上报（最小闭环）
- Sandbox：转写 + 每 5 秒/动态整理（二选一先落地）+ 生成 HTML deck
- Timeline：RunEvent 逐步展示（node.status/node.tool/node.asset 优先）
- WorkflowDraft：最小 schema + 沙盒验证 + 发布门槛（draft → workflow）

### Phase 2（Web 自动化增强）

- 浏览器自动化任务进入沙盒运行（抓取/表单/截图/下载），并以事件流可回放。

### Phase 3（强隔离沙盒）

- 需要桌面级/容器级隔离时，再评审引入（成本、时延、运维、安全合规、可观测）。

// 1. 密钥管理（BYOK）
class KeyManager {
// 仅在请求时传递密钥，不存储
async withUserKey<T>(key: string, fn: () => Promise<T>): Promise<T> {
try {
process.env.UPSTREAM_AUTH_KEY = key;
return await fn();
} finally {
delete process.env.UPSTREAM_AUTH_KEY; // 执行后立即清除
}
}

// 2. 工具白名单与副作用控制
const TOOL_WHITELIST = ["transcribe", "organize", "generate-html", "validate-workflow"];
const SIDE_EFFECT_TOOLS = ["send-email", "write-db"]; // 有副作用的工具

class ToolRouter {
async routeTool(toolName: string, params: any): Promise<any> {
// 白名单校验
if (!TOOL_WHITELIST.includes(toolName)) {
throw new Error(`Tool ${toolName} is not allowed`);
}
// 有副作用的工具需二次确认 + dry-run
if (SIDE_EFFECT_TOOLS.includes(toolName)) {
if (!params.dryRun && !params.confirm) {

3. 双画布“双向联动”：避免“两张皮”，提升用户体验
当前你有 Workflow Canvas（X6）和 Content Canvas（Excalidraw），核心痛点是“工作流节点和内容画布的内容脱节”，建议新增 “画布联动协议”：- 当用户在 Workflow Canvas 选中一个“内容整理节点”时，Content Canvas 自动定位到对应的文本/标注区域；- 当用户在 Content Canvas 编辑内容后，自动触发沙盒的“增量更新”，并在 Workflow Canvas 节点上显示“未验证”标记，提醒用户重新运行沙盒验证；- 技术实现上，用 X6 的事件监听+Excalidraw 的状态同步 API，通过 localStorage 或 WebSocket 实现双向通信（Phase 1 优先用 localStorage，低成本快速验证）。

之前的商业化方案是“个人/团队/企业”三级订阅，这里补充 2 个差异化变现抓手和 1 个获客策略，让付费转化更顺畅：
1. 新增“按量付费+订阅叠加”模式：降低用户首次付费门槛
纯订阅制对个人用户不友好（比如偶尔用一次的用户不愿按月付费），建议：- 免费版：限制“沙盒运行次数（5 次/天）+ 工作流存储数量（3 个）+ 不支持 PPTX 导出”，满足轻量用户需求；- 按量付费版：免费版额度用完后，按“沙盒运行时长（0.01 元/分钟）+ 工作流存储（0.5 元/个/月）+ PPTX 导出（1 元/次）”收费，吸引偶尔使用的用户；- 订阅版：在按量付费基础上，推出“包月/包年”套餐，包含“无限沙盒运行+无限工作流存储+高级导出+团队协作”，并对订阅用户免除按量付费的手续费，引导用户从按量付费转向订阅。

这是长期商业化的核心抓手，也是区别于其他 AI 工具的关键，建议分阶段落地：- Phase 1（内容沉淀阶段）：允许用户将自己的工作流“导出为模板”，平台审核后放入免费模板市场，提升用户复用率（比如“周报生成工作流”“方案整理工作流”）；- Phase 2（变现阶段）：开放付费模板市场，创作者可给工作流模板定价（5-50 元/个），平台抽成 20%-30%；同时推出“模板会员”，用户付费后可免费使用所有付费模板，平台和创作者按分成比例结算；- Phase 3（生态阶段）：开放工具插件市场，允许第三方开发者开发“沙盒工具”（比如“Excel 导出工具”“思维导图生成工具”），开发者自主定价，平台抽成，丰富沙盒的能力边界。

问题诊断：工作流运行失败时，AI 教练会自动定位失败原因（如 “工具调用超时”“输入参数格式错误”），并给出解决方案。
解决的用户痛点
新手用户：“不用看教程，跟着 AI 就能搭建工作流”；
进阶用户：“AI 能帮我优化工作流，提升生成效果”。
对应的商业价值
免费版：提供基础的引导式搭建，付费版解锁 “高级优化建议 + 问题诊断”；
提升留存率：降低学习成本，减少新手用户的流失率，间接提升付费转化。
5. 生态功能：第三方工具集成 —— 提升用户粘性，拓展变现渠道
核心目标：让 Janvas 3 成为用户 “工作流的中枢”，而不是孤立的工具。升级功能：开放平台 + 第三方工具集成

开发者平台：开放 API，允许第三方开发者开发 “沙盒工具插件”（如 “Excel 导出插件”“思维导图生成插件”），平台收取插件销售额的 20%-30% 分成。
对应的商业价值
提升用户粘性：用户在办公场景中频繁使用 Janvas，留存率大幅提升；
生态变现：第三方插件分成成为新的收入来源；
企业合作：与飞书、钉钉等企业合作，推出 “联合套餐”，共享客户资源，降低获客成本。
二、 商业层面：4 套 “高转化 + 高 LTV” 变现组合拳
基于上述功能，设计 “用户分层 + 场景化套餐 + 增值服务 + 生态变现” 的组合变现策略，最大化单用户生命周期价值（LTV）。

2. 按量付费 + 订阅叠加：降低获客门槛，提升转化效率
针对 “偶尔使用的用户” 和 “订阅用户的超额需求”，设计 按量付费补充方案：
免费版超额后：按 “沙盒运行时长（0.01 元 / 分钟）+ PPTX 导出（1 元 / 次）+ 大文件解析（0.5 元 / MB）” 收费；
订阅用户超额后：比如团队版用户的协作人数超过 10 人，按 “5 元 / 人 / 月” 额外收费；企业版用户的私有部署存储超过配额，按 “0.1 元 / GB / 月” 收费；
核心优势：避免 “用户因‘偶尔使用’不愿订阅” 的问题，同时提升订阅用户的 LTV。
3. 增值服务：高毛利变现，拉高企业用户价值
除了基础套餐，推出 3 类高毛利增值服务，针对企业用户的深度需求：

4. 生态变现：工作流市场 + 插件市场，打造长期收入生态
这是 Janvas 3 从 “工具” 升级为 “平台” 的核心，分 2 个阶段落地：
阶段 1：工作流模板市场（短期变现）
免费模板：用户可免费下载平台官方模板（如 “周报生成”“会议纪要整理”），提升用户复用率；
付费模板：优质创作者（如行业专家、资深用户）可发布付费模板，定价 5-50 元 / 个，平台抽成 20%；
模板会员：推出 “模板会员”（19 元 / 月），会员可免费使用所有付费模板，平台与创作者按分成比例结算；
核心价值：创作者通过模板变现，平台获得分成收入，同时丰富模板生态，吸引更多用户。
阶段 2：第三方插件市场（长期生态）
工具插件：第三方开发者可开发 “沙盒工具插件”（如 “Excel 导出插件”“思维导图生成插件”），开发者自主定价，平台抽成 20%-30%；
集成插件：开发者可开发 “第三方工具集成插件”（如 “对接 Salesforce 插件”“对接 ERP 插件”），满足企业用户的个性化集成需求；
核心价值：插件市场丰富 Janvas 的能力边界，同时带来持续的分成收入，形成 “平台 - 开发者 - 用户” 的生态闭环。
三、 功能与商业的联动策略：让功能驱动付费转化
每个核心功能都要设置 “免费试用门槛” 和 “付费解锁点”，引导用户自然转化：
交互式回放：免费版可 “只读回放”，付费版解锁 “断点调试 + 分支实验”；
多模态输入：免费版限制 “文件大小 10MB”，付费版解锁 “无限制文件上传 + 大文件解析”；
团队协作：免费版支持 “2 人协作”，付费版解锁 “多人协作 + 精细化权限”；
AI 教练：免费版提供 “基础引导”，付费版解锁 “高级优化建议 + 问题诊断”。
核心原则：让用户 “先体验到功能的价值，再付费解锁完整能力”，避免 “付费前看不到价值” 的转化障碍。
