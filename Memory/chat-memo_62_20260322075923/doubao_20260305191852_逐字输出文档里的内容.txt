# AgentCrawl Plan8 技术架构白皮书（全新重构）
版本：v8.0日期：2026-03-05受众：架构师、后端 / 前端工程师、平台运维、技术管理者
1. 目标与边界
Plan8 的技术目标是把历史方案中的 “可执行细节” 并回主干：
面向普通办公用户，而非开发者优先。
保留 OpenClaw 的 IM 交互能力与 Web 门户。
以 Gitea 作为版本化协作与数据底座。
以 “一切皆仓库” 驱动插件与流程生态。
通过三层信任架构保证可控、可追溯、可回滚。

3. 三层信任架构（Plan8 强制）
表格
层级	自由度	输出类型	允许动作
探索层（OpenClaw）	高	建议	读、分析、建议，不直接落关键写
命令层（CLI Gateway）	中	事实	关键状态写入、快照、审计
工作流层（Engine）	低	保证	版本冻结插件的确定性执行
3.1 写操作唯一入口
架构红线：所有关键状态变更必须经过 CLI Gateway，禁止绕过。
3.2 可逆执行链路
所有写操作先生成快照。
audit replay 默认 dry-run。
reexecute 必须生成新执行记录，不覆盖历史。
4. 架构红线规则（必须执行）
关键状态写必须经 CLI Gateway。
工作流层仅调用 “已发布且版本冻结” 的插件。
所有写操作必须可追溯（请求、执行者、结果、快照）。
audit replay 默认 dry-run。
采集一期必须具备：账号池、代理池、会话租约、断点续爬。
免费试用默认只读（限制高风险写操作）。
合规边界优先：禁止绕过访问控制，禁止采集受限内容。
5. Promotion Pipeline（能力晋升）

作用：把高 Token 的临时智能，持续转化为低成本的确定性能力。

# AgentCrawl Plan8 商业计划书（全新重构）
版本：v8.0日期：2026-03-05定位：面向普通办公用户的 AI 原生协作平台范围：一人公司、中小团队（5-200 人）、数据驱动业务团队
1. 战略定义
AgentCrawl 的核心不是做新的开发者平台，而是把成熟的开源底座重组为普通人可直接使用的办公协作系统：
交互入口：OpenClaw in IM + OpenClaw Web
协作底座：Gitea
关键机制：Everything is a Repository
增长飞轮：AI探索 -> 技能固化 -> 工作流自动化
一句话：让团队在一个系统里完成项目、任务、审批、知识、素材与自动化，并保留可追溯历史。
2. 市场问题与机会
2.1 已验证痛点（来自历史方案验证）
工具碎片化：办公人员常同时使用 8-12 个工具。
协作低效：信息散落导致检索与对齐成本高。
版本失控：缺少统一的版本化协作底座。
数据主权弱：关键数据散落于第三方 SaaS。
2.2 市场规模（用于融资叙事）
TAM：$50B+（企业 AI 应用与协作软件大盘）
SAM：$5B（AI 协作与自动化细分）
SOM：$500M（初期聚焦中国与亚太中小团队）
注：上述数字沿用历史计划中的内部口径，用于融资阶段一致叙事。
2.3 直观成本对比
表格
方案	典型构成	参考成本
传统工具栈	Slack + Trello + Notion + Workspace	~$40 / 人 / 月
AgentCrawl 团队版	统一协作 + 插件 + 记忆 + 素材检索	~$19-49 / 人 / 月（SaaS 口径）
目标价值不是 “替代一个工具”，而是 “替代一组割裂工具 + 增加数据资产能力”。
