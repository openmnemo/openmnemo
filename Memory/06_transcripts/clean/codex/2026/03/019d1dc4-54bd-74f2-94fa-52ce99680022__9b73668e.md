**我查到的最值得看的项目**
- [GitNexus](https://github.com/abhigyanpatwari/GitNexus)
  它不是简单把仓库文件切块后向量化，而是先做代码结构理解，再做知识图谱，再叠加 BM25 + semantic + RRF 混合检索。README 里明确写了 Tree-sitter 解析、知识图谱、混合搜索、向量索引、MCP 工具链，这个思路对你们很有参考价值。
- [PageIndex](https://github.com/VectifyAI/PageIndex)
  这个最像你说的“记忆树 + 探针理论”。它不是向量库路线，而是先生成层级树，再做 tree search。它更适合作为“主线/里程碑/分支”的导航层，不适合作为底层唯一索引。
- [Cognita](https://github.com/truefoundry/cognita)
  这个项目最值得借的是“增量索引流程”。它会比较数据源状态和向量库状态，判断新增、更新、删除，再重新解析、切块、嵌入、写回。这个非常适合你们后面用 Gitea/Git 做增量重建。
- [elastic/semantic-code-search-indexer](https://github.com/elastic/semantic-code-search-indexer)

- [codebase-index-cli](https://github.com/dudufcb1/codebase-index-cli)
  这个非常贴近你们当前路线：本地索引、SQLite/Qdrant 双存储、Tree-sitter、Git commit 跟踪、实时代码库更新。它说明“Git 变化驱动索引更新”这条路完全可落地。
- [RepoSearchRAG](https://github.com/MarkoKolarski/RepoSearchRAG)
  这个项目虽然小，但它把“query expansion + 多策略检索 + listwise reranker + Recall@K 评估”讲得很清楚。对你们后面做“探针拿回来的候选节点怎么排序”很有帮助。
- [claude-context-local](https://github.com/FarhanAliRaza/claude-context-local)
  它是比较朴素的一类：本地 embedding + FAISS + SQLite metadata。适合借鉴本地部署和存储组织，但不够回答你们“记忆树”这个更高层问题。
- [Quivr](https://github.com/QuivrHQ/quivr)
  它更像通用 RAG 工作流框架，优点是检索流程可配置、支持 reranker、history filter、rewrite。适合借鉴“工作流编排”，不适合直接拿来当记忆树方案。

**这些项目说明了什么**
GitHub 仓库内容“向量化”目前大致有 4 条路线：

- 纯向量路线
  代表是 `claude-context-local`、部分 `codebase-index-cli`
  做法是文件/代码块切分后直接 embedding。
- 结构化代码理解 + 图谱 + 混合检索
  代表是 `GitNexus`
  这是我认为最强、最接近你们未来方向的。
- 增量生产化索引流水线
  代表是 `Cognita`
  重点不是怎么搜，而是怎么稳定地持续更新索引。
- 树形索引 / 非向量导航
  代表是 `PageIndex`
  这个和你“记忆树”思想最接近。

1. `Git/Gitea` 负责原始事实和版本线

- 原始对话永久留存
- 每轮/每段归档快照
- 分支隔离
- 变更触发增量重建

2. `RAG/向量库` 负责语义召回  
不是全量把所有对话等权做 embedding，而是优先对这些对象做：
- 小归档节点
- 里程碑节点
- 决策节点
- 待办节点
- 实体关系摘要节点

3. `图数据库` 负责关系，不负责主线树  
图谱很适合管：
- 这个节点属于哪个主题
- 这个里程碑关联哪些任务
- 哪些分支互相有关
- 哪些会话共用同一实体/项目/技术

4. `记忆树` 本身要做成独立逻辑层  
这一层才是你真正的中间件。  
它应该至少有：
- `main spine`：主线里程碑
- `branch`：某个主题/阶段的分支
- `leaf`：小归档节点
- `source refs`：指回原始对话、commit、artifact
- `summary + confidence + children + parent`

**最重要的一点**

- 代码仓库天然有文件边界、函数边界、AST

- 3 轮或 1 个闭环任务 = 小归档单元

**一、单一会话的记忆树怎么规划**
单会话树的目标只有一个：

**让一次超长对话，在任何时刻都能只把“最小必要上下文”送进模型。**

我建议单会话树固定成 4 层：

```text
SessionRoot
├── StageNode          当前会话的大阶段
│   ├── MilestoneNode  一个阶段性结论/决策/转折
│   │   ├── ArchiveBatch   一组相近叶子的小归档
│   │   │   ├── ConversationLeaf
│   │   │   ├── ConversationLeaf
│   │   │   └── ConversationLeaf
```

每层职责要分清：

- `ConversationLeaf`
  - 最小单元
  - 建议不是死板“每 3 轮”，而是 `1-3 个 user turn / 一个明确主题片段 / 一次大工具输出`
- `ArchiveBatch`
  - 把相近的 leaf 收成一个小包
  - 只保留局部锚点，不删原文
- `MilestoneNode`
  - 记录“这一段真正形成了什么”
  - 最适合放：结论、决策、约束、未解决问题
- `StageNode`
  - 会话主线
  - 只保留本会话的高层导航，不装细节

单会话的压缩触发，不建议只按轮次，应该是 4 个信号一起判断：

- `轮次触发`
  - 比如 2-4 个 user turn
- `token 触发`
  - 当前活跃上下文逼近预算
- `主题漂移触发`
  - 文件集、关键词、意图明显变化
- `工具洪峰触发`
  - 出现大段代码、日志、终端输出

单会话的 probe 顺序应该固定：

1. `probe_main(session)`
2. `probe_branch(milestone_id)`
3. `probe_leaf(batch_or_leaf_id)`
4. `hydrate_raw(raw_ref)`

停止条件也要固定，不要纯靠模型自由发挥。辅助文档里“机制优先、能力兜底”这个方向是对的 [辅助文档](/d:/demo1/openmnemo/Memory/doubao_20260316153420_是这样的_OpenClaw就是O_P_E_N_C_L_A_W.txt#L98)：

- 命中主线且信息完整，停
- 命中分支且证据足够，停
- 必须下钻到叶子才能回答，再下钻
- 需要原文证据时才 `hydrate_raw`

**二、一个仓库的记忆树怎么规划**

```text
RepoRoot(openmnemo)
├── MainlineStage        仓库长期主线
│   ├── RepoMilestone    阶段里程碑
│   ├── RepoMilestone
│   └── RepoMilestone
├── WorkstreamBranch(memory)
├── WorkstreamBranch(report-chat)
├── WorkstreamBranch(gitea)

```

- `Decision 节点`
  - 为什么做了这个取舍
  - 比如 `Git commit 只做事件层，不等于运行时记忆对象`
- `Skill 节点`
  - 这条线沉淀出了什么方法论
  - 比如“如何做 unit-first recall”“如何做 mixed retrieval”

这里你已经有很好的基础了，因为当前代码里已经有 `memory_unit / source_asset / archive_anchor / DataLayerAPI` 这套对象，不是要推翻，而是让它们先成为仓库记忆树的底层素材。

仓库树的主线不要按“时间”单独组织，最好按“阶段 + 工作流”组织：

- `Phase`
- `Workstream`
- `Milestone`
- `Session milestone`

这样以后问：

- “这个仓库为什么走到今天这条路？”
- “report chat 这条线之前踩过哪些坑？”
- “这个项目沉淀了什么 skill？”

都能直接从 repo tree 回答。

**三、所有仓库怎么规划**
这里我不建议叫“所有仓库的一棵记忆树”。

我建议正式改名叫：

**Global Memory Forest**

因为全局层如果硬做成一棵树，会很快出问题：

- 一个 repo 同时属于多个主题
- 一个 skill 会跨很多 repo
- 一个决策可能同时影响工作、家庭、个人项目
- 分身隔离会变得很难做

所以全局层更适合：

**森林做压缩导航，图做跨域关联。**

建议结构：

```text
GlobalForest
├── ScopeRoot(work)
│   ├── RepoRoot(openmnemo)
│   ├── RepoRoot(client-a)
│   └── RepoRoot(content-studio)
├── ScopeRoot(personal)
│   ├── RepoRoot(life-os)
│   └── RepoRoot(reading-notes)
├── ScopeRoot(family)
└── ScopeRoot(shared)
```

再额外叠一张图：

- `Repo A` 的 skill 可连接 `Repo B`
- `家庭分区` 的偏好不能自动流进 `工作分区`
- `全局长期目标` 可以影响多个 repo
- `分身` 只是拿某几个 ScopeRoot 的授权视图

所以“每个分身都有自己的独立记忆树”这句话，我建议改成：

**每个分身都有自己独立的记忆树视图。**

不是复制一份物理树，而是：

- 一套底层资产
- 多个隔离视图
- 视图由 scope、persona、project、privacy policy 决定

这样才能同时满足：

- 隐私隔离
- 数据不重复
- 跨域可授权
- 分身可控

**四、针对不同模型上下文长度，怎么规划压缩**
这里不要按“模型标称窗口”做规划，要按：

**有效上下文预算**

来做。

公式建议就是：

```text
有效预算 = 模型总窗口
         - system prompt
         - tools/schema
         - 当前用户输入
         - 预留输出空间
```

真正能给记忆用的，通常只是其中一部分。  
所以压缩策略不要写“某模型 128k 就上多少”，而应该写成 `Budget Profile`。

我建议直接分 5 档：

**S 档：极小预算**
- 只给：
  - 当前轮
  - session stage node
  - 1 个 repo milestone
  - 1 个 raw snippet
- 不进全局层
- 最大 probe 深度到 branch

**M 档：小预算**
- 给：
  - 当前轮
  - session 主线
  - 1-2 个 branch
  - 1 个 repo milestone
  - 1 个 skill
- raw 只给片段
- 最大 probe 深度到 leaf

**L 档：中预算**
- 给：
  - session 主线
  - repo 当前 workstream
  - 相关 skill
  - 少量 global anchor
  - 2-4 个 raw snippets
- 可以做一次 cross-branch 关联

**XL 档：大预算**
- 给：
  - session tree 主线 + 多 branch
  - repo tree 当前阶段
  - global forest 中相关 scope
  - 多个 raw evidence
- 可支持复杂代码任务、规划任务、跨仓库对比

**XXL 档：超大预算**
- 仍然不要全量灌入
- 只是把 raw evidence 比例提高
- 适合长代码审查、长规划、长复盘

核心原则是：

- 小模型优先 `树`
- 大模型增加 `叶子和原文`
- 任何模型都不要直接全量喂历史

还可以按任务类型再加模板：

- `问答模板`
  - session 70%
  - repo 20%
  - global 10%
- `代码开发模板`
  - repo 50%
  - session 30%
  - skill 20%
- `复盘模板`
  - repo 40%
  - global 30%
  - session 30%
- `跨仓库灵感模板`
  - global 40%
  - repo 35%
  - session 25%

这样就清楚了：

- `树` 负责压缩和导航
- `图` 负责跨主题跨仓库关联
- `向量/FTS` 负责召回
