  硬编码偏移量 `+ 12` 是 `'\n## Messages\n'` 的长度（1 + 11 + 1 = 13，而非 12）。`'\n## Messages'` 是 12 个字符，`body` 的起始位置应是 `msgSection + 13`（跳过末尾换行符），或者 `msgSection + '\n## Messages\n'.length`。当前实现会将 `body` 起始于 `## Messages` 末尾字符（`s`）之后，而非换行符之后，如果 `## Messages` 后直接跟 `\n`，body 实际上以 `\n` 开头，对后续正则解析无影响，但若文件格式有变化（如无末尾换行），会导致偏移错误。这是一个隐藏的脆弱点（fragile magic number）。

- **[MEDIUM]** `build.ts:249-263` — `loadManifestsRecursive`：如果目录中存在同名（不同扩展名）文件，会被当作目录进行递归尝试（`readdirSync` 对普通文件会抛异常并被 catch 忽略），产生不必要的异常开销。应先用 `statSync` 或 `readdirSync` 的 `{ withFileTypes: true }` 选项区分文件和目录。

- **[LOW]** `build.ts:207-209` — 当 `!ghPagesBranch` 但 `cname` 有值时，动态 import `node:fs` 的 `writeFileSync`：

  ```ts
  const { writeFileSync: wf } = await import('node:fs')
  wf(join(output, 'CNAME'), cname + '\n', 'utf-8')
  ```

  `node:fs` 的 `writeFileSync` 在文件顶部已经静态导入（第7行），此处重复动态导入是冗余代码，应直接使用顶部导入的 `writeFileSync`。

### 文件：`heartbeat/config.ts`

- **[MEDIUM]** `config.ts:93-96` — `saveConfig` 中对 `readonly` 字段使用了 `?? DEFAULT_*` 的可选链：

  ```ts
  `locale = ${tomlString(cfg.locale ?? DEFAULT_LOCALE)}`,
  ```

  `Config` 接口中 `locale`、`gh_pages_branch`、`cname`、`webhook_url` 均为非可选的 `readonly string`，TypeScript 类型上这些字段不可能为 `undefined`，`?? DEFAULT_*` 是多余的（编译时有 `strictNullChecks`）。更深层的问题是：若传入的 `Config` 对象在运行时这些字段真的是 `undefined`（比如来自外部源），类型系统没有捕获，却被 `??` 静默处理。

- **[LOW]** `config.ts:267-270` — `tomlString` 函数：

  ```ts
  function tomlString(value: string): string {
    const escaped = value.replace(/\\/g, '\\\\').replace(/"/g, '\\"')
    return `"${escaped}"`
  }
  ```

  未转义 TOML 字符串中的其他特殊字符：换行符（`\n`）、回车符（`\r`）、制表符（`\t`）。若用户的 `webhook_url` 或其他字段包含换行符（理论上不应该，但未在输入验证层阻止），生成的 TOML 文件会语法错误。

- **[LOW]** `config.ts:148` — `registerProject` 中：

  ```ts
  const name = resolved.split('/').pop() ?? ''
  ```

  若 `resolved` 为根路径（`/`），`split('/').pop()` 返回空字符串 `''`，name 为空，这在 `saveConfig` 中会跳过 name 写入（`if (project.name)`），行为正确但 edge case 隐晦。
