## CRITICAL

### C1 — XSS: Raw search index JSON injected directly into `<script>` block
**File:** `src/report/render/search.ts`, line 62–66

```ts
const indexJson = JSON.stringify(index)
const extraHead = `<script>
const SEARCH_INDEX = ${indexJson};
</script>`
```

`JSON.stringify` does NOT escape `</script>` sequences. If any manifest field (title, snippet, client, session_id) contains the string `</script>`, the embedded block is syntactically terminated and the remainder becomes raw HTML — a classic script injection vector. An attacker (or corrupted transcript) whose title is `</script><script>alert(1)</script>` achieves arbitrary JS execution in the local report.

**Fix:** Replace `</` with `<\/` using a post-serialization replace:
```ts
const indexJson = JSON.stringify(index).replace(/<\//g, '<\\/')
```

### C2 — XSS: `clientBadge()` injects un-sanitised `client` string into a CSS class name
**File:** `src/report/render/dashboard.ts`, line 138–140

```ts
export function clientBadge(client: string): string {
  return `<span class="badge badge-${escHtml(client)}">${escHtml(client)}</span>`
}
```

The class attribute value `badge-${escHtml(client)}` is HTML-entity-escaped, but that is not sufficient for a CSS class name injected into an attribute. A manifest with `client = 'codex" onmouseover="alert(1)'` (after unescaping) becomes:

```html
<span class="badge badge-codex" onmouseover="alert(1)">…</span>
```

`escHtml` converts `"` to `&quot;` in the *text content* position but HTML parsers re-parse attribute context differently. Actually `escHtml` does escape `"` to `&quot;` which prevents the attribute break — however this is fragile because the class name itself can contain characters that are semantically unexpected (`<`, `>`, spaces). A client of `foo bar` produces `class="badge badge-foo bar"` which splits the class list unexpectedly. The deeper issue is that the CSS class scheme should only be applied for known safe values. In addition, `clientBadge` is called from `transcript-list.ts` and `transcript.ts` where `m.client` comes from attacker-controlled manifest JSON.

**Fix:** Allowlist the CSS class addition: only append the known class for `codex`, `claude`, `gemini`; otherwise use a generic `badge-unknown` class.

### C3 — XSS: `markdownToHtml` link renderer emits unescaped label after prior `escHtml`
**File:** `src/report/render/markdown.ts`, line 130–132

```ts
.replace(/\[([^\]]+)\]\(([^)]+)\)/g, (_, label: string, href: string) => {
  return `<a href="${escHtml(href)}" target="_blank" rel="noopener">${label}</a>`
})
```

At this point `applyInline` has already called `escHtml(text)` on the whole string (line 123), so `label` is a regex capture group from the already-escaped string. However `href` is also captured from the escaped string — meaning that after `escHtml` on the outer text converts `"` to `&quot;`, the regex still matches `[text](href)` forms only if neither contained `&quot;`, `&lt;` etc. This creates a correctness issue: links containing `"` in their href will not match because `escHtml` runs first and transforms the raw `"` into `&quot;`, making the `[...]( )` pattern fail to match. More critically, for the case where the regex DOES match: the `href` value comes from the already-HTML-escaped string, then `escHtml(href)` is applied again — double-escaping `&` to `&amp;amp;` etc. The label is emitted raw (from the already-escaped capture) — so XSS through label is not possible, but double-escaping of `href` corrupts all URLs that contain `&` (e.g. query strings).

**Fix:** `applyInline` should receive raw text and perform escaping per-segment, or the link regex must run before `escHtml`.

### H1 — Memory: Semaphore race condition — not truly one-at-a-time, and `activeCalls` is module-level mutable state
**File:** `src/report/summarize.ts`, lines 30–43

```ts
let activeCalls = 0
const MAX_CONCURRENT = 5

async function withSemaphore<T>(fn: () => Promise<T>): Promise<T> {
  while (activeCalls >= MAX_CONCURRENT) {
    await new Promise<void>(resolve => setTimeout(resolve, 100))
  }
  activeCalls++
  ...
```

This is a busy-poll loop with a TOCTOU race. Between the `while` check returning false and `activeCalls++` executing, another promise continuation can run (Node event loop is single-threaded but yields at `await`). Specifically: two concurrent callers both see `activeCalls = 4`, both exit the loop, both increment — producing `activeCalls = 6 > MAX_CONCURRENT`. Under high manifests counts this allows more than 5 concurrent API calls.

Additionally, `activeCalls` is module-level state. If `buildReport` is ever called twice concurrently (e.g. from a test suite or two heartbeat triggers), the counter is shared and will corrupt both runs.

**Fix:** Use a proper semaphore that does the check-and-increment atomically within the microtask queue, or use a queue-based concurrency limiter (e.g. p-limit). As a minimal fix, increment *before* the await:
```ts
// The correct pattern: increment speculatively, decrement if over limit
// Better: use a Promise-chain queue
```

### H2 — Logic error: `isoWeek` Thursday calculation is wrong
**File:** `src/report/stats.ts`, lines 140–146  
**File:** `src/report/render/charts.ts`, lines 264–271 (identical formula)

```ts
thursday.setDate(d.getDate() + (4 - ((d.getDay() + 6) % 7 + 1)))
```

`d.getDay()` returns 0=Sunday…6=Saturday. The standard ISO week Thursday formula is:

```ts
thursday.setDate(d.getDate() + (4 - d.getDay() + (d.getDay() === 0 ? -6 : 1)))
// or equivalently:
thursday.setDate(d.getDate() + 4 - ((d.getDay() + 6) % 7))
```

The correct expression adjusting for Monday=1 ISO day is `4 - ((d.getDay() + 6) % 7)`.

The code has `(d.getDay() + 6) % 7 + 1` where the `+ 1` is extra. Trace through:
- Sunday: `d.getDay()=0` → `(0+6)%7+1 = 6+1 = 7` → offset `= 4-7 = -3` → Thursday is 3 days before Sunday = Thursday. Correct by coincidence.
- Monday: `d.getDay()=1` → `(1+6)%7+1 = 0+1 = 1` → offset `= 4-1 = 3` → Monday + 3 = Thursday. Correct.
- Saturday: `d.getDay()=6` → `(6+6)%7+1 = 5+1 = 6` → offset `= 4-6 = -2` → Saturday - 2 = Thursday. Correct.
- Thursday: `d.getDay()=4` → `(4+6)%7+1 = 3+1 = 4` → offset `= 4-4 = 0` → Thursday stays Thursday. Correct.

Actually all cases check out. The formula is algebraically equivalent to the standard form despite the unusual expression — no week number bug. **Retracting severity to LOW** (style/readability concern only, not a logic bug).

### H3 — Path traversal in local HTTP server
**File:** `src/cli/cmd-report.ts`, lines 62–64

```ts
const urlPath = (req.url ?? '/').split('?')[0] ?? '/'
const safePath = urlPath.replace(/\.\./g, '')
const filePath = join(root, safePath)
```

The sanitisation replaces literal `..` but does not handle:
- URL-encoded forms: `%2e%2e`, `%2e.`, `.%2e`
- `....` (produces `..` after strip: `"....".replace(/\.\./g, '')` → `""` — safe, but `"..%2f"` etc. are not handled)
- Unicode normalised paths on some OS

More critically, `join(root, safePath)` with an absolute-looking path like `/%2fabsolute%2fpath` after URL decoding could escape the root on some platforms. The `join` call does not verify the result is still under `root` (no `startsWith(root)` check).

For a **localhost-only** dev server this is LOW risk in practice, but serves as a teaching moment. The `req.url` should be decoded with `decodeURIComponent` (in a try/catch), and the final `filePath` should be validated to start with `root`.

### H4 — `parseMessagesFromMarkdown`: off-by-one in slice offset


### M2 — Doughnut `stroke-dashoffset` formula is incorrect
**File:** `src/report/render/charts.ts`, line 142

```ts
stroke-dashoffset="${(-offset * circumference / 1).toFixed(2)}"
```

The `/ 1` is a no-op (division by 1). The standard SVG doughnut technique requires rotating by `circumference * offset` and then correcting for the default SVG dash start position. The correct formula should be:

```ts
// After rotating -90deg, the dashoffset for the i-th segment is:
// -(circumference * accumulated_fraction_before_this_segment)
```

The current code emits `(-offset * circumference / 1)`. At this point `offset` is the fraction *already accumulated before adding the current segment's fraction*, so the formula computes the correct negated dashoffset. The `/ 1` is vestigial (possibly a remnant of `/ total` being removed). The output is numerically correct but the expression is misleading and the dead `/ 1` should be cleaned up.

**Severity: LOW** — no bug, just dead code noise.

### M3 — `buildSearchIndex` budget calculation double-counts
**File:** `src/report/render/search.ts`, lines 33–41

```ts
const entryBytes = JSON.stringify(entry).length
if (totalBytes + entryBytes > MAX_INDEX_BYTES) {
  const remaining = MAX_INDEX_BYTES - totalBytes - (entryBytes - snippet.length) - 10
  if (remaining < 20) break
  snippet = snippet.slice(0, remaining)
  entry.snippet = snippet
}
totalBytes += JSON.stringify(entry).length  // ← re-serialises after mutating entry
entries.push(entry)
```

When the budget overflow branch is NOT taken, `totalBytes` is incremented by `JSON.stringify(entry).length` which is correct. When the overflow branch IS taken and the snippet is truncated, `totalBytes` is also incremented by `JSON.stringify(entry).length` of the *mutated* entry. This double-serialisation is fine for correctness, but the `remaining` calculation has a subtle bug:

`entryBytes - snippet.length` approximates the non-snippet overhead. But `snippet.length` here is the *original* snippet length (before truncation) and `entryBytes` already includes the original snippet. The approximation is valid. However: `JSON.stringify(entry).length` after mutation will be slightly different from `MAX_INDEX_BYTES - totalBytes` because `remaining` already subtracted 10 as a padding fudge. The budget can still slightly exceed `MAX_INDEX_BYTES` if the JSON overhead of `snippet` changes (e.g. escaped characters). This is a minor correctness issue, not a crash risk.

### M4 — `loadManifestsRecursive` does not distinguish files from directories
**File:** `src/report/build.ts`, lines 166–182

```ts
for (const entry of entries) {
  const fullPath = join(dir, entry)
  if (entry.endsWith('.json')) {
    // parse as manifest
  } else {
    // Recurse into subdirectories
    loadManifestsRecursive(fullPath, out)
  }
}
```

Any non-`.json` file (e.g. `.DS_Store`, `README.md`, a binary) is recursed into with `readdirSync`. On a directory it silently fails (caught exception), but if there is a deep tree of non-json, non-directory files, this causes an O(n) `readdirSync` calls that all fail. More critically, if the `manifests/` directory ever contains a non-json regular file, this passes it as a directory to `loadManifestsRecursive`, which catches the `readdirSync` error and continues silently. This is not a crash but is wasteful. Use `statSync` or `lstatSync` to only recurse into actual directories.

### M6 — `renderHeatmap` loop termination condition is confusing and may produce 54 weeks
**File:** `src/report/render/charts.ts`, line 43

```ts
while (cur <= today || weeks.length < 53) {
  if (weeks.length >= 53) break
```

The outer `while` condition uses `||`, meaning it continues while EITHER the date is before today OR fewer than 53 weeks are built. The inner `break` immediately exits if 53 weeks are reached. The combination means: if `today` is already past and `weeks.length < 53`, it keeps looping; but if `weeks.length >= 53`, it breaks. The net effect is always exactly 53 weeks. **No bug**, but the logic is hard to follow. Should be simplified to `while (weeks.length < 53)`.

**File:** `src/report/render/charts.ts`, lines 36, 82

Day labels are placed at `x="-2"`. With `leftPad=0`, the week columns start at `x=0`.
