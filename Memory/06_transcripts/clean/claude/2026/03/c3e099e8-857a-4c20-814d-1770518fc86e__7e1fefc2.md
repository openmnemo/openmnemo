```typescript
function openDb(dbPath: string): InstanceType<typeof Database> {
  const dir = dirname(dbPath)
  if (!existsSync(dir)) mkdirSync(dir, { recursive: true })
  const db = new Database(dbPath)
  db.pragma('journal_mode = WAL')
  return db
}

export function searchTranscripts(dbPath: string, query: string, limit = 20): SearchResult[] {
  const sanitized = sanitizeFtsQuery(query)
  if (!sanitized) return []
  if (!existsSync(dbPath)) return []

  const db = new Database(dbPath, { readonly: true })
  try {
    db.exec(CREATE_FTS_SQL)

    return stmt.all(sanitized, limit) as SearchResult[]

    db.close()
  }
}
```

## CORRECTNESS

**[CRITICAL] `searchTranscripts` opens DB in readonly mode then tries to `CREATE VIRTUAL TABLE`**

```typescript
const db = new Database(dbPath, { readonly: true })
// ...
db.exec(CREATE_FTS_SQL)  // will throw: attempt to write a readonly database
```

`CREATE VIRTUAL TABLE IF NOT EXISTS` is a write operation. On a fresh DB or one where the FTS table doesn't exist yet, this throws. The FTS table must be created during `upsertSearchIndex`, not lazily in the read path.

**[CRITICAL] `segdir` check is wrong and fragile**

```typescript
const segCount = (db.prepare('SELECT COUNT(*) as cnt FROM transcripts_fts_segdir').get() as { cnt: number }).cnt
if (segCount === 0) db.exec(REBUILD_FTS_SQL)
```

`transcripts_fts_segdir` is an internal fts4 shadow table. It may not exist if the FTS table was just created, causing a throw. More importantly, `segdir` row count is not a reliable proxy for "is the index populated" — it reflects internal segment merging state, not whether rows are indexed. A freshly rebuilt index with few rows can have 0 segdir entries. The correct approach is to always rebuild after upsert (which `upsertSearchIndex` already does), and remove this check entirely from the read path.

**[HIGH] `db.prepare(UPSERT_SQL).run(...params)` — spread of array into variadic**

`better-sqlite3`'s `Statement.run()` accepts either positional args or a single object. Spreading a large array works in JS but the binding count must exactly match the `?` placeholders. If `ALL_COLUMNS` ever drifts from the actual table schema (e.g. a column is added to `DATA_COLUMNS` but not `CREATE_TABLE_SQL`), this silently binds wrong values or throws at runtime with no clear error. Prefer passing the array directly: `stmt.run(params)` — better-sqlite3 accepts an array as the first argument when using `?` placeholders.

**[HIGH] `findLatestSession` returns `null` instead of falling back when no match found**

```typescript
  } catch {
    return findLatestFromJsonl(...)
  }
}
return null  // ← no fallback here
```

The catch block falls back to JSONL, but if the DB exists and is readable yet simply has no matching row, the function returns `null` without trying `findLatestFromJsonl`. The original sql.js version's intent should be checked — if JSONL fallback is the intended behavior for "no result", this is a regression.

## EDGE CASES

**[HIGH] No WAL pragma in `searchTranscripts`**

`openDb` sets `journal_mode = WAL`, but `searchTranscripts` constructs its own `new Database(dbPath, { readonly: true })` directly, bypassing `openDb`. A readonly connection doesn't need to set WAL (the mode is set by the writer), but this inconsistency means if `searchTranscripts` is ever called before any write has occurred, the DB will be in DELETE journal mode for that connection. Not a crash, but worth noting the two code paths are inconsistent.

**[MEDIUM] `upsertSearchIndex` does a full FTS rebuild on every single upsert**

```typescript
db.exec(REBUILD_FTS_SQL)  // runs after every single manifest insert
```

`INSERT INTO transcripts_fts(transcripts_fts) VALUES('rebuild')` does a full table scan and reindex. For bulk imports this is O(n²). The rebuild should be deferred to after a batch completes, not called per-row. At minimum, expose a separate `rebuildFtsIndex(dbPath)` function that callers invoke once after a batch.

**[MEDIUM] Race condition on concurrent writes**

`upsertSearchIndex` opens a new `Database` instance on every call with no connection pooling or locking beyond WAL. WAL handles concurrent readers fine, but two simultaneous writers will serialize via SQLite's busy timeout (default: 0ms in better-sqlite3, meaning immediate `SQLITE_BUSY` throw). There's no `db.pragma('busy_timeout = 5000')` set. Under parallel import this will throw.

**[LOW] `existsSync` + `mkdirSync` in `openDb` is a TOCTOU race**

Minor: between the `existsSync` check and `mkdirSync`, another process could create the directory. `mkdirSync(dir, { recursive: true })` is idempotent and doesn't throw if the dir exists, so just remove the `existsSync` guard and call `mkdirSync` unconditionally.

---

## BEST PRACTICES

**[MEDIUM] Prepared statements are not reused across calls**

Every call to `upsertSearchIndex` and `searchTranscripts` creates a new `Database` instance and re-prepares statements. better-sqlite3's primary performance advantage over sql.js is that prepared statements are compiled once and reused. The current pattern throws away that benefit. A module-level connection cache (keyed by `dbPath`) with pre-prepared statements would be the idiomatic approach.

**[LOW] `CREATE_TABLE_SQL` and `CREATE_FTS_SQL` are re-executed on every upsert**

`db.exec(CREATE_TABLE_SQL)` and `db.exec(CREATE_FTS_SQL)` run on every call to `upsertSearchIndex`. SQLite handles `IF NOT EXISTS` cheaply, but it's still unnecessary overhead and signals the schema initialization isn't separated from the data write path.

---

## SECURITY

**[MEDIUM] `sanitizeFtsQuery` strips too aggressively for non-Latin scripts**

```typescript
return raw.replace(/[^\p{L}\p{N}\s']/gu, ' ').trim()
```

This is used correctly with `\p{L}` Unicode categories, so CJK/Arabic/etc. characters pass through. However, the single-quote `'` is allowed through, and in fts4 `MATCH` syntax, unbalanced quotes cause a parse error that will throw at the SQLite level. The query should either strip single quotes or wrap the sanitized string in a try/catch at the `stmt.all()` call site.

**[LOW] No path validation on `dbPath`**

`dbPath` is constructed by callers from `globalRoot` which comes from config. If config is user-controlled and not validated upstream, a crafted path could write the SQLite file to an arbitrary location. This is low severity given the threat model, but worth noting if `globalRoot` isn't validated at config load time.

---

## Summary

The two CRITICAL issues (write op in readonly connection, broken segdir check) will cause runtime failures in production. The HIGH issues around `findLatestSession` fallback behavior and the per-row FTS rebuild are likely regressions from the sql.js version. Fix those five before shipping.
