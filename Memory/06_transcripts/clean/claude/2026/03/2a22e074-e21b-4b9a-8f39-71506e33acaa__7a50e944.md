## HIGH

### H1. `release.yml`: The version-commit push can re-trigger itself (potential infinite loop)

The workflow triggers on `push: branches: [main]`. The "Version packages" step does `git push` to `main`. That push will fire the workflow again. On the second run, the changeset `.md` files have been consumed, so `steps.changesets.outputs.has` will be `false`, and it falls through to the publish step -- so it does terminate. However, this relies on `changeset version` always deleting every `.md` file in a single run. If `changeset version` ever partially fails (e.g., a malformed changeset file survives), the workflow would loop: version -> push -> version -> push.

Mitigation options:
- Add `[skip ci]` to the commit message: `git commit -m "chore: release [skip ci]"`
- Or use a conditional like `if: github.actor != 'github-actions[bot]'` on the job
- Or push with a PAT/app token and use a path filter to break the cycle

### H2. `release.yml`: Publish step lacks `--provenance` flag

The manual `publish.yml` uses `--provenance` for npm provenance attestation (and has `id-token: write` for it). The automated `release.yml` also requests `id-token: write` but the `pnpm release` script in `package.json` does not pass `--provenance`. The permission is granted but unused, and published packages won't have provenance attestation from the automated pipeline.

Fix: change the root `package.json` release script to:
```
"release": "pnpm build && pnpm publish -r --access public --no-git-checks --provenance"
```

## MEDIUM

### M1. `release.yml`: `git push` uses the default `GITHUB_TOKEN` -- no branch protection bypass

The `GITHUB_TOKEN` cannot push to a branch with branch protection rules that require PR reviews or status checks. If branch protection is ever enabled on `main`, this step will silently fail. This is fine today if there's no branch protection, but worth noting as a latent issue.

### M2. `release.yml`: Missing build/test/lint before publish

The publish step runs `pnpm release` which calls `pnpm build && pnpm publish ...`, but there is no `typecheck`, `lint`, or `test` step. The manual `publish.yml` runs all three before publishing. A broken package could be published if a changeset-less push lands on `main` with a type error.

Mitigation: add `pnpm typecheck && pnpm lint && pnpm test` before the publish step, or gate the release job on the CI workflow succeeding (e.g., `needs: [ci]` or a required status check).

### M3. `release.yml`: Concurrency group has no `cancel-in-progress`

```yaml
concurrency: release-${{ github.ref }}
```

This will queue a second run if one is already in progress. For a release pipeline, that's probably fine (you don't want to cancel a publish midway). But if the intent is to prevent the re-triggered run from overlapping with the version run, adding `cancel-in-progress: false` explicitly documents the intent. As-is, the default is `false`, so behavior is correct -- this is just a clarity nit.

### M4. CLI: `init` alias `i` conflicts with `import` alias `imp` prefix

No actual conflict today since `i` and `imp` are distinct strings. However, `i` is a common mental shorthand for "import" and could confuse users. Commander resolves aliases exactly (not by prefix), so there's no runtime bug -- this is a UX concern only.

### M5. CLI: `daemon install` alias `in` shadows a potential future subcommand

The alias `in` for `daemon install` is a reserved word in JavaScript (`for...in`). Commander doesn't care about JS reserved words in alias strings, so no runtime issue. But `in` is also a plausible short name for a future `daemon info` subcommand. Minor namespace concern.
