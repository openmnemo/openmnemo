function writeVbsLauncher(scriptPath: string): string {

### HIGH

#### H1: `vbsPath` 路径本身包含空格时 schtasks `/tr` 参数会被截断

```typescript
const trCommand = `wscript.exe /B "${vbsPath}"`
```

`trCommand` 是作为 **单个字符串** 传递给 `execCommand`，而 `execCommand` 内部使用 `execFileSync`，其第二个参数是 args 数组——但这里 `trCommand` 是整体拼入 `/tr` 参数的字符串值，最终由 schtasks 自己解析。

问题在于 `schtasks /tr` 接收的值会被 schtasks.exe 的命令行解析器二次解析。如果 `vbsPath` 是 `C:\Users\John Doe\.memorytree\heartbeat-launcher.vbs`，外层引号已经正确包裹，这部分实际上是 OK 的。

但更深层问题是：`vbsPath` 本身**不做任何转义就直接插入**，若路径含有 `"` 字符（虽然 Windows 路径名不能含双引号，但 UNC 路径等极端情况），会破坏引号平衡。此风险极低但值得注意。

**更实质的 HIGH 问题是**：`trCommand` 在传给 `/tr` 参数时，schtasks 要求 `/tr` 的值本身**再次用外层引号包裹**才能包含空格，而这里 `trCommand` 没有做这层额外的引号处理。实测中 schtasks 对含空格的 `/tr` 值的处理行为因 Windows 版本不同而异——部分版本要求整个命令串用额外引号括住，部分版本不需要。当前写法在路径含空格时存在静默注册失败或任务动作被截断的风险。

### MEDIUM

#### M1: VBScript 中 `node` 使用裸命令，依赖 PATH，但 schtasks 任务在 SYSTEM 账户或受限环境下运行时 PATH 可能不含 node

```vbs
WshShell.Run "node ""...""", 0, False
```

Task Scheduler 任务默认以当前用户身份运行（因为 schtasks 没有指定 `/ru`），用户的 PATH 通常包含 node。但如果管理员改变了任务运行账户，或者 node 只安装在用户级别的 nvm/fnm 管理器下，PATH 中可能没有 `node`。原始代码有同样问题，此变更未改善也未恶化，但值得记录。

**建议**：使用 `process.execPath`（当前 Node.js 可执行文件的绝对路径）代替裸 `node`，同时 VBScript 里对该路径做 `""` 转义。

---

#### M2: VBScript 使用 `Run(..., 0, False)` — `False` 表示不等待，无法获知 heartbeat 是否执行成功

```vbs
WshShell.Run "node ...", 0, False
```

第三个参数 `False` 表示异步执行（fire-and-forget）。Task Scheduler 本身已经是定时触发，异步是合理的；但这意味着即使 node 脚本崩溃或路径错误，Task Scheduler 仍会报告任务"成功"启动，用户通过 `status` 子命令无法感知执行失败。这是一个可观测性问题而非正确性问题。

---

#### M3: `writeVbsLauncher` 写入文件编码为 `utf-8`，但 Windows 上的 wscript.exe 默认以 ANSI/系统代码页读取 `.vbs` 文件

```typescript
writeFileSync(vbsPath, content, 'utf-8')
```

VBScript 文件内容目前只含 ASCII 字符，所以 UTF-8 与 ANSI 二进制相同，实际上没有问题。但如果未来 `scriptPath` 中含有非 ASCII 字符（中文用户名、中文路径），UTF-8 编码的 VBScript 在没有 BOM 的情况下会被 wscript.exe 误解析。

高风险场景：用户名为中文的 Windows 用户，`homedir()` 返回含中文路径，VBScript 中的路径字符串会出现乱码，导致任务静默失败。

---

#### M4: `uninstallSchtasks` 中 `existsSync` + `unlinkSync` 存在 TOCTOU（检查后操作）竞态

```typescript
if (existsSync(vbsPath)) {
  try { unlinkSync(vbsPath) } catch { /* ignore */ }
}
```

两次系统调用之间文件可能被删除，导致竞态。实际上 `unlinkSync` 已经被 `try/catch` 包裹，所以结果是正确的——直接去掉 `existsSync` 检查更干净，但当前写法功能上没有 bug，只是多余的防御性代码带来了轻微逻辑噪音。

### LOW

#### L1: 函数注释描述不完整

```typescript
// wscript.exe /B suppresses the window; VBScript Run(..., 0, False) hides it.
```

`/B` 是 wscript 的"batch mode"标志（抑制脚本错误弹窗），实际上**不抑制 cmd 窗口**（wscript.exe 本身就没有 cmd 窗口）。真正隐藏窗口的是 VBScript `Run` 的第二个参数 `0`（即 `SW_HIDE`）。注释将两者的作用混淆了，对后续维护者有误导性。

---

#### L2: `installSchtasks` 失败时的错误信息不够具体

```typescript
process.stderr.write('Failed to create scheduled task.\n')
```

`execCommand` 抛出的异常被静默丢弃（`catch { }` 空块），schtasks 的实际错误输出（如权限不足、任务名冲突）丢失。至少应该将 `err` 的 `stderr` 字段输出给用户，便于排查。

---

#### L3: `vbsLauncherPath()` 与 `writeVbsLauncher()` 是独立函数，但 uninstall 侧直接调用 `vbsLauncherPath()`，路径逻辑分散

当前结构是合理的，但如果未来路径计算逻辑变化，需要同时修改两处（`writeVbsLauncher` 里的 `mkdirSync(dirname(vbsPath))` 和 `vbsLauncherPath()` 本身），轻微的 DRY 隐患。实际上两处已经都通过 `vbsLauncherPath()` 取路径，这点是 OK 的——只是 `mkdirSync` 也在 `writeVbsLauncher` 内部，uninstall 不需要，结构已经合理。此条可忽略。
