## Second-Pass Review: sql.js → better-sqlite3 Migration

---

### CRITICAL

**C1. `db.prepare(UPSERT_SQL).run(...params)` — spread of 18 args hits V8 argument limit on Node 22/24**

`better-sqlite3`'s `Statement.run()` accepts either a rest spread or a single array. Spreading 18 values is fine today, but the real problem is that `run(...params)` where `params` is `unknown[]` is not type-safe — TypeScript infers the spread as `unknown[]` and the call compiles only because `run` is typed as `run(...bindParameters: BindingDictionary): RunResult`. The actual runtime type of each element is `string | number`, but TypeScript cannot verify that because `params` is built via a `.map()` returning `string | number | ''`. If `ManifestEntry` ever gains a field whose value survives the `typeof value === 'number'` branch as something else (e.g. a boolean from a future schema change), it silently coerces to a string — no compile error, no runtime error, wrong data. The fix is `db.prepare(UPSERT_SQL).run(params)` (single array argument), which is also what reviewer 1 flagged for a different reason. Noting here because the type-safety angle was missed.

**C2. FTS4 content table `docid = t.rowid` join is broken after any DELETE or VACUUM**

`SEARCH_SQL` joins on `transcripts_fts.docid = t.rowid`. With a content table (`content='transcripts'`), FTS4 stores the rowid of the source row at index time. If any row in `transcripts` is deleted and re-inserted (which the upsert does via `ON CONFLICT DO UPDATE` — SQLite's `UPDATE` path preserves rowid, but a future `DELETE` + `INSERT` would not), or if `VACUUM` is run (which renumbers rowids in non-WAL mode), the docid→rowid mapping silently diverges. The `REBUILD_FTS_SQL` re-syncs it, but only if called. Since `searchTranscripts` only rebuilds when `segdir` count is zero (already flagged as fragile by reviewer 1), a post-vacuum search returns rows joined to the wrong transcript. This is a data-correctness bug, not just a reliability issue.

### HIGH

**H1. `findLatestSession` — `resolve(cwd)` called with an arbitrary DB string value on line 226 of recall.ts**

```typescript
const cwdResolved = toPosixPath(resolve(cwd)).toLowerCase()
```

`cwd` here comes directly from a SQLite row (`String(row['cwd'] ?? '')`). `resolve()` on Windows will interpret a value like `C:\Users\...` correctly, but a value like `\\server\share\path` (UNC path stored by a Windows user) causes `resolve()` to return an unexpected result. More critically, if the DB row contains a relative path (e.g. `project`), `resolve()` silently anchors it to the current process working directory, making `cwdMatches` return a false positive. There is no validation that the stored `cwd` is absolute before calling `resolve()`.

**H2. `import.ts` line 105 — fire-and-forget `upsertSearchIndex` swallows all errors silently**

```typescript
upsertSearchIndex(globalDbPath, manifest)  // sync, no await
```

`upsertSearchIndex` is now synchronous and can throw (e.g. disk full, permission denied, better-sqlite3 native binding not found). Previously it was `await`-ed inside a `try/catch` in the caller. Now the call is bare — any throw propagates up through `importTranscript` and will cause the entire import to fail, including the already-written manifest JSON and clean markdown. The caller (`syncCurrentProject` in recall.ts) catches per-transcript errors and continues, so the import count will be wrong (session counted as failed when files were actually written). The fix is wrapping the call in a try/catch and logging rather than throwing.

**H3. `ManifestEntry` fields not in `ALL_COLUMNS` are silently dropped — type system gives no warning**

`ALL_COLUMNS` is missing `cleaning_mode`, `repo_mirror_enabled`, and `global_manifest_path` (present in `ManifestEntry` per import.ts lines 85–87). The `params` map casts `manifest` to `Record<string, unknown>` and iterates only `ALL_COLUMNS`, so those three fields are never written to SQLite. `findLatestSession` returns a raw DB row, and callers in recall.ts do `String(session['global_clean_path'] ?? '')` — but `global_clean_path` IS in `ALL_COLUMNS`, so that specific one is fine. However, any future caller that reads `cleaning_mode` or `repo_mirror_enabled` from the returned row will silently get `undefined`. TypeScript cannot catch this because the return type is `Record<string, unknown>`. The schema and `ALL_COLUMNS` are not derived from the same source of truth.

### MEDIUM

**M1. Module-level `ALL_COLUMNS` is a `readonly` tuple of string literals — but `UPSERT_SQL` is built at module load time with no schema validation**

`UPSERT_SQL` is constructed by joining `ALL_COLUMNS` at module evaluation time. If `ManifestEntry` (from `@openmnemo/types`) is updated to add or rename a column, `ALL_COLUMNS` must be manually kept in sync. There is no compile-time check that `ALL_COLUMNS[number]` is a subset of `keyof ManifestEntry`. The cast `(manifest as Record<string, unknown>)[col]` suppresses the type error. A `satisfies` constraint or a mapped type check would catch drift at compile time.

**M2. `searchTranscripts` test suite never tests against a DB that was NOT populated by `upsertSearchIndex` in the same test**

All `searchTranscripts` tests call `upsertSearchIndex` first in the same process. This means the FTS table always exists and is always populated before search runs. The critical path — a DB that has the `transcripts` table but no `transcripts_fts` table (legacy DB, the stated use case in the comment on line 153 of db.ts) — is never exercised. The `CREATE VIRTUAL TABLE IF NOT EXISTS` + readonly bug (already flagged by reviewer 1) would be caught by such a test.

**M3. `sanitizeFtsQuery` single-quote passthrough — test exists but tests the wrong thing**

The test at line 263 checks that `'"auth* (bug)-'` becomes `'auth bug'` — the double-quote is stripped but the single quote in the input is not present in that test string. There is no test for input like `it's` or `O'Brien` which produces `it's` or `O'Brien` — a string with an unbalanced single quote. FTS4 treats single quotes as string delimiters in MATCH expressions. `MATCH 'it''s'` is valid SQL (escaped), but `MATCH it's` (after sanitization strips nothing) is a parse error. The test suite does not cover this case, so the known bug from reviewer 1 has no regression test.

**M4. `findLatestSession` — `activationTime` comparison uses lexicographic string compare on ISO timestamps**

```typescript
db.prepare('SELECT * FROM transcripts WHERE started_at < ? ORDER BY started_at DESC LIMIT 20').all(activationTime)
```

This works correctly only if `started_at` values are consistently formatted ISO 8601 with the same precision and timezone offset. If any transcript has `started_at` stored as a Unix timestamp integer, a locale-formatted date, or an ISO string with a non-UTC offset (e.g. `2024-06-01T08:00:00+08:00`), the lexicographic comparison gives wrong results. The JSONL fallback (line 200) has the same issue: `if (started >= activationTime) continue`. There is no normalization or SQLite `datetime()` coercion applied.

### LOW

**L1. `better-sqlite3` import in `recall.ts` is a direct `import Database from 'better-sqlite3'` — duplicates the import already in `db.ts`**

`recall.ts` imports `better-sqlite3` directly for `findLatestSession`, while `db.ts` also imports it. This means two separate modules both hold a reference to the native addon. On most platforms this is fine (Node caches native modules), but it creates a maintenance coupling: if the import is ever aliased or mocked in tests, both files must be updated. `findLatestSession` should use a thin wrapper from `db.ts` (e.g. a `queryTranscripts` export) rather than opening SQLite directly.

**L2. `db.test.ts` — `makeManifest` uses hardcoded Unix paths for `cwd`, `global_raw_path`, etc.**

```typescript

global_raw_path: '/global/raw/file.jsonl',

Any test that exercises the full `findLatestSession` → `cwdMatches` path with these fixtures will produce false negatives on Windows. The test suite currently does not test `findLatestSession` directly, so this is latent, but it will bite when those tests are added.

**L3. `recall.ts` — `syncCurrentProject` calls `importTranscript` which calls `upsertSearchIndex` synchronously, but `syncCurrentProject` is still `async` and its callers `await` it**

This is correct and not a bug, but the async boundary is now misleading: `syncCurrentProject` is `async` only because it `await`s `importTranscript`, which is `async` only because of other I/O inside it (not because of the DB call). If a future refactor removes those other awaits, the function silently becomes a synchronous operation wrapped in a resolved Promise, which is fine but wastes a microtask tick per transcript. **L4. `db.test.ts` — no test for `upsertSearchIndex` with a manifest that has `undefined` or `null` fields**

The `params` map in `upsertSearchIndex` converts `undefined`/`null` to `''`. There is no test that passes a manifest with missing optional fields (e.g. `branch: undefined`) and verifies the stored value is `''` rather than the string `'undefined'`. The `typeof value === 'number'` branch also means a `0` message count is stored correctly, but there is no test for `message_count: 0` specifically.
