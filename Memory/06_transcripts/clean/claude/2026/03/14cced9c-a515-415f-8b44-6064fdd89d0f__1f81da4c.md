### HIGH

#### H1: `vbsPath` 本身含双引号时 `trCommand` 会崩溃

```typescript
const trCommand = `wscript.exe /B "${vbsPath}"`
```

`vbsPath` 来自 `resolve(homedir(), '.memorytree', 'heartbeat-launcher.vbs')`。

`homedir()` 在 Windows 上通常不含引号，但并未做任何转义。若 homedir 路径中含双引号（极端情况，但理论上可能），`trCommand` 字符串就会被 schtasks 解析错误。

更重要的是：此字符串作为 `/tr` 参数的值传入 `execCommand`，而 `execFileSync` 会将整个字符串作为单一参数传给 schtasks。schtasks 自己再解析 `/tr` 参数值时按 CMD 语法处理，内嵌引号如果不规范会导致任务注册成功但运行时路径错误。

**实际风险程度**：在路径含空格的典型 Windows 机器（如 `C:\Users\John Doe\`）上，`vbsPath` 为 `C:\Users\John Doe\.memorytree\heartbeat-launcher.vbs`，外层已有双引号包裹，**空格情况是安全的**。但 vbsPath 自身含 `"` 时没有任何保护——这是一个结构性漏洞，即便当前场景不易触发。

---

#### H2: `scriptPath` 在 VBScript 内仅对 `"` 做转义，对 `^` `%` `!` 等 VBScript 特殊字符未处理

```typescript
const escaped = scriptPath.replace(/"/g, '""')
const content = [
  'Set WshShell = CreateObject("WScript.Shell")',
  `WshShell.Run "node ""${escaped}"" daemon run-once", 0, False`,
]
```

`WshShell.Run` 接受的命令字符串由 VBScript 解释器传给 CreateProcess，不经过 CMD shell，所以 `^`、`&`、`|` 等 CMD 特殊字符**不是问题**。

但 `%` 是 VBScript 字符串中的普通字符，同样无害。

**VBScript 字符串内真正的风险**：若 `scriptPath` 包含 `"` 本身（即 `"` 出现在路径中，Windows 路径规范中不允许，但如果上层传入了脏数据），双引号转义为 `""` 是 VBScript 字符串字面量的正确做法，这一点处理是**正确的**。

综合来看，对 `scriptPath` 的转义逻辑在 Windows 正常路径范围内是正确的，但：
- 没有对 `scriptPath` 做前置校验（如拒绝含 `"` 的路径）
- 转义后的内容直接嵌入 `.vbs` 文件，不是参数数组形式，是字符串模板注入结构

**严格来说这是 MEDIUM**，但由于 `scriptPath` 来自 `heartbeatScriptPath()`（由 `process.argv[1]` 或 `import.meta.url` 派生，非用户直接输入），实际攻击面很窄，所以整体可接受——但设计上存在注入结构。

### MEDIUM

#### M1: `trCommand` 中 `vbsPath` 含空格时行为依赖 schtasks 的引号解析

```typescript
const trCommand = `wscript.exe /B "${vbsPath}"`
```

此字符串作为 `/tr` 的单一参数值传给 `execFileSync`，schtasks 会将其存入注册表作为任务命令。

Windows schtasks `/tr` 参数要求：若路径含空格，外层引号必须存在。当前写法对 `vbsPath` 含空格是正确的（已有引号包裹）。

但 `vbsPath` 路径分隔符来自 Node.js `resolve()`，在 Windows 上会返回反斜杠路径，例如：
`C:\Users\John Doe\.memorytree\heartbeat-launcher.vbs`

这对 schtasks 和 wscript.exe 均正常工作——**此处无问题**，但需确认 `resolve()` 在 Windows 上确实返回反斜杠而非正斜杠（Node.js path.resolve 在 win32 上行为正确，已确认无误）。

#### M3: `uninstallSchtasks` 总是输出成功消息，即使任务从未注册过

```typescript

    // best effort
```

与 `installSchtasks` 的幂等性保护（检查 `isSchtasksRegistered()`）不对称。`uninstall` 在任务不存在时同样输出"已移除"，可能误导用户。

对比：`uninstallLaunchd` 用 `existsSync(plistPath)` 做了前置检查，行为更诚实。Linux 的 `uninstallCron` 则是过滤式的，天然幂等。

#### M4: VBScript 使用 `wscript.exe /B` 但文件编码写为 UTF-8

```typescript
writeFileSync(vbsPath, content, 'utf-8')
```

Windows 的 `wscript.exe` 默认按系统 ANSI 代码页（如 GBK/CP936）读取 `.vbs` 文件，不是 UTF-8。当前 VBScript 内容是纯 ASCII，所以**无实际问题**。

但若 `scriptPath` 包含中文字符（如 `C:\Users\张三\...`），UTF-8 编码的 VBScript 文件在 GBK 系统上会乱码，导致路径解析失败。

中文 Windows 用户名是极其常见的场景（本 repo 的目标用户包括中文用户），这是一个隐性的高风险路径。

## 与 uninstall 的对称性分析

| 操作 | install | uninstall |
|---|---|---|
| 幂等保护 | 有（`isSchtasksRegistered()` 检查） | 无（无条件执行+输出成功） |
| VBS 文件处理 | 创建 | 删除（有 `existsSync` 检查）|
| 错误反馈 | 友好提示 | 静默（best effort） |
| 与 launchd 对称性 | 一致 | **不一致**：launchd 用 `existsSync` 判断，schtasks 直接执行 |

`installLaunchd` 用文件存在性判断幂等，`installSchtasks` 用 `isSchtasksRegistered()` 查询系统——方法不同但都有保护。`uninstallSchtasks` 缺少等价的"是否注册"检查是最主要的对称性缺失。
