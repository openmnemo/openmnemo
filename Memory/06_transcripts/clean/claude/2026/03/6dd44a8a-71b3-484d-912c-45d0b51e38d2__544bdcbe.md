# Architecture & Code Quality Review — Report Generation Module

**Project:** `memorytree-workflow` at `C:/Users/ai/.codex/skills/memorytree-workflow/`
**Reviewer scope:** 21 source + test files, ~1,600 lines of production code, ~700 lines of tests

## 1. ARCHITECTURE

### 1.1 Module coupling and single responsibility

**MEDIUM — `build.ts` is doing too much**

`src/report/build.ts` (393 lines) is the orchestrator, but it also owns three responsibilities that should be in separate modules:

- Markdown message parsing (`parseMessagesFromMarkdown`, `parseCleanMarkdownMessages`) — currently exported from the orchestrator, used by tests and by `transcript.ts` indirectly
- Raw JSONL tool-extraction (`extractToolSummariesFromRaw`) — a non-trivial parser living inside the build coordinator
- `.gitignore` mutation (`ensureGitignore`) — a repo-management utility that belongs either in a `utils/git.ts` file or the existing CLI layer

The result is that `build.ts` imports from nearly every other module in the subsystem (8 distinct imports) while also defining core parsing logic. A future developer wanting to understand "how transcripts are parsed" must look inside the orchestrator, not a dedicated parser module.

**LOW — `dashboard.ts` exports `clientBadge` for use by sibling renderers**

`transcript-list.ts` and `transcript.ts` both import `clientBadge` from `dashboard.ts`. This creates a coupling between sibling render modules through what should be a shared utility. `clientBadge` belongs in `layout.ts` or a small `render/utils.ts` alongside `escHtml` and `renderNav`, not in a page-level renderer.

**LOW — Nav path encoding is fragile**

`layout.ts` has two hardcoded nav arrays (`NAV_ITEMS` and `NAV_ITEMS_ROOT`) differentiated by a `rootLevel: boolean` flag. The `../` prefix on all non-root hrefs assumes a single level of nesting. Transcripts are at `transcripts/{client}/{stem}.html` — two levels deep — but `renderNav` is called with `rootLevel = false` which produces one-level-up hrefs. This actually works only because the nav bar links to index pages one level up, but the assumption is invisible and breaks if the nesting depth ever increases or if a page is served from a deeper path.

## 3. CONSISTENCY

### 3.1 Immutability

The codebase follows the immutability guideline well overall.

**HIGH — `links.ts` mutates arrays in-place (lines 50–64)**

```ts
const fwd = forwardLinks[m.session_id]
if (fwd) {
  if (!fwd.includes(targetId)) fwd.push(targetId)   // mutation
} else {
  forwardLinks[m.session_id] = [targetId]
}
```

The arrays stored in `forwardLinks` and `backlinks` are mutated with `.push()`. The project coding standard is "NEVER mutate existing ones — return new copies." The immutable pattern here would be either `forwardLinks[id] = [...existing, targetId]` or using a `Map<string, Set<string>>` to deduplicate. This is the most obvious consistency violation in the entire module.

**MEDIUM — Module-level mutable state in `summarize.ts` (line 30)**

```ts
let activeCalls = 0
```

This is a module-level integer incremented and decremented by concurrent promises. It works correctly as written (single Node.js process, event loop is not preemptive), but it is a hidden side effect. A semaphore implemented as a class or a pure-functional token bucket would be safer and more testable. The current pattern is also not reset between tests — if the test runner ever runs `getSummary` in a mode where the API is reachable, `activeCalls` could leak across tests.

### 3.2 Error handling

Error handling is consistent and conservative throughout: all I/O is wrapped in try/catch and failures are silently skipped to avoid aborting the build. This matches the stated goal of resilience. The only concern is:

**LOW — Suppressed errors lose diagnostic information in `build.ts` (lines 76–82)**

The catch block around tool extraction is completely silent. Other modules at least use `console.warn`. A single `console.warn('[build] tool extraction failed:', rawPath)` would help diagnose data gaps without breaking the build.

## 4. TEST COVERAGE

### 4.1 Overall assessment

Coverage of the pure-logic layer (`stats.ts`, `summarize.ts`, `links.ts`, `search.ts`, `charts.ts`) is comprehensive and well-structured. The integration test in `build.test.ts` covers the happy path thoroughly.

### 4.2 Missing edge cases

**HIGH — No test for XSS injection through manifest data**

`renderTranscriptList`, `renderDashboard`, and `renderTranscript` all embed `m.title`, `m.client`, `m.session_id`, `m.cwd`, and `m.branch` into HTML via `escHtml`. There is no test verifying that a manifest with `title: '<script>alert(1)</script>'` produces escaped output. Given that manifests are parsed from JSON files that could be tampered with, this is a meaningful security boundary that should be tested.

**HIGH — No test for the inline JS injection path in `renderSearchPage`**

`renderSearchPage` embeds `JSON.stringify(index)` directly into a `<script>` tag (`const SEARCH_INDEX = ${indexJson}`). There is a test that the embedded JSON parses correctly (`search.test.ts` line 110), but no test verifying that an entry containing `</script>` in the title or snippet does not break out of the script context. For example, a title of `foo</script><script>alert(1)</script>` would produce a broken/malicious page. `JSON.stringify` does NOT escape `</script>` sequences. This is a real XSS vector.

**MEDIUM — No test for `renderMessageBody` / `applyInlineMarkdown` in `transcript.ts`**

The inline markdown renderer is not tested at all. Edge cases that could produce incorrect HTML include: nested emphasis (`**bold *and italic* bold**`), backtick-in-bold, and the handling of single `\n` vs. `\n\n` (the current implementation only creates paragraphs on `\n\n`, single newlines become `<br>` inside `applyInlineMarkdown` but are treated as paragraph content in `renderInlineText`). There is also no test for the code-block regex behaviour with unterminated backticks.

**MEDIUM — No test for `markdownToHtml` in `markdown.ts`**

The custom markdown converter handles headings, lists, code blocks, YAML front-matter, and inline formatting. There are no unit tests for any of these paths. Edge cases: YAML front-matter only skipped at `i === 0`, so a second `---` block would not be treated as front matter; nested lists are not supported (the `olMatch`/`ulMatch` regexes ignore indentation); the horizontal-rule handler fires for `---` which conflicts with YAML closing delimiter detection.

**MEDIUM — `buildReport` test does not verify transcript HTML content**

The integration test (`build.test.ts` line 189) only asserts that an HTML file exists. It does not assert that the transcript HTML contains the expected message text, the session title, or that `escHtml` was applied. A regression in rendering would not be caught.

**LOW — `buildDayBuckets` / `buildWeekBuckets` do not test the ISO week boundary correctness**

`isoWeek` in `stats.ts` implements an ISO-8601 week calculation (Thursday-anchor). The test for `buildWeekBuckets` only checks the format regex (`/^\d{4}-W\d{2}$/`) and the total count. It does not verify the actual week number for a known date, so an off-by-one in the calculation would pass.

**LOW — `buildSearchIndex` budget logic is tested loosely**

The test verifies that `json.length <= 51_000` with 100 sessions each having 200-char snippets. This passes comfortably, but the boundary case — where a single entry is so large that `remaining < 20` truncates to nothing — is not tested.

## 5. DEAD CODE / REDUNDANCY

**HIGH — `transcriptHref` is duplicated in three files**

The function that computes a relative URL from a `ManifestEntry` to its HTML file is independently implemented in:

- `dashboard.ts` line 132–136
- `transcript-list.ts` line 68–72
- `transcript.ts` line 161–166

The three versions are slightly different (root-relative vs. one-level-up vs. two-levels-up from the caller's location). This is the most significant maintenance hazard in the module — a future change to the output path structure would require finding and updating all three. This logic belongs in a single exported function in `layout.ts` that accepts the caller's depth as a parameter, or in a dedicated `render/utils.ts`.

**MEDIUM — `GoalFile` and `KnowledgeFile` interfaces are identical**

Both `goals.ts` and `knowledge.ts` define:
```ts
export interface { filename: string; title: string; content: string }
```
These are separate named types, which is fine conceptually, but they are structurally identical and both modules implement an identical `slugify` function. A shared `MarkdownFile` interface and a shared `slugify` utility would eliminate the duplication.

**MEDIUM — `build.ts` `loadMarkdownFiles` duplicates what `extractMarkdownTitle` already does in a slightly different location**

`loadMarkdownFiles` (line 336) and `extractMarkdownTitle` (line 362) are private helpers in `build.ts` that serve knowledge/goals page generation. This logic is entirely unrelated to the build orchestration concern of `build.ts` and should not be there at all — it should be co-located with `goals.ts` and `knowledge.ts`, or in a shared util.

**LOW — `pdf: false` is passed to `buildReport` by `cmd-report.ts` but `BuildReportOptions.pdf` is never read**

`BuildReportOptions` has a `pdf?: boolean` field defined in `types/report.ts`. `buildReport` in `build.ts` accepts it in the options type but never reads it. `cmd-report.ts` always passes `pdf: false`. The `pdf` flag appears to be a placeholder for a feature not yet implemented, but it adds noise to the public interface and could mislead callers into expecting PDF output.

## 6. PERFORMANCE

**MEDIUM — Unbounded `Promise.all` parallelism with a polling semaphore**

`build.ts` launches one promise per manifest (line 70–105) and calls `Promise.all` on all of them. The concurrency limit of 5 is enforced in `summarize.ts` via busy-polling (`setTimeout(resolve, 100)` in a while loop). This means:

1. All file I/O (reading clean markdown, resolving raw paths, writing HTML) runs unbounded in parallel. For a repository with thousands of transcripts, this will exhaust Node.js file descriptor limits and produce contention.
2. The 100ms polling interval introduces latency proportional to concurrency contention — if all 5 slots are busy, new tasks wait up to 100ms to start.

A proper solution would use a streaming processor (batches of N at a time) that gates both I/O and AI calls behind the semaphore, rather than scattering the gate only around the AI call.

**LOW — `manifests.find()` inside a `.map()` in `build.ts` (line 96–99)**

```ts
const backlinkManifests = backlinkIds
  .map(id => manifests.find(x => x.session_id === id))
```

This is O(N × M) where N = backlink count and M = total manifests. For large datasets this should be a pre-built lookup map (`Map<string, ManifestEntry>`). The same pattern appears to be intended in `links.ts` (where `lookup` is already a plain object), but `build.ts` does not share or reuse it.

**LOW — `buildSearchIndex` calls `JSON.stringify(entry)` twice per entry (lines 33, 41)**

The size check serializes the entry to measure bytes, then serializes it again after the potential truncation. One of these could be eliminated.

## 7. MAINTAINABILITY

**MEDIUM — Inline JavaScript embedded in a TypeScript template string (`search.ts` lines 85–138)**

The vanilla JS search logic is a 50-line string inside `renderSearchPage`. It has its own `escHtml` function that duplicates the one in `layout.ts`. This code is not type-checked, not linted, and not directly testable in isolation. The tests for it only verify substring presence. A change to the search logic requires editing a template string — no autocomplete, no type errors. This should be extracted to a static `.js` asset file bundled/inlined at build time, or at minimum the function-level duplication (`escHtml`) should be eliminated.

**LOW — `isoWeek` implementation is duplicated between `stats.ts` and `charts.ts`**

`isoWeek` (in `stats.ts`, line 134) and `toIsoWeek` (in `charts.ts`, line 263) implement the same ISO week calculation with identical logic. They are separate private functions. The canonical version should be exported from `stats.ts` and imported by `charts.ts`.

**LOW — No type guard on parsed manifest data**

`loadManifestsRecursive` in `build.ts` performs a type assertion (`as ManifestEntry`) after checking only `session_id` and `client`. A manifest missing `message_count` or `started_at` would pass the guard and cause silent arithmetic errors downstream (NaN in stats, empty date ranges). A schema validation step (even a manual field check) would catch malformed manifests early.

## 8. PLAN COMPLIANCE

**CRITICAL — Cache location deviates from spec**

The plan spec (and the `references/transcript-archive.md` global configuration document) places the global search index and derived artifacts at `~/.memorytree/`. The cache for AI summaries is placed at `join(output, '.cache')` — inside `Memory/07_reports/.cache/`. This means:

- The cache is **inside the output directory**, which is gitignored as `Memory/07_reports/`. The cache is therefore gitignored and lost every time the output directory is cleaned.
- If the user specifies a custom `--output`, the cache follows it rather than being stable across output changes.

The spec implies the cache should be stable and persistent, decoupled from the ephemeral output. It should live at a path like `join(root, 'Memory', '07_reports', '.cache')` that is stable regardless of the `--output` flag, or at a user-level path like `~/.memorytree/report-cache/`. As currently written, re-running `report build` after `rm -rf Memory/07_reports` will re-issue API calls for every transcript.

**HIGH — Model ID used as default does not match the plan's stated model**

`build.ts` line 32 and `main.ts` line 177 both reference `claude-haiku-4-5-20251001`. The performance guidelines specify `claude-haiku-4-5` as the appropriate lightweight model for worker agents. The date-stamped model ID `20251001` is an internal version that may be deprecated. The CLI `--model` default in `main.ts` and the `DEFAULT_MODEL` constant in `build.ts` must be kept in sync manually — if one changes, the other becomes stale. These should reference a single constant defined once.

**LOW — `pdf?: boolean` in `BuildReportOptions` is exposed in the type but never implemented**

If the plan included a PDF output feature, it is not present. If it was descoped, the option should be removed from the interface to avoid confusion.

## Key Recommendations (priority order)

1. **Fix the `</script>` injection** in `renderSearchPage` — replace the raw `JSON.stringify` embed with a version that escapes `</` sequences (e.g., replace `</` with `<\/` inside the JSON string before embedding).

2. **Move the summary cache** to a stable path outside the output directory, such as `join(root, 'Memory', '.report-cache')`, so it survives `rm -rf Memory/07_reports`.

3. **Extract `transcriptHref` into a single shared function** in `layout.ts` parameterized by caller depth, and delete the three copies.

4. **Fix `links.ts` mutation** — replace the `.push()` pattern with an immutable array update to comply with the project's own coding standard.

5. **Add a unit test module for `markdown.ts` and `transcript.ts` render functions**, including at least one XSS injection case for manifest-controlled fields.

6. **Consolidate the default model constant** into `types/report.ts` or a `constants.ts` file so `build.ts` and `main.ts` share the same reference.
