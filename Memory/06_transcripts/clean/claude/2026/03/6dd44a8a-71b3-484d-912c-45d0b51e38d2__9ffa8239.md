# Plan: `memorytree report` — Full HTML Memory Website

## Context

The user wants to export the entire MemoryTree workspace (transcripts, goals, knowledge) as a
self-contained multi-page HTML website, generated by the heartbeat daemon. Statistics are computed
purely by code; only per-session summaries call the Claude API (cached by raw_sha256 so tokens are
spent only once per transcript). The output must work from `file://` with zero external CDN
dependencies — all SVG charts are rendered inline.

## Key Implementation Notes

1. **HTML escaping**: All user content interpolated into HTML must go through `escHtml()` — handles `& < > " '`. Critical for XSS prevention.

2. **Path resolution**: `manifest.repo_clean_path` is POSIX-relative, resolve with `join(root, path)`. Fall back to `global_clean_path` if repo path is empty or missing.

3. **`--no-ai` flag gotcha**: Commander's `--no-ai` sets `opts.ai = false`. Map to `noAi: !opts.ai`.

4. **Memory efficiency**: Process transcripts one at a time in `build.ts` — parse → summarize → render → write → discard. Keep only manifests in memory throughout.

5. **Parallel AI calls**: Max 5 concurrent summary requests (simple counter semaphore, no extra deps).

6. **`.gitignore`**: `buildReport()` checks if `Memory/07_reports/` is in `.gitignore` and appends it if not.

7. **`@anthropic-ai/sdk` optional**: Dynamic import with `MODULE_NOT_FOUND` catch → returns `''`. Build never fails without the package.

8. **Serve command**: Uses `node:http` built-in, no extra dep. `EADDRINUSE` → clear error + exit 1.

## Gitea 的核心结构（真实理解）

```
Gitea 实例（自部署 or GitHub）
│
├── 个人账号（@username）
│   ├── 可以创建个人仓库（public / private）
│   ├── 可以加入多个组织
│   └── 个人仓库只属于自己
│
└── 组织（Organization）
    ├── 任何人都可以创建，有独立命名空间
    ├── 默认创建 Owners 团队（创建者是管理员）
    ├── 可以创建多个团队（Teams），每个团队权限不同
    │   ├── 团队可以访问组织全部仓库，或只指定的仓库
    │   └── 成员可以同时属于多个团队
    ├── 组织下可以创建多个仓库
    └── 一个用户可以同时加入多个组织
```

**仓库可见性三档：**
- `private`：只有成员/协作者能看
- `public`：所有人（包括未登录）都能看
- `internal`：只有这个 Gitea 实例上的登录用户能看（自部署时很有用）

## 外部对话导入的三层策略

### Layer 1 — 有结构化导出的平台（现在就能做）

| 平台 | 导出格式 | 处理方式 |
|------|---------|---------|
| **豆包** | TXT（如你现在这批文件） | 写 `parseDoubao()` parser |
| **ChatGPT** | JSON（设置 → 数据控制 → 导出） | 写 `parseChatGPT()` parser |
| **Kimi** | TXT/MD 导出 | 写 `parseKimi()` parser |
| **Perplexity** | 分享链接或复制文本 | AI 辅助解析 |

命令设计：
```bash
openmnemo import --client doubao ./chat-memo_61_20260316154801/
openmnemo import --client chatgpt ./conversations.json
openmnemo import --client generic ./任意对话.txt   # AI 辅助提取结构
```

---

### Layer 2 — 无导出功能的网页端（Phase 2 浏览器插件）

Claude.ai、Gemini Web、文心一言、通义千问——这些平台没有标准导出，只能靠：
- **浏览器插件**（已在路线图 Phase 2）：实时捕获 DOM，不需要手动导出
- **临时方案**：用户复制全文 → `openmnemo import --client generic` → AI 自动识别角色和内容

---

### Layer 3 — 语音录音（你的豆包语音备忘录场景）

你的这批豆包文件看起来是**语音转文字**的对话（标题里有"嗯、你好是这样的"这类口语），豆包已经帮你做了转录。但如果是原始音频：

```
音频文件（MP3/M4A）
    ↓ Whisper API（本地或云端）
转录文本
    ↓ openmnemo import --client generic
AI 识别角色边界 → 结构化 clean Markdown
    ↓
正常归档流程
```
