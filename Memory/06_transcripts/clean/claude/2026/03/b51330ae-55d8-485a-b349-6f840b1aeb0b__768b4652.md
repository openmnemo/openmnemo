## 1. Mem0 (mem0.ai / OpenMemory)

**What it is:** An open-source memory middleware layer that sits between an application and an LLM. Rather than logging raw chat history, it treats memory as structured, atomic, dynamically-updatable facts. Raised $24M in Oct 2025, 41K+ GitHub stars.

**How it extracts structured memory — the two-phase pipeline:**

- **Phase 1 — Extraction:** On each new exchange, Mem0 feeds the LLM three inputs: the latest user/agent turn, a rolling conversation summary, and the last N messages. It instructs the LLM to extract *new atomic facts only from the latest turn* (the rest is context only). Compound statements are automatically split — "vegetarian and allergic to nuts" becomes two separate memory entries.
- **Phase 2 — Update (async):** Each new fact is compared against the top-k similar entries already in the vector store. The LLM then chooses one of four operations: ADD, UPDATE, MERGE, or DELETE. This keeps the store coherent and non-redundant without blocking real-time response.

**Memory layers / scopes:**
- User memory — persists across all sessions for a given user
- Session memory — intra-conversation context only
- Agent memory — scoped to a specific agent instance

**Storage: hybrid datastores:**
- **Vector store** (Pinecone, Qdrant, Chroma, PGVector, etc.) — for semantic similarity search
- **Graph store (Mem0ᵍ)** — entity extraction + relation generation creates directed labeled edges (e.g., `lives_in`, `prefers`). A Conflict Detector + LLM-powered Update Resolver handles contradictions across the graph.
- **Key-value store** — for quick structured fact lookup

**OpenMemory** is the MCP-server variant, self-hosted, targeted at coding agents (Cursor, VS Code, Claude Code). It tags every memory by type, filters by project/repo, and integrates with the IDE without requiring manual prompting.

**Performance:** 26% accuracy improvement over OpenAI baseline, 91% lower p95 latency, 90% token savings vs. passing raw context.

## 2. MemGPT

**Core idea:** Treat the LLM as a CPU and design an OS-style virtual memory system around it. The paper is "MemGPT: Towards LLMs as Operating Systems" (arXiv:2310.08560, UC Berkeley).

**Two-tier architecture:**

**Tier 1 — Main Context (RAM):**
- `system_prompt` — read-only instructions describing the control flow and available function calls
- `core_memory` — writeable fixed-size working context, updated only via function calls; stores key user facts and agent persona
- `messages` (FIFO queue) — rolling conversation history; when the queue fills, a recursive summarizer evicts the oldest messages, generates a new summary, and discards the evicted messages from context (but persists them to recall storage)

**Tier 2 — External Context (disk):**
- `recall_storage` — searchable database of all evicted messages, queryable by the LLM via function calls
- `archival_storage` — long-term arbitrary-length text objects, backed by a vector DB (LanceDB by default); the LLM can iteratively page through results

**How context management works:**
The LLM itself is the memory manager. It emits function calls (not just text) — e.g., `archival_memory_search("authentication bug")`. By setting `request_heartbeat=true` in the output, it chains multiple function calls without waiting for user input. When the FIFO queue hits the flush threshold, the queue manager evicts messages and runs recursive summarization automatically.

**Key insight for openmnemo:** The self-managed memory pattern — the LLM deciding what to promote to core memory, what to archive, and what to discard — is a powerful architectural primitive.

## 3. Zep

**What it is:** A memory layer service for AI agents built around a temporally-aware knowledge graph engine called **Graphiti**. Outperforms MemGPT on the Deep Memory Retrieval benchmark (94.8% vs 93.4%) while reducing latency by 90%.

**Three-tier graph architecture (three subgraphs in one knowledge graph):**

1. **Episode Subgraph (Episodic Memory)** — Raw, immutable event nodes. Each node = one message, JSON document, or transactional snapshot, annotated with the original event timestamp. This is the ground-truth corpus — nothing is lost.

2. **Semantic Entity Subgraph (Semantic Memory)** — Entities and facts extracted from episodes. Each entity is embedded in 1024-dimensional space. Relations can form hyperedges involving multiple entities. Over time this subgraph reflects what the agent "knows" as consolidated knowledge, mirroring Tulving's episodic→semantic consolidation model.

3. **Community Subgraph** — Clusters of strongly connected entities, with high-level summarizations of each cluster. Represents the macro-level semantic structure.

**Bi-temporal model (the key differentiator):**
Every node and edge carries two timestamps:
- `event_time (T)` — when the fact actually occurred in the real world
- `ingestion_time (T')` — when it was added to Zep's graph

This allows precise reasoning about what was true *at any point in time* and handles retroactive corrections cleanly — old facts are *invalidated*, not deleted.

**Multi-faceted retrieval:** Full-text search (word similarity) + cosine similarity (semantic) + BFS graph traversal (contextual, nodes closer in graph = more similar conversational contexts). Uses union, not intersection.

**Procedural memory** in Zep's framework: implicit knowledge in LLM weights + explicit knowledge in agent code. Not stored in the graph.

Sources: [arxiv 2501.13956](https://arxiv.org/abs/2501.13956) · [getzep.com](https://www.getzep.com/) · [Graphiti GitHub](https://github.com/getzep/graphiti)

## 4. OpenClaw — "小龙虾"

**What it is:** OpenClaw (previously ClawdBot/MoltBot) is an open-source AI agent framework that went viral in early 2026, hitting 250K+ GitHub stars. The nickname "小龙虾" (crayfish/lobster) comes from the Chinese tech community. Its main claim: it gives the AI real persistent memory that "gets smarter the more you use it."

**Memory architecture — three tiers, all plain Markdown:**

| Tier | File | Loaded when | Content |
|------|------|-------------|---------|
| Always-on | `MEMORY.md` | Every session, always in context | ~100 lines of curated durable facts, preferences, key decisions |
| Daily log | `memory/YYYY-MM-DD.md` | Today + yesterday auto-loaded | Raw observations, not curated |
| Deep knowledge | `memory/people/`, `projects/`, `topics/`, `decisions/` | On-demand via vector search | Long-form notes, organized by domain |

The agent has two memory tools: `memory_search` (semantic search over ~400-token chunks with 80-token overlap, scoring = `vectorWeight × vectorScore + textWeight × textScore` plus temporal decay so recent memories rank higher) and `memory_get` (read a specific file at a line offset).

**Skills system:** Each skill is a folder with a `SKILL.md` file — YAML frontmatter + natural-language instructions. Workspace skills override global skills. Shared via **ClawHub** registry. The **Librarian skill** is the most architecturally relevant: it promotes durable knowledge upward — daily observations → structured knowledge → `MEMORY.md`.

**SOUL.md:** Defines agent identity, personality, and permission boundaries. Loaded at startup.

**Heartbeat mechanism:** Every 30 seconds a message is sent to the agent from `heartbeat.md` containing pending tasks and periodic reminders. The agent acts if there's work; returns a no-op keyword otherwise. This is what enables proactive/autonomous behavior.

**Key insight for openmnemo:** The three-tier Markdown architecture with a Librarian-style promotion pipeline maps well onto a Git-based memory store. Daily `.md` files = commits; `MEMORY.md` = a curated README that the agent keeps up to date.

## 5. Git-Based Knowledge Management for AI

**Existing work — chat with git history (Couchbase approach):**
Each commit is output as a JSON object: `{ hash, author_name, author_email, date, message, body }`. The body is concatenated into a `text` field, embedded with OpenAI (1536-dim, dot_product similarity), stored, and queried with RAG. The approach uses `--all --reverse` to index all branches in chronological order.

**brainwires-rag (Rust):** A library + MCP server that does semantic search over commit history with on-demand indexing, exposable directly to AI assistants.

**LanceDB + Continue MCP:** Indexes git blame (line-by-line authorship) + commit diffs using sentence transformers, stored in a local LanceDB instance, queryable via MCP.

**Conventional Commits as a standardization layer:** The `<type>[scope]: <description>` format (`feat:`, `fix:`, `docs:`, `refactor:`, etc.) makes commits machine-readable, enables automated changelog generation, and dramatically improves RAG quality because the type prefix acts as a filterable metadata field. Tooling: `commitlint` + `semantic-release`.

**Temporal knowledge graph angle (Graphiti):** Graphiti's model of "new data integrates immediately, old facts are invalidated not deleted, every fact traces back to its source episode" is essentially what Git already does natively. The gap is the extraction layer — turning raw diffs and commit messages into semantic entities.

**Key insight for openmnemo:** Your `chat-memo_*/` directories already produce conversation transcripts committed to Git. The conventional commit types in your history (`feat:`, `docs:`, `chore:`) are already metadata fields. A lightweight extraction pipeline that:
1. Reads each commit's diff
2. Extracts atomic facts via LLM (Mem0-style)
3. Stores them in a SQLite vector index keyed by `(commit_sha, session_id, turn_index)`

...would give you a RAG source that is fully version-controlled, auditable, and incrementally updatable.

## 6. Conversation Transcript RAG — Chunking & Metadata Best Practices

**Chunking strategy for transcripts:**
- **Primary boundary: speaker turn** — never let a fixed-size splitter cut across a turn change. One speaker's continuous speech = one chunk (or a small window of consecutive turns).
- **Sliding window over turns** — 2–4 turns per chunk with 1-turn overlap preserves conversational context across boundaries.
- **Target chunk size:** 250–512 tokens for factual retrieval; up to 1024 for analytical queries.
- **Semantic chunking** — group sentences by embedding similarity when turn structure is absent.

**Metadata per chunk (minimum viable set):**

| Field | Purpose |
|-------|---------|
| `session_id` | Groups all chunks from one conversation |
| `turn_index` | Absolute position in conversation |
| `speaker` | Filter by user vs. assistant |
| `timestamp_iso` | Temporal filtering, recency ranking |
| `client` | Source system (claude, doubao, etc.) |
| `chunk_id` | Unique ID for auditing / re-embedding |
| `topics[]` | LLM-extracted topic tags for faceted search |
| `entity_mentions[]` | Named entities for graph linking |

**Hybrid search for retrieval:** Vector similarity (semantic) + BM25 keyword search (exact term recall), union of results, then re-rank. This is the same pattern used by Zep, OpenClaw, and production Elasticsearch deployments.

**Contextual enrichment per chunk:** Prepend a short LLM-generated summary sentence to each chunk before embedding — Microsoft's Azure AI Search benchmarks show up to 85% token savings and significantly higher retrieval precision vs. naive chunking.

**Common failure modes to avoid:**
- Cutting through a turn boundary (loses speaker attribution)
- Not including `session_id` in metadata (can't reconstruct conversation context)
- Indexing raw tool-call events without filtering (noise)
- Over-large chunks that contain multiple topic shifts

Sources: [Weaviate chunking strategies](https://weaviate.io/blog/chunking-strategies-for-rag) · [Cohere chunking guide](https://docs.cohere.com/page/chunking-strategies) · [Microsoft Azure context-aware RAG](https://techcommunity.microsoft.com/blog/azure-ai-foundry-blog/context-aware-rag-system-with-azure-ai-search-to-cut-token-costs-and-boost-accur/4456810) · [Databricks ultimate chunking guide](https://community.databricks.com/t5/technical-blog/the-ultimate-guide-to-chunking-strategies-for-rag-applications/ba-p/113089)
