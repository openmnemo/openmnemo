## Step 2: Schema migration strategy in `db.ts`

### The core problem

`CREATE TABLE IF NOT EXISTS` will not add new columns to an existing DB. The two new columns (`content TEXT`, `commit_layer TEXT`) must be added via `ALTER TABLE ... ADD COLUMN` when the DB already exists.

### Migration function

Add a private `migrateSchema(db)` function that runs after `CREATE TABLE IF NOT EXISTS`:

```sql
ALTER TABLE transcripts ADD COLUMN content      TEXT NOT NULL DEFAULT '';
ALTER TABLE transcripts ADD COLUMN commit_layer TEXT NOT NULL DEFAULT '';
```

SQLite `ALTER TABLE ADD COLUMN` is safe and idempotent-friendly — wrap each in a try/catch that ignores `duplicate column name` errors (SQLite error code `SQLITE_ERROR`, message contains `duplicate column name`). This is the standard pattern for additive migrations without a version table.

```ts
function migrateSchema(db: InstanceType<typeof Database>): void {
  for (const col of ['content', 'commit_layer']) {
    try {
      db.exec(`ALTER TABLE transcripts ADD COLUMN ${col} TEXT NOT NULL DEFAULT ''`)
    } catch (e: unknown) {
      // ignore "duplicate column name" — column already exists
      if (!(e instanceof Error) || !e.message.includes('duplicate column name')) throw e
    }
  }
}
```

Call `migrateSchema(db)` inside `openDb()` after the pragma setup, so every write-path open automatically migrates. The read path (`searchTranscripts`) opens readonly and never calls `migrateSchema` — that is correct, reads against an old schema still work because the columns default to `''`.

### Updated FTS4 virtual table

Drop the old `CREATE_FTS_SQL` and replace with:

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS transcripts_fts USING fts4(
  content='transcripts',
  title, cwd, branch, content, commit_layer
)
```

Note: the FTS4 `content=` option refers to the source table name, not the column named `content`. SQLite FTS4 allows a column named `content` inside a `content='tablename'` virtual table — they are distinct concepts. This is valid SQL.

Because `CREATE VIRTUAL TABLE IF NOT EXISTS` will not alter an existing FTS table, the migration strategy is: drop and recreate the FTS table when the column count is wrong. Add a private `migrateFts(db)`:

```ts
function migrateFts(db: InstanceType<typeof Database>): void {
  // Check if FTS table has the new columns by inspecting its schema
  // fts4 stores column list in sqlite_master as the CREATE statement
  const row = db.prepare(
    "SELECT sql FROM sqlite_master WHERE type='table' AND name='transcripts_fts'"
  ).get() as { sql: string } | undefined

  const needsRebuild = !row || !row.sql.includes('commit_layer')
  if (needsRebuild) {
    db.exec('DROP TABLE IF EXISTS transcripts_fts')
    db.exec(CREATE_FTS_SQL)  // new 5-column definition
    db.exec(REBUILD_FTS_SQL) // repopulate from transcripts table
  }
}
```

Call `migrateFts(db)` inside `initSchema` and `upsertSearchIndex` after `migrateSchema`.

- Layer 3 — `content`: full clean markdown text

### New function signature

```ts
  globalRoot: string,

```

### Logic

```ts
export function searchRecall(globalRoot, query, limit = 20): SearchRecallResult {
  const dbPath = join(globalRoot, 'index', 'search.sqlite')

  // Layer 1: commit_layer column only
  const layer1 = searchTranscriptsByColumns(dbPath, query, ['commit_layer'], limit)
  if (layer1.length > 0) return { layer: 1, results: layer1 }

  // Layer 2: title, cwd, branch (existing behaviour)
  const layer2 = searchTranscripts(dbPath, query, limit)
  if (layer2.length > 0) return { layer: 2, results: layer2 }

  // Layer 3: content column (full text)
  const layer3 = searchTranscriptsByColumns(dbPath, query, ['content'], limit)
  return { layer: 3, results: layer3 }
}
```

Add a private helper `searchTranscriptsByColumns` in `db.ts` that accepts a column list and builds a column-scoped FTS4 MATCH query using the `column:term` syntax FTS4 supports:

```ts
// FTS4 column filter syntax: "title:foo OR cwd:foo"
function buildColumnQuery(columns: string[], sanitized: string): string {
  return columns.map(col => `${col}:${sanitized}`).join(' OR ')
}
```

Export this helper as `searchTranscriptsByColumns(dbPath, query, columns, limit)` from `db.ts`.

For now, Layer 1 will always return empty (since `commit_layer` is `''` for all rows), so the effective behaviour is Layer 2 first, then Layer 3 fallback — which is a clean, testable state.
