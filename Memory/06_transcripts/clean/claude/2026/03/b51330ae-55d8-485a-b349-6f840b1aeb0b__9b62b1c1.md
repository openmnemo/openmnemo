**[HIGH-2] Gemini tool event double-counting: the `toolUse` node will be visited twice**

```

For the `model` node:

The check is: `getNested(rec, 'toolUse', 'name')` — this resolves to null (no `toolUse` inside `toolUse`). Then `rec['toolName']` and `rec['functionName']` are checked — both undefined. `rec['tool']` is undefined.

Actually on re-reading: the inner object `{ name: 'read_file', input: {...} }` has no `role` and no `toolName`/`functionName`/`tool` field. `getNested` on this inner node for `'toolUse'` returns null. So no second tool event is pushed from the inner node.

The test assertion `toBeGreaterThanOrEqual(1)` (line 174) would pass regardless of double-counting, making it a weak guard. **The assertion does not verify the exact count is 1**, so if the parser does produce 2 tool events via a future fixture change, this test would still pass. **Severity**: HIGH — using `toBeGreaterThanOrEqual(1)` instead of `toBe(1)` for tool events leaves double-counting bugs undetectable.

**[HIGH-3] Doubao session_id extraction: URL query string not handled**

The parser extracts session_id via `url.split('/').pop()` (parse.ts line 417). The test URL `https://www.doubao.com/chat/38416801786598914` has no query string, so `pop()` returns `'38416801786598914'` correctly.

But real Doubao URLs may include query parameters such as `?source=...` or fragment identifiers. For example:
```

In that case `pop()` returns `'38416801786598914?conv=abc'`, making `session_id` include the query string. The test does not cover this case. If this is a real-world scenario, both the parser and a corresponding test are missing it.

**Severity**: HIGH — if real exports include query strings, session IDs will be incorrect and SQLite lookups will silently fail deduplication.
