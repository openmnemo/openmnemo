**CRITICAL-1: Re-trigger infinite loop is not reliably prevented**

The "Version packages" step pushes a commit to `main` using `GITHUB_TOKEN`. The common belief is that `GITHUB_TOKEN` pushes don't trigger `on: push` workflows, and that is true -- but only when the default `GITHUB_TOKEN` is used *and* no other GitHub Apps or webhook configurations re-trigger workflows. The protection works today, but it is fragile and undocumented in the workflow itself. More importantly, the real guard should be structural: the second run (triggered or not) would find no changesets and fall through to the publish step, so it would not loop infinitely. However, there is a subtler problem: `cancel-in-progress: true` on the concurrency group means the *push from the version step* can cancel the currently running workflow before the publish step executes. The sequence is:

1. Run A starts (push with changesets), versions packages, pushes commit.
2. Push triggers Run B on the same ref/group.
3. `cancel-in-progress: true` cancels Run A before it reaches publish.
4. Run B starts, finds no changesets, runs publish. This works -- but only by accident.

If `GITHUB_TOKEN` push suppression ever stops working (e.g., a GitHub App is installed that re-dispatches, or the repo switches to a PAT), the `cancel-in-progress` will actively sabotage the workflow. This is a latent correctness issue.

Recommendation: Either (a) split version and publish into two separate jobs where publish `needs: version` and runs conditionally, or (b) remove `cancel-in-progress: true` and add an explicit re-trigger guard (check if the last commit message is `chore: release`).

**HIGH-1: Changeset detection script has a false-positive edge case**

```bash
if ls .changeset/*.md 2>/dev/null | grep -qv README.md; then
```

`ls` outputs bare filenames only when stdout is a terminal; when piped, it outputs one-per-line, which is fine. But the `grep -qv README.md` pattern is a substring match -- it excludes any filename *containing* `README.md` anywhere, which is correct for the standard `README.md` file. However, `ls .changeset/*.md` will also match `.changeset/README.md` itself, and `grep -qv` will exclude it. So far so good.

The real issue: if the glob matches nothing, `ls` prints an error to stderr (suppressed by `2>/dev/null`) and produces no stdout. `grep -qv` on empty input returns exit code 1, so `has=false`. That is correct. But if a changeset file happens to be named something like `README.md.bak.md` (unlikely but possible with manual edits), the grep would incorrectly exclude it. More practically, this entire block should use a more robust approach:

```bash
count=$(find .changeset -maxdepth 1 -name '*.md' ! -name 'README.md' | head -1 | wc -l)
if [ "$count" -gt 0 ]; then
```

Or simply:
```bash
if compgen -G '.changeset/!(*README).md' > /dev/null 2>&1; then
```

Line 54:

`changeset version` can also modify:

**MEDIUM-1: No build/test step before publish**

The publish step (`pnpm release` which runs `pnpm build && pnpm publish -r ...`) does build, but there is no `pnpm test` or `pnpm typecheck` gate. The `publish.yml` manual workflow runs both `pnpm typecheck` and `pnpm test` before publishing. The release workflow skips both. If CI on the push passed, this is likely fine, but there is a race: the version commit pushed in the same workflow run has not been CI-validated. The publish runs against the checkout from *before* the version bump, not the newly pushed commit.

Recommendation: Add `pnpm typecheck && pnpm test` before `pnpm release`, or restructure so publish runs on the version-bump commit after CI passes.

**MEDIUM-2: `workflow_dispatch` + `push` trigger overlap**

Both triggers run the same job. A manual dispatch on `main` while a push is in-flight will be deduplicated by the concurrency group (one cancels the other). This is probably fine for the intended use case (manual re-run if something failed), but `workflow_dispatch` does not restrict to `main` -- it can be triggered on any branch. Running this workflow on a feature branch would attempt to push version commits and publish packages from that branch, which is almost certainly unintended.

Recommendation: Add a branch guard at the job level:
```yaml
if: github.ref == 'refs/heads/main'
```
