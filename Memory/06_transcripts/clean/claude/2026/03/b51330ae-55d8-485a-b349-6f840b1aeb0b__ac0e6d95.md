## CRITICAL Issues

### C1 — `recall.test.ts`: Mock shape does not match `RecallResult` interface
**File:** `packages/cli/tests/cli/recall.test.ts` lines 24–35

The `MOCK_RECALL_RESULT` object exposes `clean_path` and `summary`, which are not fields in the real `RecallResult` interface. The real interface uses `global_clean_path` and `clean_content`. The source command (`cmd-recall.ts`) calls `formatRecallText(result)`, which reads `payload.global_clean_path` and `payload.clean_content`. The mock never sets `found: true`, `imported_count`, or `repo`. If `formatRecallText` is mocked away, these discrepancies are hidden — but when the mock is removed or the function falls through, the output silently produces empty strings instead of the expected values.

```ts
// MOCK_RECALL_RESULT is wrong:
clean_path: '/global/clean/file.md',  // field does not exist on RecallResult
summary: 'Session summary text',      // field does not exist on RecallResult
// Missing required fields:
// found, repo, imported_count
```

The mock for `formatRecallText` also accepts `r: typeof MOCK_RECALL_RESULT`, but the real signature is `formatText(payload: RecallResult): string`. This creates a type-level lie that passes only because `formatRecallText` is replaced entirely.

### C2 — `report.test.ts`: Server started in test `serves an HTML file` is never closed
**File:** `packages/cli/tests/cli/report.test.ts` lines 161–194

The test starts a real HTTP server on port 19871 via `cmdReportServe(...)` but never closes it. The `server` variable in `afterEach` only tracks a server the test creates separately; the one created inside `cmdReportServe` is unreachable. The comment on lines 187–193 explicitly acknowledges this and resolves the cleanup promise immediately without doing anything. This leaves a live socket open after the test, potentially causing port conflicts in subsequent test runs and hanging the test process.

The same problem exists in the `returns 404 for missing file` test on port 19872 — started, never closed.

### C3 — `report.test.ts`: `cmdReportBuild` error message assertion is wrong
**File:** `packages/cli/tests/cli/report.test.ts` line 92

The test expects `cap.err()` to contain `'build failed'`. However, the source writes `console.error(`Report build failed: ${String(err)}`)`. The `captureOutput` helper in `report.test.ts` (unlike the other test files) _does_ intercept `console.error`, so the capture will work. The assertion string `'build failed'` appears inside the full message `Report build failed: Error: build failed`, so the test passes. This is technically not a bug but is fragile — the full expected string should be `'Report build failed:'`.

## HIGH Issues

### H1 — `import.test.ts`: No test for `transcriptHasContent` returning false (empty transcript)
**File:** `packages/cli/tests/cli/import.test.ts`

`cmd-import.ts` line 42–45 has a third error path: when the parsed transcript contains no importable messages, it returns exit code 1 with the message `does not contain any importable messages`. This is never tested. An empty JSONL file (only a `session_meta` event, no messages) triggers this path and it is a realistic user error.

### H2 — `discover.test.ts`: Mock uses `vi.doMock` but module may already be resolved
**File:** `packages/cli/tests/cli/discover.test.ts` lines 61–88

The second and third tests mock `@openmnemo/core` with `vi.doMock`, then call `await import('../../src/cmd-discover.js')` to get a fresh module. However, `vi.resetModules()` is called in `beforeEach` and `afterEach`, which is correct. But the first test (`returns 1 when root does not exist`) imports `cmdDiscover` at the top of the file using a static top-level import (`import { cmdDiscover } from '../../src/cmd-discover.js'`). That import is resolved once at module load time, before any `vi.doMock` can take effect. This means:
- Tests 2 and 3 get a fresh `cmdDiscover` with the mock applied (correct).
- Test 1 uses the statically-imported `cmdDiscover`, which has the real `@openmnemo/core` module.

This is consistent with daemon.test.ts's pattern (always use `await import(...)` after `vi.doMock`), but `discover.test.ts` mixes the two styles. If the real `discoverSourceFiles` is called in test 1 with a non-existent root, it short-circuits before reaching `discoverSourceFiles`, so the test currently passes by accident — but this is fragile if `cmd-discover.ts` ever re-orders its checks.

### H3 — `recall.test.ts`: `recall` is not tested for the `found: false` path
**File:** `packages/cli/tests/cli/recall.test.ts`

`cmdRecall` has no test for when `recall()` returns `{ found: false, ... }`. The source simply calls `formatRecallText(result)` in both cases, but the `formatText` function in `recall.ts` produces entirely different output for `found: false` (only `project`, `imported`, `result` fields). The mock always returns `found: true` data. A "no session found" scenario is a normal user experience and should be tested.

### H4 — `report.test.ts`: `cmdReportBuild` does not test `Open:` line in output
**File:** `packages/cli/tests/cli/report.test.ts` line 70

The source writes two `console.log` lines:
```ts
console.log(`Report generated at: ${resolve(opts.output)}`)
console.log(`Open: ${join(resolve(opts.output), 'index.html')}`)
```
The test only checks for `'Report generated at:'`. The second line is never asserted, meaning if it were accidentally removed, no test would catch it.

### H5 — `search.test.ts`: `dbPath` construction is untested/invisible
**File:** `packages/cli/tests/cli/cmd-search.ts` / `search.test.ts`

`cmd-search.ts` constructs `dbPath = join(globalRoot, 'index', 'search.sqlite')` and passes it to `searchTranscripts(dbPath, ...)`. A dedicated test asserting what `dbPath` was passed to `searchTranscripts` (like `report.test.ts` does for `buildOptions`) would close this gap.

## MEDIUM Issues

### M1 — `captureOutput` helper is duplicated across all 5 test files
All five files define an identical (or near-identical) `captureOutput` function. The `report.test.ts` version has an extended variant that also captures `console.log`/`console.error`. The other four do not — meaning if any of those commands ever switch from `process.stdout.write` to `console.log`, those tests silently stop capturing output without failing. This is inconsistent with `report.test.ts`, which correctly handles both. A shared `tests/helpers/capture.ts` utility would eliminate this drift.

### M2 — `discover.test.ts`: `discovered_count` is 0 in the mock, but the assertion checks it
**File:** `packages/cli/tests/cli/discover.test.ts` line 84

The mock returns an empty array from `discoverSourceFiles()`. The payload sets `discovered_count: discovered.length` where `discovered` is the array returned by `discoverSourceFiles`. The test asserts `expect(out).toContain('discovered_count: 0')`. This is correct, but there is no test for `discovered_count > 0` — the `imports:` section of `formatDiscoverText` is only reached when imports are non-empty, and that branch is never exercised.

### M3 — `import.test.ts`: `rawUploadPermission: 'not-set'` used for `format: 'json'` test
**File:** `packages/cli/tests/cli/import.test.ts` lines 120–138

In `cmdImport`, when `matchesCurrentRepo` is true, `rawUploadPermission` is forwarded to `importTranscript`. The JSON format test uses the real `importTranscript` call (no mock), which means the test is doing a full integration write to `tmpDir`. This is intentional but undocumented — if the test is slow or fails due to a side effect, the root cause is non-obvious. At minimum a comment explaining the integration nature would help.

### M4 — `recall.test.ts` misses the `activationTime` propagation test
**File:** `packages/cli/tests/cli/recall.test.ts`

The `cmdRecall` function passes `activationTime` directly to `recall(...)`. The JSON-format test sets `activationTime: '2024-06-01T09:00:00Z'`, but since `recall` is mocked to ignore its arguments, there is no test verifying that the value is actually forwarded. The daemon.test.ts pattern of capturing args passed to mocked functions (e.g., `savedConfigs.push(cfg)`) should be applied here.

### M5 — `report.test.ts` `cmdReportServe` with `port: 0` leaves server untracked
**File:** `packages/cli/tests/cli/report.test.ts` lines 146–158

`port: 0` asks the OS to assign an ephemeral port. The comment says "we just verify the function returns 0 without throwing". This is fine, but since `cmdReportServe` calls `process.exit(1)` on `EADDRINUSE`, any global state from a previous leaked server could cause the test process to exit without a helpful message. The `server` variable in `afterEach` does not track this server.

### M6 — `discover.test.ts` and `recall.test.ts` call `vi.resetModules()` in `beforeEach` (redundant)
Both files call `vi.resetModules()` in both `beforeEach` and `afterEach`. The call in `beforeEach` is redundant given the call in `afterEach` ran for the previous test. This is a minor noise issue that matches no pattern in `daemon.test.ts` (which only calls it in `afterEach`).

## LOW Issues

### L1 — `report.test.ts`: Hardcoded ports 19871 / 19872 risk inter-test conflicts
Ports are fixed integers. If these tests run in parallel with another suite (e.g., via `--pool=threads`) that uses the same ports, tests will flake. Using `port: 0` and reading back `server.address().port` after the listen callback fires is the robust pattern — but this requires exposing the server reference from `cmdReportServe`, which is an API change.

### L2 — `import.test.ts`: Codex JSONL uses non-standard `type: 'response_item'`
**File:** `packages/cli/tests/cli/import.test.ts` lines 87–91

The test creates a Codex JSONL with `type: 'response_item'`. This matches the Codex format, but the `payload.type: 'message'` + `content[0].type: 'input_text'` structure should be verified against `parseCodexTranscript` to confirm the fixture actually produces 2 messages (`message_count: 2`). If the parser changes, the test would silently produce a different `message_count` without a structural fixture failure.

### L3 — `search.test.ts` does not test the `limit` parameter propagation
The `cmdSearch` function passes `options.limit` directly to `searchTranscripts(dbPath, query, options.limit)`. No test verifies that the limit value is forwarded correctly.

### L4 — Consistency: `import.test.ts` imports `cmdImport` statically at top level
**File:** `packages/cli/tests/cli/import.test.ts` line 10

Unlike `recall.test.ts` and `search.test.ts` (which dynamically import after `vi.doMock`), `import.test.ts` imports statically. This is fine because `import.test.ts` never mocks `@openmnemo/core` — it relies on real integration behavior. However, if a future test in that file needs to mock core functions, the static import will prevent the mock from taking effect unless a developer knows to switch to dynamic import. This is inconsistent with the established pattern in `daemon.test.ts`.
