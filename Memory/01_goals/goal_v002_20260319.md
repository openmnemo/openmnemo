## North Star

以 OpenMnemo 为数据底座，构建 AI 原生协作平台（AgentCrawl）。
平台采用分层开源策略：数据层/存储层/推理层全部开源引流，协作层与插件层商业化变现。
用数据主权（私有部署 Gitea）对抗云端锁定型竞品，用插件生态形成网络效应护城河。

## 产品分层架构

```
【开源 · 引流 · 生态基础】
  Layer 1 数据层   → OpenMnemo
                   跨客户端对话归档、Git 版本控制
                   三路混合检索：SQLite FTS（关键词）+ 向量（语义）+ 图（实体关系）
                   支持 Claude/Codex/Gemini/Doubao，可扩展插件式解析器

  Layer 2 存储层   → Gitea（自托管）/ GitHub
                   私有部署、数据主权、版本历史

  Layer 3 推理层   → AgentCrawl Reasoning Engine（类 EvoScientist 架构）
                   多 LLM 提供商、多 agent 协作、跨 session 记忆推断
                   开源 → 社区贡献 agent/prompt/workflow → 反哺插件生态
                   ★ AI Chat Web UI（自研，嵌入报告页 → 独立 Web 客户端）
                     本地版：report serve 扩展 /api/chat，SSE streaming，FTS 检索增强
                     GitHub Pages 版：纯前端降级，用户自填 API key

【商业化 · 护城河】
  Layer 4 协作层   → 多用户权限、团队记忆共享、审计日志、冲突解决
                   企业愿意付费 / 开源社区没动力做

  Layer 5 插件层   → 插件市场（审核/分发/计费）+ 企业专属插件
                   网络效应：插件生态越丰富，用户迁移成本越高
```

## 记忆体系技术架构

### 三路混合检索

```
用户查询
    ↓
并行三路检索：
  ① SQLite FTS4    → 关键词精确匹配（title/cwd/branch/content/commit_layer）
  ② sqlite-vec     → 语义向量相似度（embedding 1536维）
  ③ Kuzu 图数据库  → 实体关系追踪（人物/项目/技术/概念/session/commit）
    ↓
RRF（Reciprocal Rank Fusion）合并排序
    ↓
返回 top-N session，附带摘要 + 关联 commit
```

### 图 Schema

节点类型：`Person` / `Project` / `Technology` / `Concept` / `Session` / `Commit`

关系类型：
- `(Person)-[:DISCUSSED]->(Concept)`
- `(Person)-[:WORKED_ON]->(Project)`
- `(Session)-[:CONTAINS]->(Concept)`
- `(Session)-[:RELATED_TO]->(Commit)`
- `(Concept)-[:DEPENDS_ON]->(Technology)`
- `(Concept)-[:EVOLVED_FROM]->(Concept)`
- `(Project)-[:USES]->(Technology)`

### LLM 提取层

heartbeat 同步时，用 Haiku 从对话全文提取：
- **关键事实** → embedding → sqlite-vec
- **实体 + 关系** → 节点/边 → Kuzu
- **对话全文** → FTS 索引 → SQLite FTS

### 可插拔存储后端（适配器模式）

| 阶段 | 向量后端 | 图后端 | 部署方式 |
|------|---------|--------|---------|
| Phase 0（个人） | `sqlite-vec` | `Kuzu` | 嵌入式，零额外进程 |
| Phase 1（团队） | `Qdrant` | `Neo4j Community` | Docker 独立服务 |
| Phase 1+（托管） | `Pinecone` / `Weaviate` 等 | `Memgraph` / `Neo4j Aura` | 云托管 |

所有后端实现统一的 `VectorAdapter` / `GraphAdapter` 接口，配置文件一行切换，上层代码零感知。升级后端时需运行迁移脚本重新 index（数据源不变，仅重建索引）。

## AI Chat 功能架构

### 战略定位

报告页 AI Chat 是**独立 Web 客户端的 0 号原型**：
- 先在现有报告页嵌入 chat widget，让用户养成"用自然语言查记忆"的习惯
- server 端 `/api/chat` 就是未来 Web 客户端的后端 API 雏形
- 前端 chat widget 代码直接演进为独立客户端的核心组件

### 两种部署模式

```
本地版（openmnemo report serve）
  用户查询
      ↓
  POST /api/chat  →  server 端
      ↓                  ↓ 1. SQLite FTS 检索相关 session（RAG context）
      ↓                  ↓ 2. 拼装 system prompt + context + 用户问题
      ↓                  ↓ 3. 调用 Claude API（streaming）
  SSE stream  ←──────────┘
  前端逐字渲染

GitHub Pages 版（纯静态）
  用户查询
      ↓
  检测无本地 server → 降级模式
      ↓ 1. 客户端 JSON 索引搜索（build 时预生成）
      ↓ 2. 展示相关 session 列表（无 LLM 调用）
      ↓    或：用户本地起 openmnemo proxy，前端连本地代理
  前端渲染结果
  （注：Anthropic API 不支持浏览器直连，GitHub Pages 版 Phase 0.6 先做搜索展示，LLM 调用 Phase 1 解决）
```

### 技术选型

**自研，不引入第三方 chat UI 框架**，原因：
- 现有报告系统是纯 inline HTML + vanilla JS，零构建工具依赖
- 引入 React/Vue 组件库会破坏零依赖设计，增加维护成本
- 自研可完全控制交互形态、API 设计，直接演进为独立客户端

**实现规模估算**：
- `packages/report/src/render/chat.ts` — chat widget HTML/CSS/JS（~150行）
- `packages/cli/src/cmd-report.ts` — 扩展 `/api/chat` 端点（~120行）
- `packages/report/src/chat/` — server 端 RAG + streaming 逻辑（~150行）

### Chat Widget 交互设计

```
报告页右下角悬浮按钮 💬
    ↓ 点击展开
┌─────────────────────────┐
│  Ask your memory...     │  ← 标题栏 + 关闭按钮
├─────────────────────────┤
│                         │
│  [AI 回复，支持 markdown] │  ← 消息区，支持代码高亮
│                         │
│  [用户消息]              │
│                         │
├─────────────────────────┤
│  [输入框]        [发送]  │  ← 底部输入区，Enter 发送
└─────────────────────────┘
```

功能：
- 多轮对话（保持 conversation history）
- 回复支持 markdown 渲染（复用现有 `markdown.ts`）
- 引用来源：回复中附带检索到的 session 链接
- 流式输出（SSE），逐字显示
- 本地/GitHub Pages 模式自动切换

### 演进路径

```
Phase 0.5  报告页嵌入 chat widget（本地版）
    ↓
Phase 0.6  GitHub Pages 降级模式（JSON 索引搜索展示，无 LLM 调用）
    ↓
Phase 1    chat 接入三路混合检索（FTS + 向量 + 图），回答质量大幅提升
    ↓
Phase 2    独立 Web 客户端（chat widget 代码直接复用，加用户系统）
    ↓
Phase 3    多用户协作 chat（团队共享记忆库）
```

## 商业模式

- **模型参考**：GitLab CE/EE 分层模式
- **免费**：Layer 1-3 全部开源，个人开发者/小团队免费自托管
- **付费**：Layer 4-5，按席位/功能收费，SaaS 托管版 + 私有部署授权
- **定价预期**：个人 Pro $19/月，团队 $49/席位/月，企业私有部署 License

## 差异化优势

1. **数据主权**：用户数据存在自己的 Gitea，不依赖云端第三方
2. **跨客户端归档**：唯一同时支持 Claude/Codex/Gemini/Doubao 的开源工具
3. **Git 关联记忆**：对话与 commit 关联，能检索「做某个功能时的思考过程」
4. **三路混合检索**：FTS 关键词 + 向量语义 + 图实体关系，检索质量远超单一方案
5. **可插拔存储后端**：从嵌入式（sqlite-vec/Kuzu）平滑升级到企业级（Qdrant/Neo4j），上层代码零改动，重建索引即可
6. **AI Chat 原生入口**：报告页内嵌 chat widget，用自然语言查询自己的记忆，培养用户习惯，为独立 Web 客户端铺路
7. **插件生态壁垒**：协作层 + 插件市场一旦形成，迁移成本极高

## 执行阶段

### Phase 0（当前）：数据层夯实
- SQLite FTS 扩展到对话全文 + Git commit 关联索引
- 引入 sqlite-vec（向量）+ Kuzu（图），实现三路混合检索
- 设计并实现 `VectorAdapter` / `GraphAdapter` 可插拔接口
- heartbeat 加入 Haiku LLM 提取层（事实/实体/关系）
- GiteaAdapter 完整链路验证（E2E 测试 + 配置文档）
- 清晰定义 `DataLayerAPI` 边界（供推理层调用）
- **★ AI Chat（Phase 0.5）**：report serve 扩展 `/api/chat`，嵌入 chat widget，FTS RAG + Claude streaming
- **★ AI Chat GitHub Pages 降级（Phase 0.6）**：纯前端模式，用户自填 API key

### Phase 1（3 个月内）：推理层 MVP
- Fork/借鉴 EvoScientist 架构，构建推理层，作为**独立开源仓库**发布（引流 + 社区贡献）
- 跨 session 记忆推断：「你上周讨论过这个问题，当时的结论是…」
- 向量/图后端升级为 Qdrant + Neo4j（支持团队规模）
- 多渠道增强分两步：先做单向 Webhook 富文本卡片（飞书/Telegram 摘要卡片，ROI 最高），再做双向 IM 交互（Phase 3）
- 发布 EvoSkill 插件，让 EvoScientist 用户能把研究对话自动归档到 OpenMnemo

### Phase 2（3-6 个月）：协作层 Alpha
- 多用户 + 权限控制（org_id / user_id / project_id / session_id 四层隔离）
- 团队记忆共享（基于 Gitea 多用户仓库）
- 插件系统基础架构

### Phase 3（6-12 个月）：商业化
- 插件市场上线
- SaaS 托管版（云端向量/图后端）
- 企业 License
- IM 双向交互（飞书/钉钉 bot）

## 成功信号

- Phase 0：三路混合检索可用，GiteaAdapter 端到端跑通，DataLayerAPI 接口定义完成，AI Chat 本地版可用
- Phase 1：推理层能回答「我上周在某项目里讨论过什么」，向量+图后端可切换
- Phase 2：2+ 用户共享同一 Gitea 记忆仓库并协作
- Phase 3：插件市场有 10+ 社区插件，付费用户 > 50

## 非目标（当前阶段）

- Phase 0 不引入独立向量/图服务（用嵌入式 sqlite-vec + Kuzu）
- 不做 LLM 训练/微调
- 不做通用 IM 应用
- 不在 Phase 0-1 考虑多用户架构
